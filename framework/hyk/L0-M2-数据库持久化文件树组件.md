# 数据库持久化文件树组件:PersistentFileTree

@framework

## 1. 能力声明（Capability Statement）

- `C1` 持久化创建能力：创建文件树节点并稳定写入数据库，同时保留节点附加属性。
- `C2` 持久化查询能力：从数据库恢复树结构并输出树视图，同时恢复节点附加属性。
- `C3` 持久化修改能力：修改节点基础字段和附加属性，并保持数据库中的树结构一致。
- `C4` 持久化移动能力：调整父子关系并保证结果仍是有效树。
- `C5` 持久化删除能力：按文件树删除策略执行删除并把结果写入数据库。
- `C6` 非能力项：不负责数据库连接实现细节，不负责文件内容存储，不负责鉴权本身。

## 2. 边界定义（Boundary / 参数）

- `FT` 文件树能力来源（reference）：必须绑定 `Ref L3.FileTree`。来源：`C1 + C2 + C3 + C4 + C5`。
- `DB` 数据库会话来源（reference）：必须绑定 `Ref L3.DatabaseSessionManager`。来源：`C1 + C2 + C3 + C4 + C5`。
- `TX` 事务边界（transaction）：单次结构变更必须在同一事务边界内完成。来源：`C1 + C3 + C4 + C5`。
- `AT` 节点附加属性（attributes）：节点附加属性必须在写库、读库和恢复时保持一致。来源：`C1 + C2 + C3`。
- `DM` 删除模式（delete_mode）：`soft | hard`，由文件树规则决定。来源：`C5`。
- `CM` 级联模式（cascade_mode）：`forbid | subtree`，由文件树规则决定。来源：`C5`。
- `RL` 树结构约束（relation_rule）：节点至多一个父节点，树中不得成环。来源：`C2 + C4`。
- `PI` 持久化接口约束（persistence_interface）：文件树动作只能通过持久化组装接口进入数据库会话，不允许直接绕过。来源：`C1 + C2 + C3 + C4 + C5`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 文件树组件基（引用基）：`Ref L3.FileTree`。来源：`R4 + R5 + C1 + C2 + C3 + C4 + C5 + FT + AT + RL + DM + CM`。
  - 提供节点、树关系、统一变更规则和附加属性承载能力。
- `B2` 数据库会话管理组件基（引用基）：`Ref L3.DatabaseSessionManager`。来源：`R4 + R5 + C1 + C2 + C3 + C4 + C5 + DB + TX`。
  - 提供会话获取、事务承载、生命周期闭合。
- `B3` 持久化组装接口基（组装接口基）：提供文件树动作到数据库读写动作的统一映射接口。来源：`R4 + R5 + C1 + C2 + C3 + C4 + C5 + PI + TX + AT`。
  - 负责把 `B1` 的树语义和节点附加属性接入 `B2` 的会话与事务边界。

## 4. 基组合原则（Base Combination Principles）

- `R1` 文件树规则继承组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：持久化文件树必须继承 Ref L3.FileTree 的节点、树关系、创建/修改/移动/删除和附加属性规则；不得改写其基础树语义。
  - `R1.3` 引用规则：RefRule L3.FileTree.R1~R6。
  - `R1.4` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R1.5` 边界绑定：`FT/AT/RL/DM/CM`。
- `R2` 数据库会话规则继承组合
  - `R2.1` 参与基：`B2`。
  - `R2.2` 组合方式：所有数据库访问都必须继承 Ref L3.DatabaseSessionManager 的会话获取、归还和生命周期规则；不得在本层直接重写会话管理。
  - `R2.3` 引用规则：RefRule L3.DatabaseSessionManager.R1~R7。
  - `R2.4` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R2.5` 边界绑定：`DB/TX`。
- `R3` 创建与查询持久化组合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：节点创建与树查询先服从 `B1` 的树语义，再通过 `B3` 映射到 `B2` 的数据库读写；节点附加属性必须随节点一起写入和恢复，不允许静默丢失。
  - `R3.3` 输出结构：`PERSISTED_TREE_VIEW`。
  - `R3.4` 输出能力：`C1 + C2`。
  - `R3.5` 边界绑定：`FT/DB/PI/TX/AT/RL`。
- `R4` 修改与移动持久化组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：修改基础字段或附加属性、移动节点前都必须先由 `B1` 校验树结构合法性，再由 `B3` 在 `B2` 提供的同一事务边界内提交到数据库。
  - `R4.3` 输出结构：`PERSISTED_MUTATION`。
  - `R4.4` 输出能力：`C3 + C4`。
  - `R4.5` 边界绑定：`FT/DB/PI/TX/AT/RL`。
- `R5` 删除持久化组合
  - `R5.1` 参与基：`B1 + B2 + B3`。
  - `R5.2` 组合方式：删除模式和级联模式由 `B1` 决定，数据库状态变更由 `B3` 接入 `B2` 执行；`B2` 不得决定软删或硬删；未删除节点的附加属性不得被无故改写。
  - `R5.3` 输出结构：`PERSISTED_DELETE_RESULT`。
  - `R5.4` 输出能力：`C5`。
  - `R5.5` 边界绑定：`FT/DB/PI/TX/AT/DM/CM`。
- `R6` 完整持久化文件树组合
  - `R6.1` 参与基：`B1 + B2 + B3`。
  - `R6.2` 组合方式：先继承 `B1` 的文件树规则和 `B2` 的数据库会话规则，再通过 `B3` 完成持久化组装；所有树动作都必须经 `B3` 进入 `B2`，并保持节点附加属性可追踪。
  - `R6.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R6.4` 边界绑定：`FT/DB/PI/TX/AT/RL/DM/CM`。
- `R7` 禁止组合
  - `R7.1` 参与基：`B1 + B2 + B3`。
  - `R7.2` 组合方式：跳过 `B1` 直接写数据库、跳过 `B3` 直接拼接文件树与会话、绕过事务边界做结构变更、由 `B2` 决定删除策略、静默丢失节点附加属性的组合均无效。
  - `R7.3` 输出能力：`C6`。
  - `R7.4` 边界绑定：`FT/DB/PI/TX/AT/RL/DM/CM`。

## 5. 验证（Verification）

- `V1` 推导一致性：`B1/B2/B3` 都必须能由至少一个 `C*` 和一个参数项推导得到。
- `V2` 引用一致性：`B1` 和 `B2` 必须明确引用已有组件，且引用规则可追溯到对应 `L3` 文档。
- `V3` 规则一致性：每个 `R*` 必须明确参与基、组合方式、引用规则、输出能力、边界绑定。
- `V4` 目标覆盖性：`R1~R6` 输出能力并集必须覆盖 `C1~C5`。
- `V5` 边界符合性：所有有效组合都必须满足树结构约束、附加属性一致性、删除策略、事务边界和持久化接口约束。
- `V6` 最小必要性：移除 `B1` 后树语义失效；移除 `B2` 后持久化失效；移除 `B3` 后组合失效。
- `V7` 结论表达：逐条输出 `R* -> C* / 参数边界 / 引用规则` 的通过或失败结论。
- `V8` 引用落点一致性：`Ref/RefRule`、`package.py references`、`ProjectConfig.bindings` 和 `assembly.py/component.py` 的组装逻辑必须一一对应。
- `V9` 属性持久化一致性：节点附加属性在创建、查询、修改、持久化恢复后都必须保持一致；不得静默丢失。
