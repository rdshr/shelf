# 项目总纲与树形结构:Project

@framework

## 1. 能力声明（Capability Statement）

- `C1` 配置接收能力：接收项目配置并形成可判定的装配输入。
- `C2` 组件装配能力：按配置从组件注册表中选择组件包并完成装配。
- `C3` 接口交付能力：把装配结果暴露为统一项目接口面。
- `C4` 非能力项：不定义具体组件内部实现。

## 2. 边界定义（Boundary / 参数）

- `PC` 项目配置（config）：必须包含项目元信息和组件挂载清单。来源：`C1 + C2`。
- `RG` 组件注册表（registry）：项目只能从注册表选择组件包。来源：`C2`。
- `MT` 挂载清单（list）：每项必须包含 `mount_id/package/settings/enabled`；当组件存在引用契约时，还必须包含 `bindings`。来源：`C1 + C2`。
- `OUT` 输出接口面（set）：结果只能通过已装配接口暴露。来源：`C3`。
- `CT` 装配约束（constraint）：`mount_id` 唯一、路由前缀唯一、禁用组件不进入运行态。来源：`C2 + C3`。
- `BD` 绑定约束（constraint）：引用绑定必须指向已挂载组件且满足被引用包契约；项目层只能连接组件暴露接口，不得改写组件内部结构约束。来源：`C2`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 配置面：L2.M0[R1] + L1.M0[R1]。来源：`C1 + PC + MT`。
  - 补充说明：提供项目配置读取与合法化。
- `B2` 装配面：L2.M0[R2] + L1.M0[R5]。来源：`C2 + RG + MT + CT + BD`。
  - 补充说明：提供组件选择、实例化、引用解析和挂载。
- `B3` 交付面：L2.M0[R3] + L1.M0[R3]。来源：`C3 + OUT + CT`。
  - 补充说明：提供项目接口输出与装配结果观察。

## 4. 基组合原则（Base Combination Principles）

- `R1` 配置进入组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：所有项目输入必须先形成合法项目配置，非法挂载项不得进入装配阶段。
  - `R1.3` 输出结构：`CONF`。
  - `R1.4` 输出能力：`C1`。
  - `R1.5` 边界绑定：`PC/MT`。
- `R2` 装配执行组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：装配面只能按合法项目配置从组件注册表选择组件包；当组件声明引用契约时，必须先解析 bindings 再生成挂载结果；项目层只能组合组件暴露接口，不得反向改写组件内部实体承载或连接约束。
  - `R2.3` 输出结构：`MOUNT_SET`。
  - `R2.4` 输出能力：`C2`。
  - `R2.5` 边界绑定：`PC/RG/MT/CT/BD`。
- `R3` 项目交付组合
  - `R3.1` 参与基：`B2 + B3`。
  - `R3.2` 组合方式：交付面只能暴露已装配接口，不允许暴露未挂载组件。
  - `R3.3` 输出结构：`APP_SURFACE`。
  - `R3.4` 输出能力：`C3`。
  - `R3.5` 边界绑定：`MT/OUT/CT`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：绕过项目配置直接 new 组件、绕过注册表选择组件、暴露未挂载接口的组合均无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`PC/RG/MT/OUT/CT`。

## 5. 验证（Verification）

- `V1` 推导一致性：每个 `B*` 必须能由至少一个 `C*` 与一个参数项推导得到。
- `V2` 规则一致性：每个 `R*` 必须明确参与基、组合方式、输出能力、边界绑定。
- `V3` 目标覆盖性：`R1~R3` 输出能力并集必须覆盖 `C1~C3`。
- `V4` 边界符合性：所有有效项目都必须满足项目配置、注册表、装配约束和绑定约束。
- `V5` 最小必要性：移除任一 `B*` 后，配置接收、组件装配或接口交付至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 参数边界` 的通过或失败结论。
