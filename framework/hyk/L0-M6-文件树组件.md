# 文件树组件:FileTree

@framework

## 1. 能力声明（Capability Statement）

- `C1` 节点创建能力：在合法父节点下创建新节点，并承载节点附加属性。
- `C2` 树结构查询能力：输出当前树视图和节点快照，并保留节点附加属性。
- `C3` 节点修改能力：修改节点基础字段和附加属性而不破坏树结构。
- `C4` 节点移动能力：改变父子关系且不形成环。
- `C5` 节点删除能力：按删除模式和级联模式收敛删除结果。
- `C6` 非能力项：不负责持久化介质、不负责数据库会话、不负责鉴权本身。

## 2. 边界定义（Boundary / 参数）

- `ID` 节点标识（id）：每个节点必须有稳定唯一标识。来源：`C1 + C2 + C3 + C4 + C5`。
- `PID` 父节点标识（parent_id）：非根节点必须指向已存在父节点。来源：`C1 + C2 + C4`。
- `NM` 节点名称（name）：节点名称必须合法且非空。来源：`C1 + C3`。
- `AT` 节点附加属性（attributes）：键值映射，可选；组件必须原样承载、校验并在查询时返回。来源：`C1 + C2 + C3`。
- `RL` 关系约束（relation_rule）：节点至多一个父节点，树中不得形成环。来源：`C2 + C4`。
- `DM` 删除模式（delete_mode）：`soft | hard`。来源：`C5`。
- `CM` 级联模式（cascade_mode）：`forbid | subtree`。来源：`C5`。
- `ST` 节点状态（state）：至少支持 `active | deleted`。来源：`C2 + C5`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 节点单元：提供节点身份、基础字段、状态、附加属性和时间边界。来源：`R5 + C1 + C2 + C3 + ID + NM + AT + ST`。
- `B2` 树关系单元：提供父子关系承载以及单父、无环、无悬空的树形关系约束。来源：`R5 + C1 + C2 + C4 + PID + RL`。
- `B3` 变更规则单元：提供创建、修改、移动、删除的统一变更规则、状态收敛和结果判定。来源：`R5 + C1 + C3 + C4 + C5 + NM + AT + RL + DM + CM + ST`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 节点建立组合
  - `R1.1` 参与基：`B1 + B2 + B3`。
  - `R1.2` 组合方式：先形成节点单元，再通过变更规则把节点接入树关系；根节点可无父节点，非根节点必须绑定已存在父节点；节点附加属性必须随节点一起建立，且新节点状态必须收敛为有效初始状态。
  - `R1.3` 输出结构：`NODE_SET`。
  - `R1.4` 输出能力：`C1`。
  - `R1.5` 边界绑定：`ID/PID/NM/AT/RL/ST`。
- `R2` 树查询组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：树视图和节点快照只能由现有节点和树关系推导；查询结果必须满足树形关系约束，且不得丢失节点附加属性。
  - `R2.3` 输出结构：`TREE_VIEW + SNAPSHOT`。
  - `R2.4` 输出能力：`C2`。
  - `R2.5` 边界绑定：`ID/PID/AT/RL/ST`。
- `R3` 修改与移动组合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：修改节点基础字段或附加属性不得破坏节点身份；移动节点前必须通过变更规则校验目标父节点合法，且移动后不得形成环或悬空关系。
  - `R3.3` 输出结构：`MUTATED_TREE`。
  - `R3.4` 输出能力：`C3 + C4`。
  - `R3.5` 边界绑定：`ID/PID/NM/AT/RL`。
- `R4` 删除组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：删除动作必须作用于已存在节点；删除结果通过变更规则按删除模式和级联模式收敛为明确状态或移除结果；未删除节点的附加属性不得被无故改写。
  - `R4.3` 输出结构：`DELETE_RESULT`。
  - `R4.4` 输出能力：`C5`。
  - `R4.5` 边界绑定：`ID/PID/AT/DM/CM/ST`。
- `R5` 完整文件树组合
  - `R5.1` 参与基：`B1 + B2 + B3`。
  - `R5.2` 组合方式：先建立节点和树关系，再执行查询、修改、移动、删除；所有变更动作都必须服从统一变更规则，并保持节点附加属性可追踪。
  - `R5.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R5.4` 边界绑定：`ID/PID/NM/AT/RL/DM/CM/ST`。
- `R6` 禁止组合
  - `R6.1` 参与基：`B1 + B2 + B3`。
  - `R6.2` 组合方式：没有节点单元直接挂关系、没有树关系直接查询、没有变更规则直接创建/修改/移动/删除、形成环或悬空父节点、静默丢失节点附加属性的组合均无效。
  - `R6.3` 输出能力：`C6`。
  - `R6.4` 边界绑定：`ID/PID/NM/AT/RL/DM/CM/ST`。

## 5. 验证（Verification）

- `V1` 推导一致性：每个 `B*` 必须能由至少一个 `C*` 和一个参数项推导得到。
- `V2` 规则一致性：每个 `R*` 必须明确参与基、组合方式、输出能力、边界绑定。
- `V3` 目标覆盖性：`R1~R5` 输出能力并集必须覆盖 `C1~C5`。
- `V4` 边界符合性：所有有效组合都必须满足节点标识、树关系、附加属性和统一变更规则边界。
- `V5` 最小必要性：移除任一 `B*` 后，创建、查询、修改/移动、删除至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 参数边界` 的通过或失败结论。
- `V7` 能力落点一致性：`C1~C5` 必须同时出现在 `L3 文档能力声明`、`package.py capabilities` 和 `api.py/component.py` 的公开出口中。
- `V8` 属性一致性：节点附加属性在创建、查询、修改、移动、删除后都必须保持可追踪；不得静默丢失。

## 6. 能力暴露（Capability Exposure）

- `E1` 语义落点：本文件 `1. 能力声明`。
- `E2` 机器落点：组件包中的 `package.py::capabilities`。
- `E3` 接口落点：组件包中的 `api.py`。
- `E4` 运行时落点：组件包中的 `component.py`。
- `E5` 一致性要求：`E1~E4` 必须一一对应；新增能力时四处同时修改。
