# 文件树组件:FileTree

@framework

本组件是 `tree.L0.M0` 的文件域实例化。
它沿用通用树的节点实体与连接接口结构，并通过文件域边界与组合规则把通用树收敛为文件树。

## 1. 能力声明（Capability Statement）

- `C1` 节点创建能力：在合法目录节点下创建文件节点或目录节点，并承载节点附加属性。
- `C2` 树结构查询能力：输出当前文件树视图和节点快照，并保留节点类别与节点附加属性。
- `C3` 节点修改能力：修改节点名称和附加属性而不改写节点类别或破坏树结构。
- `C4` 节点移动能力：改变父子关系且不形成环，目标父节点必须为目录节点。
- `C5` 节点删除能力：按删除模式和级联模式收敛文件节点或目录节点的删除结果。
- `C6` 非能力项：不负责持久化介质、不负责数据库会话、不负责文件内容存储和鉴权本身。

## 2. 边界定义（Boundary / 参数）

- `ID` 节点标识（id）：每个节点必须有稳定唯一标识。来源：`C1 + C2 + C3 + C4 + C5`。
- `PID` 父节点标识（parent_id）：作为连接结果的父节点投影，非根节点必须稳定指向已存在父节点。来源：`C1 + C2 + C4`。
- `NM` 节点名称（name）：作为单段路径名，必须合法且非空，不得包含路径分隔符。来源：`C1 + C3`。
- `NK` 节点类别（node_kind）：`file | directory`。来源：`C1 + C2 + C3 + C4 + C5`。
- `AT` 节点附加属性（attributes）：键值映射，可选；组件必须原样承载、校验并在查询时返回。来源：`C1 + C2 + C3`。
- `RL` 关系约束（relation_rule）：节点连接必须满足单父、无环、无悬空。来源：`C2 + C4 + C5`。
- `PR` 父节点承载约束（parent_rule）：只有 `directory` 节点能承载子节点，根节点必须是 `directory`。来源：`C1 + C2 + C4 + C5`。
- `DM` 删除模式（delete_mode）：`soft | hard`。来源：`C5`。
- `CM` 级联模式（cascade_mode）：`forbid | subtree`。来源：`C5`。
- `ST` 节点状态（state）：至少支持 `active | deleted`。来源：`C2 + C5`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 节点实体单元：节点作为文件域实体记录的内部结构，由标识、名称、节点类别、附加属性和状态槽位构成。来源：`C1 + C2 + C3 + C5 + ID + NM + NK + AT + ST`。
- `B2` 连接接口单元：节点之间用于挂接与重绑定的接口结构，由父节点投影入口、连接位向、父节点承载约束和关系约束槽位构成。来源：`C1 + C2 + C4 + C5 + PID + NK + PR + RL`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 节点建立与挂接组合
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：先形成节点实体，再通过连接接口完成挂接；根节点只能是 directory，非根节点必须挂接到已存在目录节点；节点类别必须在创建时确定，文件节点不得承载子节点；节点附加属性必须随实体一起建立，新节点状态必须收敛为 active。
  - `R1.3` 输出结构：`NODE_SET + LINK_BINDING`。
  - `R1.4` 输出能力：`C1`。
  - `R1.5` 边界绑定：`ID/PID/NM/NK/AT/RL/PR/ST`。
- `R2` 树查询投影组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：树视图和节点快照只能由节点实体及其连接结果推导；查询结果必须同时保留 file/directory 类别并满足目录承载、单父、无环、无悬空约束，且不得丢失节点附加属性。
  - `R2.3` 输出结构：`TREE_VIEW + SNAPSHOT`。
  - `R2.4` 输出能力：`C2`。
  - `R2.5` 边界绑定：`ID/PID/NK/AT/RL/PR/ST`。
- `R3` 节点内容修改组合
  - `R3.1` 参与基：`B1`。
  - `R3.2` 组合方式：仅允许修改节点名称和附加属性；节点身份和节点类别必须保持不变，修改不得隐式改写任何连接结果或删除状态。
  - `R3.3` 输出结构：`MUTATED_NODE`。
  - `R3.4` 输出能力：`C3`。
  - `R3.5` 边界绑定：`ID/NM/NK/AT/ST`。
- `R4` 接口重绑定组合
  - `R4.1` 参与基：`B1 + B2`。
  - `R4.2` 组合方式：移动节点时必须先解除原有挂接，再通过连接接口重绑定到目标父节点；目标父节点必须是已存在的 directory 节点，重绑定后不得形成环或悬空关系，节点身份、节点类别和附加属性必须保持连续。
  - `R4.3` 输出结构：`REBOUND_TREE`。
  - `R4.4` 输出能力：`C4`。
  - `R4.5` 边界绑定：`ID/PID/NK/AT/RL/PR`。
- `R5` 删除收敛组合
  - `R5.1` 参与基：`B1 + B2`。
  - `R5.2` 组合方式：删除动作必须作用于已存在节点实体及其连接结果；文件节点删除只作用于自身，目录节点删除必须按删除模式和级联模式收敛为明确的节点状态或连接移除结果；未删除节点的附加属性不得被无故改写。
  - `R5.3` 输出结构：`DELETE_RESULT`。
  - `R5.4` 输出能力：`C5`。
  - `R5.5` 边界绑定：`ID/PID/NK/AT/DM/CM/ST/PR/RL`。
- `R6` 完整文件树组合
  - `R6.1` 参与基：`B1 + B2`。
  - `R6.2` 组合方式：通过节点实体承载文件节点和目录节点信息，通过连接接口形成文件树拓扑；在此基础上开展创建、查询、修改、移动和删除；所有结果都必须保持父子关系可投影、目录承载约束成立且节点附加属性可追踪。
  - `R6.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R6.4` 边界绑定：`ID/PID/NM/NK/AT/RL/PR/DM/CM/ST`。
- `R7` 禁止组合
  - `R7.1` 参与基：`B1 + B2`。
  - `R7.2` 组合方式：没有节点实体直接挂接、没有连接接口直接形成父子树、根节点被定义为 file、file 节点承载子节点、绕过连接接口直接改写父节点投影、节点创建后切换 node_kind、重绑定形成环或悬空父节点、删除后状态与连接结果不收敛、静默丢失节点附加属性的组合均无效。
  - `R7.3` 输出能力：`C6`。
  - `R7.4` 边界绑定：`ID/PID/NM/NK/AT/RL/PR/DM/CM/ST`。

## 5. 验证（Verification）

- `V1` 推导一致性：每个 `B*` 必须能由至少一个 `C*` 和一个参数项推导得到。
- `V2` 规则一致性：每个 `R*` 必须明确参与基、组合方式、输出能力、边界绑定。
- `V3` 目标覆盖性：`R1~R5` 输出能力并集必须覆盖 `C1~C5`。
- `V4` 边界符合性：所有有效组合都必须满足节点标识、节点类别、目录承载约束、连接约束、附加属性和删除收敛边界。
- `V5` 最小必要性：移除任一 `B*` 后，节点承载或连接挂接至少一项缺失，导致创建、查询、修改/移动、删除至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 参数边界` 的通过或失败结论。
- `V7` 能力落点一致性：`C1~C5` 必须同时出现在 `L3 文档能力声明`、`package.py capabilities` 和 `api.py/component.py` 的公开出口中。
- `V8` 属性一致性：节点附加属性在创建、查询、修改、移动、删除后都必须保持可追踪；不得静默丢失。
- `V9` 类别一致性：节点类别在创建后必须保持稳定；查询、移动和删除结果不得隐式改写 file/directory 语义。

## 6. 能力暴露（Capability Exposure）

- `E1` 语义落点：本文件 `1. 能力声明`。
- `E2` 机器落点：组件包中的 `package.py::capabilities`。
- `E3` 接口落点：组件包中的 `api.py`。
- `E4` 运行时落点：组件包中的 `component.py`。
- `E5` 一致性要求：`E1~E4` 必须一一对应；新增能力时四处同时修改。
