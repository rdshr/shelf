# 项目文件树基础组件:ProjectFileTree

@framework

本组件是当前项目的文件树基础组件实例化框架。
它不改变被引用组件的最小可行基，只固定引用对象、边界取值和项目级组合规则。
本版本只覆盖软删除，不包含延迟删除、定时删除和清理工作流。

## 1. 能力声明（Capability Statement）

- `C1` 持久化承载能力：通过固定会话链和持久化树链支撑项目文件树的稳定读写。
- `C2` 持久化节点创建能力：创建节点并稳定写入项目文件树。
- `C3` 持久化树查询能力：输出树视图和节点快照，并保留节点附加属性。
- `C4` 持久化节点修改与移动能力：修改节点名称、附加属性或父子关系并保持持久化一致。
- `C5` 软删除能力：把节点及其子树收敛为已删除状态，不做物理删除。
- `C6` 非能力项：不覆盖系统目录规则、延迟删除、定时删除、文件内容联动删除和鉴权链路。

## 2. 边界定义（Boundary / 参数）

- `SF` 会话工厂挂载：固定引用 `mount_id=session-factory-main`，`package=session_factory`。来源：`C1 + C2 + C3 + C4 + C5`。
- `DB` 会话管理挂载：固定引用 `mount_id=db-main`，`package=db_session`，并满足 `bindings.session_factory = session-factory-main`。来源：`C1 + C2 + C3 + C4 + C5`。
- `FT` 文件树规则挂载：固定引用 `mount_id=tree-rules`，`package=file_tree`。来源：`C1 + C2 + C3 + C4 + C5`。
- `PT` 持久化文件树挂载：固定引用 `mount_id=persistent-tree`，`package=persistent_file_tree`，并满足 `bindings.file_tree = tree-rules`、`bindings.db_session = db-main`。来源：`C1 + C2 + C3 + C4 + C5`。
- `URL` 数据库地址：由 `session-factory-main.settings.default_database_url` 固定提供。来源：`C1 + C2 + C3 + C4 + C5`。
- `MD` 会话模式：固定为 `sync`。来源：`C1 + C2 + C3 + C4 + C5`。
- `PO` 保留策略：固定为 `shared`。来源：`C1 + C2 + C3 + C4 + C5`。
- `CO` 关闭责任：固定为 `caller`。来源：`C1 + C2 + C3 + C4 + C5`。
- `DM` 删除模式：固定为 `soft`。来源：`C5`。
- `CM` 级联模式：固定为 `subtree`。来源：`C5`。
- `AO` 删除覆盖约束（api_override）：项目交付接口不得允许调用方覆盖 `DM/CM` 固定值。来源：`C5`。
- `TB` 表名：固定为 `persistent_tree_nodes`。来源：`C1 + C2 + C3 + C4 + C5`。
- `NK` 节点类别：固定继承 `file_tree` 的 `file | directory` 节点类别，不允许项目实例改写。来源：`C2 + C3 + C4 + C5`。
- `PR` 父节点承载约束：固定继承只有 `directory` 节点能承载子节点且根节点必须是 `directory` 的规则。来源：`C2 + C3 + C4 + C5`。
- `AT` 节点附加属性：节点 `attributes` 必须在创建、修改、查询、快照和恢复时保持一致。来源：`C2 + C3 + C4`。
- `API` 项目接口面：以 `/persistent-tree` 作为主交付接口面。来源：`C1 + C2 + C3 + C4 + C5`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 项目配置装配基（自有基）：提供本项目的挂载清单、bindings 和边界取值收敛。来源：`R1 + R2 + C1 + C2 + C3 + C4 + C5 + SF + DB + FT + PT + URL + MD + PO + CO + DM + CM + AO + TB`。

- `B2` 会话工厂组件基（引用基）：`Ref L3.SessionFactory`。来源：`C1 + C2 + C3 + C4 + C5 + SF + URL`。
  - 实例化绑定：`mount=session-factory-main`。

- `B3` 数据库会话管理组件基（引用基）：`Ref L3.DatabaseSessionManager`。来源：`C1 + C2 + C3 + C4 + C5 + DB + MD + PO + CO`。
  - 实例化绑定：`mount=db-main`。

- `B4` 文件树组件基（引用基）：`Ref L3.FileTree`。来源：`C1 + C2 + C3 + C4 + C5 + FT + NK + PR + AT`。
  - 实例化绑定：`mount=tree-rules`。

- `B5` 数据库持久化文件树组件基（引用基）：`Ref L3.PersistentFileTree`。来源：`C1 + C2 + C3 + C4 + C5 + PT + TB + DM + CM + AT`。
  - 实例化绑定：`mount=persistent-tree`。

- `B6` 项目文件树组装接口基（组装接口基）：提供项目级边界锁定、删除覆盖限制和统一接口收敛。来源：`R4 + R5 + C1 + C2 + C3 + C4 + C5 + SF + DB + FT + PT + MD + PO + CO + DM + CM + AO + API`。
  - 负责把 `B2~B5` 接成当前项目可交付的文件树基础组件。

## 4. 基组合原则（Base Combination Principles）

- `R1` 会话链实例化组合
  - `R1.1` 参与基：`B1 + B2 + B3 + B6`。
  - `R1.2` 组合方式：先由项目配置装配基固定数据库地址，再挂载会话工厂，最后以 bindings.session_factory = session-factory-main 实例化数据库会话管理组件；会话链只能通过项目文件树组装接口进入上层交付。
  - `R1.3` 引用规则：RefRule L3.SessionFactory.R1~R6 + RefRule L3.DatabaseSessionManager.R1~R7。
  - `R1.4` 输出能力：`C1`。
  - `R1.5` 边界绑定：`SF/DB/URL/MD/PO/CO`。

- `R2` 文件树持久化链实例化组合
  - `R2.1` 参与基：`B1 + B3 + B4 + B5 + B6`。
  - `R2.2` 组合方式：先挂载文件树规则组件，再把 db-main 和 tree-rules 通过 bindings 接入 persistent-tree；所有持久化树动作都必须经 persistent_file_tree 完成，并由项目文件树组装接口统一暴露；项目实例必须固定继承 file/directory 类别语义和 directory 承载子节点规则。
  - `R2.3` 引用规则：RefRule L3.FileTree.R1~R7 + RefRule L3.PersistentFileTree.R1~R7。
  - `R2.4` 输出能力：`C2 + C3 + C4 + C5`。
  - `R2.5` 边界绑定：`FT/PT/DB/TB/NK/PR/DM/CM/AT`。

- `R3` 会话边界收敛组合
  - `R3.1` 参与基：`B3 + B5 + B6`。
  - `R3.2` 组合方式：persistent_file_tree 在本项目实例中只能以 sync + shared + caller 的会话边界使用 db_session，不得切换为其他模式。
  - `R3.3` 输出能力：`C1`。
  - `R3.4` 边界绑定：`MD/PO/CO`。

- `R4` 软删除收敛组合
  - `R4.1` 参与基：`B4 + B5 + B6`。
  - `R4.2` 组合方式：删除动作固定收敛为软删除；文件节点删除只作用于自身，目录节点删除以 subtree 方式收敛为 deleted 状态，不做物理移除，不接入延迟删除或定时删除流程；项目交付接口不得允许调用方覆盖删除模式和级联模式。
  - `R4.3` 输出能力：`C5`。
  - `R4.4` 边界绑定：`NK/DM/CM/AO`。

- `R5` 属性贯通组合
  - `R5.1` 参与基：`B4 + B5 + B6`。
  - `R5.2` 组合方式：节点附加属性必须随节点一起创建、更新、查询和恢复；项目实例不得静默丢弃 attributes。
  - `R5.3` 输出能力：`C2 + C3 + C4`。
  - `R5.4` 边界绑定：`AT`。

- `R6` 项目交付组合
  - `R6.1` 参与基：`B1 + B5 + B6`。
  - `R6.2` 组合方式：项目对外以 /persistent-tree 作为主交付接口面，完成创建、查询、修改、移动、删除和快照能力交付；交付接口必须服从项目文件树组装接口定义，不得直接暴露底层可变删除边界。
  - `R6.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R6.4` 边界绑定：`API/PT/AO`。

- `R7` 禁止组合
  - `R7.1` 参与基：`B1 + B2 + B3 + B4 + B5 + B6`。
  - `R7.2` 组合方式：缺少任一引用基、缺少项目文件树组装接口基、bindings 不闭合、绕过 persistent_file_tree 直接改数据库、在本实例中改写既定会话边界、把根节点定义为 file、允许 file 节点承载子节点、把删除动作接入延迟/定时清理链路、静默丢失节点附加属性的组合均无效。
  - `R7.3` 输出能力：`C6`。
  - `R7.4` 边界绑定：`SF/DB/FT/PT/URL/MD/PO/CO/NK/PR/DM/CM/AO/TB/AT/API`。

## 5. 验证（Verification）

- `V1` 引用闭合性：`session-factory-main -> db-main -> persistent-tree` 的绑定链必须闭合。
- `V2` 目标覆盖性：`R1~R6` 输出能力并集必须覆盖 `C1~C5`，且 `R1/R3` 只负责承载能力，`R2/R4/R5/R6` 负责具体树动作能力。
- `V3` 边界收敛性：会话边界必须固定为 `sync/shared/caller`，删除边界必须固定为 `soft/subtree`，且项目接口不得允许覆盖。
- `V4` 属性一致性：节点附加属性在创建、修改、查询、快照和恢复后都必须一致。
- `V5` 软删除一致性：删除后节点必须进入 `deleted` 状态，且不触发物理删除或定时清理流程。
- `V6` 文件域一致性：节点类别必须固定为 `file | directory`，且只有 `directory` 节点能承载子节点。
- `V7` 最小必要性：移除任一 `B*` 后，数据库会话链、树规则链或持久化交付链至少一项失败。
- `V8` 结论表达：逐条输出 `R* -> C* / 边界 / 引用规则` 的通过或失败结论。
