# 事务作用域组件:TransactionScope

@framework

## 1. 能力声明（Capability Statement）

- `C1` 事务进入能力：在受管数据库会话上建立显式事务边界。
- `C2` 事务内执行能力：保证同一事务中的数据库动作绑定到同一受管会话。
- `C3` 事务收敛能力：把事务结果收敛为提交或回滚两种终态之一。
- `C4` 事务观察能力：输出事务标识、绑定会话句柄和当前事务相位。
- `C5` 非能力项：不负责数据库配置来源，不负责会话工厂生成，不负责 ORM 语义。

## 2. 边界定义（Boundary / 参数）

- `DS` 数据库会话来源（reference）：必须绑定 `Ref L3.DatabaseSessionManager`。来源：`C1 + C2 + C3 + C4`。
- `HP` 会话句柄（handle）：一个事务作用域必须且只能绑定一个受管会话句柄。来源：`C1 + C2 + C4`。
- `TX` 事务标识（transaction_id）：每个事务作用域必须有稳定唯一标识。来源：`C1 + C4`。
- `TS` 事务相位（transaction_state）：只允许 `open | committed | rolled_back`。来源：`C2 + C3 + C4`。
- `AR` 原子收敛规则（atomic_rule）：一个事务作用域只能以 `commit` 或 `rollback` 结束，不允许双重终态。来源：`C2 + C3`。
- `AI` 组装接口约束（assembly_interface）：事务进入、执行、提交、回滚和观察都必须通过本层组装接口接入数据库会话。来源：`C1 + C2 + C3 + C4`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 数据库会话管理组件基（引用基）：`Ref L3.DatabaseSessionManager`
  - 提供受管会话、句柄定位和生命周期闭合能力。
  - 来源：`L2[R4,R5] + C1 + C2 + C3 + C4 + DS + HP`。
- `B2` 事务记录单元（自有基）
  - 提供 `transaction_id -> handle_id -> state` 的统一承载结构。
  - 负责记录事务起点、终点和当前相位。
  - 来源：`L2[R5] + C1 + C2 + C3 + C4 + TX + HP + TS`。
- `B3` 事务收敛规则单元（自有基）
  - 提供提交、回滚和终态收敛规则。
  - 负责保证事务只在允许相位内推进，且只能落入单一终态。
  - 来源：`L2[R5] + C2 + C3 + AR + TS`。
- `B4` 事务组装接口基（组装接口基）
  - 提供事务进入、事务内执行、提交、回滚和观察的统一接口。
  - 负责把 `B1` 的受管会话能力与 `B2/B3` 的事务语义接成事务作用域。
  - 来源：`L2[R4,R5] + C1 + C2 + C3 + C4 + AI + HP + TX + TS`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 会话规则继承组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：事务作用域组件必须继承 `Ref L3.DatabaseSessionManager` 的会话句柄和生命周期规则；不得在本层重写会话管理语义。
  - `R1.3` 引用规则：`RefRule L3.DatabaseSessionManager.R1~R7`。
  - `R1.4` 输出能力：`C1 + C4`。
  - `R1.5` 边界绑定：`DS/HP`。
- `R2` 事务进入组合
  - `R2.1` 参与基：`B1 + B2 + B4`。
  - `R2.2` 组合方式：事务进入时必须先定位一个有效会话句柄，再建立唯一事务标识，并把事务相位初始化为 `open`；一个事务不得绑定多个会话句柄。
  - `R2.3` 输出结构：`TRANSACTION_SCOPE`。
  - `R2.4` 输出能力：`C1`。
  - `R2.5` 边界绑定：`DS/HP/TX/TS/AI`。
- `R3` 事务内执行组合
  - `R3.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R3.2` 组合方式：事务内所有数据库动作都只能作用于当前 `open` 相位的事务，并且必须持续绑定同一会话句柄；不得跨事务复用执行上下文。
  - `R3.3` 输出结构：`TRANSACTION_EXECUTION`。
  - `R3.4` 输出能力：`C2`。
  - `R3.5` 边界绑定：`HP/TX/TS/AR/AI`。
- `R4` 提交与回滚组合
  - `R4.1` 参与基：`B2 + B3 + B4`。
  - `R4.2` 组合方式：事务只能从 `open` 相位收敛到 `committed` 或 `rolled_back`；提交后不得再回滚，回滚后不得再提交；双重终态均无效。
  - `R4.3` 输出结构：`COMMIT_RESULT + ROLLBACK_RESULT`。
  - `R4.4` 输出能力：`C3`。
  - `R4.5` 边界绑定：`TX/TS/AR/AI`。
- `R5` 事务观察组合
  - `R5.1` 参与基：`B1 + B2 + B4`。
  - `R5.2` 组合方式：事务快照必须同时来源于事务记录和受管会话句柄；不得对未记录事务做反向推断。
  - `R5.3` 输出结构：`TRANSACTION_SNAPSHOT`。
  - `R5.4` 输出能力：`C4`。
  - `R5.5` 边界绑定：`DS/HP/TX/TS/AI`。
- `R6` 完整事务作用域组合
  - `R6.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R6.2` 组合方式：先继承受管会话规则，再建立事务边界，之后执行事务内动作，最后通过提交或回滚完成单一终态收敛。
  - `R6.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R6.4` 边界绑定：`DS/HP/TX/TS/AR/AI`。
- `R7` 禁止组合
  - `R7.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R7.2` 组合方式：没有会话句柄直接建事务、没有事务记录直接提交或回滚、在非 `open` 相位执行数据库动作、同一事务落入双重终态、绕过组装接口直接对数据库会话做事务控制的组合均无效。
  - `R7.3` 输出能力：`C5`。
  - `R7.4` 边界绑定：`DS/HP/TX/TS/AR/AI`。

## 5. 验证（Verification）

- `V1` 推导一致性：`B1/B2/B3/B4` 都必须能由至少一个 `C*` 和一个参数项推导得到。
- `V2` 引用一致性：`B1` 必须明确引用已有组件，且引用规则可追溯到对应 `L3` 文档。
- `V3` 规则一致性：每个 `R*` 必须明确参与基、组合方式、引用规则、输出能力、边界绑定。
- `V4` 目标覆盖性：`R1~R6` 输出能力并集必须覆盖 `C1~C4`。
- `V5` 边界符合性：所有有效组合都必须满足会话句柄、事务标识、事务相位、原子收敛规则和组装接口约束。
- `V6` 最小必要性：移除 `B1` 后受管会话来源失效；移除 `B2` 后事务定位失效；移除 `B3` 后提交/回滚收敛失效；移除 `B4` 后事务组装失效。
- `V7` 结论表达：逐条输出 `R* -> C* / 参数边界 / 引用规则` 的通过或失败结论。
- `V8` 引用落点一致性：`Ref/RefRule`、`package.py references`、`ProjectConfig.bindings` 和 `assembly.py/component.py` 的组装逻辑必须一一对应。

- `E2` 引用落点：本文件 `3. 最小可行基` 和 `4. 基组合原则` 中的 `Ref L3.* / RefRule L3.*`。

- `E5` 运行时落点：组件包中的 `component.py` 与 `assembly.py`。
- `E6` 一致性要求：`E1~E5` 必须一一对应；新增引用基或新增本层规则时必须同步修改。
