# 基础组件模块框架:BaseComponent

@framework

## 1. 能力声明（Capability Statement）

- `C1` 输入承接能力：接收组件输入并完成合法化。
- `C2` 内部处理能力：在组件边界内完成状态变化或数据处理。
- `C3` 接口暴露能力：对外提供稳定可调用接口。
- `C4` 包装配能力：以独立组件包形式进入项目注册表和装配根。
- `C5` 能力声明能力：把组件能力同时暴露为人可读语义和机可读声明。
- `C6` 引用组合能力：当组件复用已有组件时，能够显式声明引用基并通过组装接口形成更大组件。
- `C7` 非能力项：不承担项目级硬编码装配。

## 2. 边界定义（Boundary / 参数）

- `I` 输入契约（contract）：输入必须可校验。来源：`C1`。
- `P` 处理闭环（process）：状态变化必须在组件内部闭合。来源：`C2`。
- `O` 对外接口（interface）：暴露接口必须稳定。来源：`C3`。
- `CFG` 组件配置（config）：组件必须有独立配置模型。来源：`C1 + C4`。
- `PKG` 包出口契约（contract）：必须声明 `package_id/capabilities/settings_model/create_runtime/create_router/cleanup`。来源：`C3 + C4`。
- `REF` 引用契约（contract）：当组件复用已有组件时，必须声明 `references` 并满足被引用包契约。来源：`C4 + C6`。
- `CAP` 能力声明契约（contract）：必须同时存在 `L3 能力声明`、`package.py capabilities`、`api.py 或 component.py 的可调用出口` 三个落点。来源：`C3 + C5`。
- `ASM` 组装接口契约（contract）：当组件存在引用基时，必须存在显式组装接口承接引用组件与本层规则。来源：`C2 + C6`。
- `L` 生命周期（lifecycle）：创建、使用、释放必须有明确边界。来源：`C2 + C3 + C4`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 输入基：L0.M5[R1,R2]。来源：`C1 + I + CFG`。
  - 补充说明：提供输入校验和配置合法化。
- `B2` 处理基：L0.M4[R2,R3]。来源：`C2 + P + L`。
  - 补充说明：提供内部状态和处理闭环；当组件不复用已有组件时，该基由组件自身实现。
- `B3` 接口与包出口基：L0.M7[R6] + L0.M4[R5]。来源：`C3 + C4 + C5 + O + PKG + CAP + L`。
  - 补充说明：提供稳定接口、独立组件包出口和能力机读声明。
- `B4` 组装接口基：L0.M7[R1,R2]。来源：`C2 + C6 + REF + ASM + L`。
  - 补充说明：当组件复用已有组件时，显式承接引用基、本层规则和生命周期边界。

补充约定：

- `B*` 必须标明是 `自有基`、`引用基` 或 `组装接口基`。
- 当组件复用已有组件时，至少要出现一个 `引用基` 和一个 `组装接口基`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 输入合法化组合
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：组件输入和组件配置都必须先在输入基完成合法化。
  - `R1.3` 输出结构：`INPUT_UNIT`。
  - `R1.4` 输出能力：`C1`。
  - `R1.5` 边界绑定：`I/CFG`。
- `R2` 处理闭环组合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：处理基只能消费已合法化输入，并在组件边界内完成闭环。
  - `R2.3` 输出结构：`RUNTIME_UNIT`。
  - `R2.4` 输出能力：`C2`。
  - `R2.5` 边界绑定：`I/P/L`。
- `R3` 包出口组合
  - `R3.1` 参与基：`B2 + B3`。
  - `R3.2` 组合方式：接口与包出口基必须把运行时封装为稳定接口，并导出可被项目装配根消费的组件包契约；同时必须输出机读能力清单，并能映射到具体接口或公开方法。
  - `R3.3` 输出结构：`PACKAGE_ENTRY + CAPABILITY_MANIFEST + INTERFACE_SURFACE`。
  - `R3.4` 输出能力：`C3 + C4 + C5`。
  - `R3.5` 边界绑定：`O/PKG/CAP/L`。
- `R4` 引用组装组合
  - `R4.1` 参与基：`B2 + B4`。
  - `R4.2` 组合方式：当组件声明引用基时，所有被引用组件都必须先通过引用契约解析，再由组装接口基接入本层规则；不得绕过组装接口直接拼装。
  - `R4.3` 输出结构：`REFERENCE_GRAPH + ASSEMBLY_PORT`。
  - `R4.4` 输出能力：`C2 + C6`。
  - `R4.5` 边界绑定：`P/REF/ASM/L`。
- `R5` 完整组件组合
  - `R5.1` 参与基：`B1 + B2 + B3 (+ B4)`。
  - `R5.2` 组合方式：先输入合法化，再形成运行时闭环；若存在引用基，则必须先完成引用组装；最后导出稳定接口、组件包出口和能力清单。
  - `R5.3` 输出能力：`C1 + C2 + C3 + C4 + C5 + C6`。
  - `R5.4` 边界绑定：`I/P/O/CFG/PKG/REF/CAP/ASM/L`。
- `R6` 禁止组合
  - `R6.1` 参与基：`B1 + B2 + B3 (+ B4)`。
  - `R6.2` 组合方式：没有组件配置模型、没有包出口契约、没有机读能力清单、存在引用基却没有组装接口基、绕过引用契约直接拼装、接口绕过运行时、生命周期不闭合的组合均无效。
  - `R6.3` 输出能力：`C7`。
  - `R6.4` 边界绑定：`I/P/O/CFG/PKG/REF/CAP/ASM/L`。

## 5. 验证（Verification）

- `V1` 推导一致性：每个 `B*` 必须能由至少一个 `C*` 与一个参数项推导得到。
- `V2` 规则一致性：每个 `R*` 必须明确参与基、组合方式、输出能力、边界绑定。
- `V3` 目标覆盖性：`R1~R5` 输出能力并集必须覆盖 `C1~C6`。
- `V4` 边界符合性：所有有效组件都必须满足输入、配置、包出口、引用契约、组装接口、能力声明和生命周期边界。
- `V5` 最小必要性：移除任一 `B*` 后，输入合法化、内部处理、引用组装或独立包出口至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 参数边界` 的通过或失败结论。
- `V7` 能力一致性：`L3` 文档中的能力声明、`package.py` 中的 `capabilities`、`api.py/component.py` 中的公开出口必须一一对应，不允许只存在于其中一处。
- `V8` 结构落点一致性：每个组件包都必须稳定产出 `component.py + package.py + api.py` 三个能力落点文件。
- `V9` 引用一致性：当组件存在引用基时，`L3` 文档中的 `Ref/RefRule`、`package.py references`、`ProjectConfig.bindings` 和 `assembly.py/component.py` 中的组装逻辑必须一一对应。
