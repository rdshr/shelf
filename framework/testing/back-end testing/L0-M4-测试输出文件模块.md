# 测试输出文件模块:TestOutputFile

@framework

## 1. 能力声明（Capability Statement）

- `C1` 测试获取输出文件能力：测试系统根据传入的参数，成功提取并返回最终处理生成的结果文件或输出数据的能力。
- `C2` 测试获取异常列表能力：测试系统获取任务运行过程中产生的异常记录明细，或整体执行结果统计汇总数据的能力。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：本模块不负责无效参数/无效组合场景下具体错误码语义的一致性。

## 2. 边界定义（Boundary / 参数）

- `POSTGETOUTPUTFILE` 边界：execute_id (属性: 必选，类型: string)
                           node_id (属性: 必选，类型: string)
                           range_start (属性: 必选，类型: integer)
                           range_end (属性: 必选，类型: integer) 来源：`C1`。
- `POSTGETERRORLIST` 边界：execute_id (属性: 必选，类型: string)
                          node_id (属性: 必选，类型: string)
                          range_start (属性: 必选，类型: integer)
                          range_end (属性: 必选，类型: integer) 来源：`C2`。

## 3. 最小可行基（Minimum Viable Bases）
    
- `B1` 测试获取输出文件基：
       由
         调用接口: POST /api/get_output_file
         参数：无
         请求体：execute_id
                node_id
                range_start
                range_end
         调用结果：调用成功状态码返回2000，调用失败状态码返回!2000
       构成的测试获取输出文件结构。来源：`C1 + POSTGETOUTPUTFILE`。

- `B2` 测试获取异常列表基：
       由
         调用接口: POST /api/get_error_list
         参数：无
         请求体：execute_id
                node_id
                range_start
                range_end
         调用结果：调用成功状态码返回2000，调用失败状态码返回!2000
       构成的测试获取异常列表结构。来源：`C2 + POSTGETERRORLIST`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 基边界有效原则（单元测试）
  - `R1.1` 参与基：`B1`、`B2`。
  - `R1.2` 组合方式：每个基独立调用并满足第 2 节边界有效条件（`range_start <= range_end`）；对依赖 execute_id/node_id 的基，使用预置有效标识样例进行单基验证。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`POSTGETOUTPUTFILE + POSTGETERRORLIST`。

- `R2` 基边界无效原则（单元测试）
  - `R2.1` 参与基：`B1`、`B2`。
  - `R2.2` 组合方式：每个基均独立调用，并人为构造“单 API 非法参数”（缺字段、类型错误、`range_start > range_end` 等）触发边界无效。
  - `R2.3` 失效结论：`N1`。
  - `R2.4` 边界绑定：`POSTGETOUTPUTFILE + POSTGETERRORLIST`。

- `R3` 随机有效组合原则（混沌集成测试）
  - `R3.1` 参与基：`{B1, B2}` 的随机子集。
  - `R3.2` 组合方式：在 `R1` 已通过且已剔除单 API 非法参数样例前提下，合法随机组合必须满足：
  - `R3.2.a` 组合形态仅允许 `{B1}`、`{B2}`、`{B1+B2}`。
  - `R3.2.b` 命中 `{B1+B2}` 时，`execute_id/node_id/range_start/range_end` 必须同轮一致。
  - `R3.2.c` 全部参与基参数均满足边界有效条件。
  - `R3.3` 输出能力：在约束下实现对 `C1~C2` 的高覆盖随机有效组合。
  - `R3.4` 边界绑定：`POSTGETOUTPUTFILE + POSTGETERRORLIST`。

- `R4` 无效组合原则（混沌集成测试）
  - `R4.1` 参与基：`B1 + B2` 及其随机组合实例。
  - `R4.2` 组合方式：在各基边界参数有效前提下，若组合违反 `R3.2` 约束（同轮上下文不一致、跨轮串用标识、窗口不一致），判定为无效组合。
  - `R4.3` 失效结论：`N1`。
  - `R4.4` 边界绑定：`POSTGETOUTPUTFILE + POSTGETERRORLIST`。

- `R5` 分层与判定优先级原则（补充）
  - `R5.1` 参与基：`B1`、`B2` 及其组合。
  - `R5.2` 组合方式：执行顺序固定为“先 `R1/R2`（单元）后 `R3/R4`（混沌集成）”；仅当 `R1/R2` 通过时才进入组合测试；冲突样例按无效优先（`R4 > R3`）。
  - `R5.3` 输出能力：`C1 + C2` 的判定链路稳定可复验。
  - `R5.4` 边界绑定：`POSTGETOUTPUTFILE + POSTGETERRORLIST`。

- `R6` 覆盖与稳定性补充原则（补充）
  - `R6.1` 参与基：`{B1, B2}` 的有效组合集合。
  - `R6.2` 组合方式：固定 `seed`、固定轮次（建议 `N>=30`）执行混沌测试；要求每个 `B*` 至少一次有效命中，且关键链路“输出文件查询→异常列表查询”可稳定复现。
  - `R6.3` 输出能力：`C1 + C2` 的覆盖充分性与稳定性。
  - `R6.4` 边界绑定：`POSTGETOUTPUTFILE + POSTGETERRORLIST`。

## 5. 验证（Verification）

- `V1` 基边界有效原则验证（对应 `R1`）：通过条件：`B1`、`B2` 在边界有效样例下均返回 `2000`；失败原因：任一样例返回 `!=2000`；证据：单 API 有效样例请求/响应对照表。

- `V2` 基边界无效原则验证（对应 `R2`）：通过条件：`B1`、`B2` 在单 API 非法参数样例下均返回 `!=2000`；失败原因：任一样例返回 `2000`；证据：单 API 无效样例请求/响应对照表。

- `V3` 随机有效组合原则验证（对应 `R3`）：通过条件：命中的有效组合均满足 `R3.2` 约束，且全部返回 `2000`；失败原因：有效组合约束未满足或任一调用返回 `!=2000`；证据：有效组合命中统计、参数一致性报告与响应日志。

- `V4` 无效组合原则验证（对应 `R4`）：通过条件：在各基边界有效前提下，触发 `R4.2` 的组合均返回 `!=2000`；失败原因：任一无效组合返回 `2000`；证据：无效组合样例集、规则命中记录与响应日志。

- `V5` 分层与门槛验证（对应 `R5`）：通过条件：执行严格遵循“先 `V1/V2` 后 `V3/V4`”，且仅当 `V1/V2` 通过才进入组合测试；失败原因：阶段串扰、样例污染或门槛失效；证据：测试流水线执行序列与样例分层清单。

- `V6` 能力导出与问题分层校验：通过条件：`C1`、`C2` 均有有效规则导出链，且失败样例可区分为“结构定义问题”与“实例配置问题”；失败原因：能力未覆盖或归因不可分层；证据：`R -> C` 覆盖报告与失败分类报告。

- `V7` 覆盖率验证（对应 `R6`）：通过条件：有效组合类型命中完整，且每个 `B*` 至少被有效执行一次；失败原因：组合类型缺失或基长期未命中；证据：组合命中统计与覆盖率报告。

- `V8` 稳定性与顺序闭环验证（对应 `R6`）：通过条件：固定 `seed` 的多轮随机执行中“输出文件查询→异常列表查询”关键链路顺序稳定且有效组合成功率为 `100%`；失败原因：顺序漂移、偶发失败或数据污染；证据：轮次执行序列、成功率统计与失败归因报告。

- `V9` 结论判定：通过条件：`V1~V8` 全部通过；失败原因：任一验证项失败；证据：汇总判定单（通过/失败、原因、证据路径）。
