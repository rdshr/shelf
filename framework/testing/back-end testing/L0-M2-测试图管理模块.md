# 测试图管理模块:TestGraphManagement

@framework

## 1. 能力声明（Capability Statement）

- `C1` 测试获取草稿图列表能力：测试系统获取当前处于草稿状态（未正式发布）的所有图列表的能力。
- `C2` 测试获取指定图配置详情能力：测试根据指定的图 ID (graph_id)，获取单个图详细配置数据的能力。
- `C3` 测试更新图配置能力：测试根据指定的图 ID，修改、更新并保存已有图配置信息的能力。
- `C4` 测试获取图的历史发布版本能力：测试根据基础图 ID (base_id)，查询该图所有已经发布的历史版本列表的能力。
- `C5` 测试获取最新版本图配置能力：测试根据基础图 ID，直接快捷拉取该图最新一个已发布版本的详细配置数据的能力。
- `C6` 测试彻底删除图及其所有版本能力：测试根据基础图 ID，将该图及其关联的所有历史版本记录从系统中整体清除的能力。
- `C7` 测试删除特定图版本能力：测试根据基础图 ID 和具体的版本号，仅删除该图某个特定历史版本，而不影响其他版本的能力。
- `C8` 测试创建新图能力：测试在系统中新建并初始化一个空白或初始状态图的能力。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：本模块不负责无效参数/无效组合场景下具体错误码语义的一致性。

## 2. 边界定义（Boundary / 参数）

- `GETGRAPHLIST` 边界：name (属性: 可选，类型: string | (string | null)，有效范围: 已存在的图名称并支持模糊查询) 
                      limit (属性: 可选，类型: integer，默认值: 10，有效范围:[1,100])
                      page (属性: 可选，类型: integer，默认值: 1，有效范围:≥1) 来源：`C1`。
- `GETGRAPHID` 边界：graph_id (属性: 必选，类型: string, 有效范围: 已存在的 graph_id) 来源：`C2`。
- `PUTGRAPHID` 边界：graph_id (属性: 必选，类型: string, 有效范围: 已存在的 graph_id)
                     name (属性: 必选，类型: string)
                     description (属性: 必选，类型: string)
                     status (属性: 必选，类型: string)
                     config_json (属性: 必选，类型: object) 来源：`C3`。
- `GETGRAPHBASEIDVERSIONS` 边界：base_id (属性: 必选，类型: string, 有效范围: 已存在的 base_id) 来源：`C4`。
- `GETGRAPHBASEIDLATEST` 边界：base_id (属性: 必选，类型: string, 有效范围: 已存在的 base_id) 来源：`C5`。
- `DELETEGRAPHBASEID` 边界：base_id (属性: 必选，类型: string, 有效范围: 已存在的 base_id) 来源：`C6`。
- `DELETEGRAPHBASEIDVERSION` 边界：base_id (属性: 必选，类型: string, 有效范围: 已存在的 base_id)
                                  version (属性: 必选，类型: integer, 有效范围: 已存在的 version) 来源：`C7`。
- `POSTGRAPH` 边界：name (属性: 必选，类型: string)
                    description (属性: 必选，类型: string)
                    status (属性: 必选，类型: string)
                    config_json (属性: 必选，类型: object)
                    base_id (属性: 必选，类型: string) 来源：`C8`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 基边界有效原则（单元测试）
  - `R1.1` 参与基：`B1`、`B2`、`B3`、`B4`、`B5`、`B6`、`B7`、`B8`。
  - `R1.2` 组合方式：每个基独立调用并满足第 2 节边界有效条件；对依赖 graph_id/base_id/version 的基，使用预置有效标识样例进行单基验证。
  - `R1.3` 输出能力：`C1 + C2 + C3 + C4 + C5 + C6 + C7 + C8`。
  - `R1.4` 边界绑定：`GETGRAPHLIST + GETGRAPHID + PUTGRAPHID + GETGRAPHBASEIDVERSIONS + GETGRAPHBASEIDLATEST + DELETEGRAPHBASEID + DELETEGRAPHBASEIDVERSION + POSTGRAPH`。

- `R2` 基边界无效原则（单元测试）
  - `R2.1` 参与基：`B1`、`B2`、`B3`、`B4`、`B5`、`B6`、`B7`、`B8`。
  - `R2.2` 组合方式：每个基均独立调用，并人为构造“单 API 非法参数”（缺字段、类型错误、越界、无效标识）触发边界无效。
  - `R2.3` 失效结论：`N1`。
  - `R2.4` 边界绑定：`GETGRAPHLIST + GETGRAPHID + PUTGRAPHID + GETGRAPHBASEIDVERSIONS + GETGRAPHBASEIDLATEST + DELETEGRAPHBASEID + DELETEGRAPHBASEIDVERSION + POSTGRAPH`。

- `R3` 随机有效组合原则（混沌集成测试）
  - `R3.1` 参与基：`{B1, B2, B3, B4, B5, B6, B7, B8}` 的随机子集。
  - `R3.2` 组合方式：在 `R1` 已通过且已剔除单 API 非法参数样例前提下，合法随机组合必须满足：
  - `R3.2.a` 含 `B2/B3` 时，`graph_id` 来源必须可追溯到同轮 `B1` 列表选定记录或同轮 `B8` 返回值。
  - `R3.2.b` 含 `B4/B5/B6/B7` 时，`base_id` 来源必须可追溯到同轮 `B8` 返回值或同轮 `B2` 已确认记录。
  - `R3.2.c` 含 `B7` 时，`version` 来源必须可追溯到同轮 `B4` 版本列表或同轮 `B5` 最新版本上下文。
  - `R3.2.d` 含 `B6` 时，`B6` 必须作为该 `base_id` 链路末步。
  - `R3.3` 输出能力：在约束下实现对 `C1~C8` 的高覆盖随机有效组合。
  - `R3.4` 边界绑定：`GETGRAPHLIST + GETGRAPHID + PUTGRAPHID + GETGRAPHBASEIDVERSIONS + GETGRAPHBASEIDLATEST + DELETEGRAPHBASEID + DELETEGRAPHBASEIDVERSION + POSTGRAPH`。

- `R4` 无效组合原则（混沌集成测试）
  - `R4.1` 参与基：`B2`、`B3`、`B4`、`B5`、`B6`、`B7` 及其随机组合实例。
  - `R4.2` 组合方式：在各基边界参数有效前提下，若出现“无来源直调依赖基、`B6` 非末步、跨轮串用失效标识、删除后继续操作同一目标”等情况，判定为无效组合。
  - `R4.3` 失效结论：`N1`。
  - `R4.4` 边界绑定：`GETGRAPHID + PUTGRAPHID + GETGRAPHBASEIDVERSIONS + GETGRAPHBASEIDLATEST + DELETEGRAPHBASEID + DELETEGRAPHBASEIDVERSION + POSTGRAPH + GETGRAPHLIST`。

- `R5` 分层与判定优先级原则（补充）
  - `R5.1` 参与基：`B1`、`B2`、`B3`、`B4`、`B5`、`B6`、`B7`、`B8` 及其组合。
  - `R5.2` 组合方式：执行顺序固定为“先 `R1/R2`（单元）后 `R3/R4`（混沌集成）”；仅当 `R1/R2` 通过时才进入组合测试；冲突样例按无效优先（`R4 > R3`）。
  - `R5.3` 输出能力：`C1 + C2 + C3 + C4 + C5 + C6 + C7 + C8` 的判定链路稳定可复验。
  - `R5.4` 边界绑定：`GETGRAPHLIST + GETGRAPHID + PUTGRAPHID + GETGRAPHBASEIDVERSIONS + GETGRAPHBASEIDLATEST + DELETEGRAPHBASEID + DELETEGRAPHBASEIDVERSION + POSTGRAPH`。

- `R6` 覆盖与稳定性补充原则（补充）
  - `R6.1` 参与基：`{B1, B2, B3, B4, B5, B6, B7, B8}` 的有效组合集合。
  - `R6.2` 组合方式：固定 `seed`、固定轮次（建议 `N>=30`）执行混沌测试；要求每个 `B*` 至少一次有效命中，且关键链路“建图→查图→版本查询/删除”可稳定复现。
  - `R6.3` 输出能力：`C1 + C2 + C3 + C4 + C5 + C6 + C7 + C8` 的覆盖充分性与稳定性。
  - `R6.4` 边界绑定：`GETGRAPHLIST + GETGRAPHID + PUTGRAPHID + GETGRAPHBASEIDVERSIONS + GETGRAPHBASEIDLATEST + DELETEGRAPHBASEID + DELETEGRAPHBASEIDVERSION + POSTGRAPH`。

## 5. 验证（Verification）

- `V1` 基边界有效原则验证（对应 `R1`）：通过条件：`B1~B8` 在边界有效样例下均返回 `2000`；失败原因：任一样例返回 `!=2000`；证据：单 API 有效样例请求/响应对照表。

- `V2` 基边界无效原则验证（对应 `R2`）：通过条件：`B1~B8` 在单 API 非法参数样例下均返回 `!=2000`；失败原因：任一样例返回 `2000`；证据：单 API 无效样例请求/响应对照表。

- `V3` 随机有效组合原则验证（对应 `R3`）：通过条件：命中的有效组合均满足 `R3.2` 约束，且全部返回 `2000`；失败原因：有效组合约束未满足或任一调用返回 `!=2000`；证据：有效组合命中统计、标识溯源报告与响应日志。

- `V4` 无效组合原则验证（对应 `R4`）：通过条件：在各基边界有效前提下，触发 `R4.2` 的组合均返回 `!=2000`；失败原因：任一无效组合返回 `2000`；证据：无效组合样例集、规则命中记录与响应日志。

- `V5` 分层与门槛验证（对应 `R5`）：通过条件：执行严格遵循“先 `V1/V2` 后 `V3/V4`”，且仅当 `V1/V2` 通过才进入组合测试；失败原因：阶段串扰、样例污染或门槛失效；证据：测试流水线执行序列与样例分层清单。

- `V6` 能力导出与问题分层校验：通过条件：`C1~C8` 均有有效规则导出链，且失败样例可区分为“结构定义问题”与“实例配置问题”；失败原因：能力未覆盖或归因不可分层；证据：`R -> C` 覆盖报告与失败分类报告。

- `V7` 覆盖率验证（对应 `R6`）：通过条件：有效组合类型命中完整，且每个 `B*` 至少被有效执行一次；失败原因：组合类型缺失或基长期未命中；证据：组合命中统计与覆盖率报告。

- `V8` 稳定性与顺序闭环验证（对应 `R6`）：通过条件：固定 `seed` 的多轮随机执行中“建图→查图→版本查询/删除”关键链路顺序稳定且有效组合成功率为 `100%`；失败原因：顺序漂移、偶发失败或数据污染；证据：轮次执行序列、成功率统计与失败归因报告。

- `V9` 结论判定：通过条件：`V1~V8` 全部通过；失败原因：任一验证项失败；证据：汇总判定单（通过/失败、原因、证据路径）。
