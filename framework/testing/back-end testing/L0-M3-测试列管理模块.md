# 测试列管理模块:TestColumnsManagement

@framework

## 1. 能力声明（Capability Statement）

- `C1` 测试获取所有列列表能力：测试系统一次性拉取并返回当前所有已配置的列定义数据集合的能力。
- `C2` 测试查询列能力：测试根据特定参数进行模糊匹配或精确过滤，从而查询目标列的能力。
- `C3` 测试新增列能力：测试向系统中添加新的列属性配置或字段元数据的能力。
- `C4` 测试更新列配置能力：测试对系统中已存在的列属性进行信息修改和保存的能力。
- `C5` 测试删除单列记录能力：测试从系统中精确移除单个列定义数据的能力。
- `C6` 测试批量删除列记录能力：测试一次性从系统中清除多条列定义数据的能力。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：本模块不负责无效参数/无效组合场景下具体错误码语义的一致性。

## 2. 边界定义（Boundary / 参数）

- `GETCOLUMNSALL` 边界：is_banned (属性: 可选，类型: boolean | (boolean | null)) 来源：`C1`。
- `GETCOLUMNSSEARCH` 边界：page (属性: 可选，类型: integer，默认值: 1，有效范围:≥1)
                          limit (属性: 可选，类型: integer，默认值: 20，有效范围:≥1)
                          column_name (属性: 可选，类型: string | (string | null))
                          column_alias (属性: 可选，类型: string | (string | null))
                          is_banned (属性: 可选，类型: boolean | (boolean | null)) 来源：`C2`。
- `POSTCOLUMNADD` 边界：column_name (属性: 必选，类型: string)
                       column_alias (属性: 必选，类型: string)
                       is_banned (属性: 必选，类型: boolean)
                       column_type (属性: 必选，类型: string)
                       metadata_json (属性: 必选，类型: object) 来源：`C3`。
- `POSTCOLUMNUPDATE` 边界：id (属性: 必选，类型: integer, 有效范围: 已存在的列id)
                          column_name (属性: 可选，类型: string | (string | null)，有效范围: 长度≥1)
                          column_alias (属性: 可选，类型: string | (string | null)，有效范围: 长度≥1)
                          is_banned (属性: 可选，类型: boolean | (boolean | null))
                          column_type (属性: 可选，类型: string | (string | null)，有效范围: 长度≥1)
                          metadata_json (属性: 可选，类型: object) 来源：`C4`。
- `DELETECOLUMNID` 边界：id (属性: 必选，类型: integer, 有效范围: 已存在的列id) 来源：`C5`。
- `DELETECOLUMNS` 边界：无，来源：`C6`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 基边界有效原则（单元测试）
  - `R1.1` 参与基：`B1`、`B2`、`B3`、`B4`、`B5`、`B6`。
  - `R1.2` 组合方式：每个基独立调用并满足第 2 节边界有效条件；对依赖 id 的基，使用预置有效标识样例进行单基验证。
  - `R1.3` 输出能力：`C1 + C2 + C3 + C4 + C5 + C6`。
  - `R1.4` 边界绑定：`GETCOLUMNSALL + GETCOLUMNSSEARCH + POSTCOLUMNADD + POSTCOLUMNUPDATE + DELETECOLUMNID + DELETECOLUMNS`。

- `R2` 基边界无效原则（单元测试）
  - `R2.1` 参与基：`B1`、`B2`、`B3`、`B4`、`B5`、`B6`。
  - `R2.2` 组合方式：每个基均独立调用，并人为构造“单 API 非法参数”（缺字段、类型错误、越界、无效标识）触发边界无效。
  - `R2.3` 失效结论：`N1`。
  - `R2.4` 边界绑定：`GETCOLUMNSALL + GETCOLUMNSSEARCH + POSTCOLUMNADD + POSTCOLUMNUPDATE + DELETECOLUMNID + DELETECOLUMNS`。

- `R3` 随机有效组合原则（混沌集成测试）
  - `R3.1` 参与基：`{B1, B2, B3, B4, B5, B6}` 的随机子集。
  - `R3.2` 组合方式：在 `R1` 已通过且已剔除单 API 非法参数样例前提下，合法随机组合必须满足：
  - `R3.2.a` 含 `B4/B5` 时，`id` 来源必须可追溯到同轮 `B1` 列表、同轮 `B2` 查询结果或同轮 `B3` 新增返回值。
  - `R3.2.b` 含 `B6` 时，`B6` 必须作为该轮清理链路末步。
  - `R3.2.c` 同轮对同一 `id` 不允许“先删后改”或“删除后继续删除/更新”。
  - `R3.3` 输出能力：在约束下实现对 `C1~C6` 的高覆盖随机有效组合。
  - `R3.4` 边界绑定：`GETCOLUMNSALL + GETCOLUMNSSEARCH + POSTCOLUMNADD + POSTCOLUMNUPDATE + DELETECOLUMNID + DELETECOLUMNS`。

- `R4` 无效组合原则（混沌集成测试）
  - `R4.1` 参与基：`B4`、`B5`、`B6` 及其随机组合实例。
  - `R4.2` 组合方式：在各基边界参数有效前提下，若出现“无来源直调依赖基、`B6` 非末步、删除后继续操作同一目标、跨轮串用失效标识”等情况，判定为无效组合。
  - `R4.3` 失效结论：`N1`。
  - `R4.4` 边界绑定：`POSTCOLUMNUPDATE + DELETECOLUMNID + DELETECOLUMNS + GETCOLUMNSALL + GETCOLUMNSSEARCH + POSTCOLUMNADD`。

- `R5` 分层与判定优先级原则（补充）
  - `R5.1` 参与基：`B1`、`B2`、`B3`、`B4`、`B5`、`B6` 及其组合。
  - `R5.2` 组合方式：执行顺序固定为“先 `R1/R2`（单元）后 `R3/R4`（混沌集成）”；仅当 `R1/R2` 通过时才进入组合测试；冲突样例按无效优先（`R4 > R3`）。
  - `R5.3` 输出能力：`C1 + C2 + C3 + C4 + C5 + C6` 的判定链路稳定可复验。
  - `R5.4` 边界绑定：`GETCOLUMNSALL + GETCOLUMNSSEARCH + POSTCOLUMNADD + POSTCOLUMNUPDATE + DELETECOLUMNID + DELETECOLUMNS`。

- `R6` 覆盖与稳定性补充原则（补充）
  - `R6.1` 参与基：`{B1, B2, B3, B4, B5, B6}` 的有效组合集合。
  - `R6.2` 组合方式：固定 `seed`、固定轮次（建议 `N>=30`）执行混沌测试；要求每个 `B*` 至少一次有效命中，且关键链路“新增→查询→更新→删除”可稳定复现。
  - `R6.3` 输出能力：`C1 + C2 + C3 + C4 + C5 + C6` 的覆盖充分性与稳定性。
  - `R6.4` 边界绑定：`GETCOLUMNSALL + GETCOLUMNSSEARCH + POSTCOLUMNADD + POSTCOLUMNUPDATE + DELETECOLUMNID + DELETECOLUMNS`。

## 5. 验证（Verification）

- `V1` 基边界有效原则验证（对应 `R1`）：通过条件：`B1~B6` 在边界有效样例下均返回 `2000`；失败原因：任一样例返回 `!=2000`；证据：单 API 有效样例请求/响应对照表。

- `V2` 基边界无效原则验证（对应 `R2`）：通过条件：`B1~B6` 在单 API 非法参数样例下均返回 `!=2000`；失败原因：任一样例返回 `2000`；证据：单 API 无效样例请求/响应对照表。

- `V3` 随机有效组合原则验证（对应 `R3`）：通过条件：命中的有效组合均满足 `R3.2` 约束，且全部返回 `2000`；失败原因：有效组合约束未满足或任一调用返回 `!=2000`；证据：有效组合命中统计、标识溯源报告与响应日志。

- `V4` 无效组合原则验证（对应 `R4`）：通过条件：在各基边界有效前提下，触发 `R4.2` 的组合均返回 `!=2000`；失败原因：任一无效组合返回 `2000`；证据：无效组合样例集、规则命中记录与响应日志。

- `V5` 分层与门槛验证（对应 `R5`）：通过条件：执行严格遵循“先 `V1/V2` 后 `V3/V4`”，且仅当 `V1/V2` 通过才进入组合测试；失败原因：阶段串扰、样例污染或门槛失效；证据：测试流水线执行序列与样例分层清单。

- `V6` 能力导出与问题分层校验：通过条件：`C1~C6` 均有有效规则导出链，且失败样例可区分为“结构定义问题”与“实例配置问题”；失败原因：能力未覆盖或归因不可分层；证据：`R -> C` 覆盖报告与失败分类报告。

- `V7` 覆盖率验证（对应 `R6`）：通过条件：有效组合类型命中完整，且每个 `B*` 至少被有效执行一次；失败原因：组合类型缺失或基长期未命中；证据：组合命中统计与覆盖率报告。

- `V8` 稳定性与顺序闭环验证（对应 `R6`）：通过条件：固定 `seed` 的多轮随机执行中“新增→查询→更新→删除”关键链路顺序稳定且有效组合成功率为 `100%`；失败原因：顺序漂移、偶发失败或数据污染；证据：轮次执行序列、成功率统计与失败归因报告。

- `V9` 结论判定：通过条件：`V1~V8` 全部通过；失败原因：任一验证项失败；证据：汇总判定单（通过/失败、原因、证据路径）。
