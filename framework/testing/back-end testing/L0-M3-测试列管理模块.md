# 测试列管理模块:TestColumnsManagement

@framework

## 1. 能力声明（Capability Statement）

- `C1` 测试获取所有列列表能力：测试系统一次性拉取并返回当前所有已配置的列定义数据集合的能力。
- `C2` 测试查询列能力：测试根据特定参数进行模糊匹配或精确过滤，从而查询目标列的能力。
- `C3` 测试新增列能力：测试向系统中添加新的列属性配置或字段元数据的能力。
- `C4` 测试更新列配置能力：测试对系统中已存在的列属性进行信息修改和保存的能力。
- `C5` 测试删除单列记录能力：测试从系统中精确移除单个列定义数据的能力。
- `C6` 测试批量删除列记录能力：测试一次性从系统中清除多条列定义数据的能力。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：不负责实际行级业务数据的存储与处理。

## 2. 边界定义（Boundary / 参数）

- `GETCOLUMNSALL` 边界：is_banned (属性: 可选，类型: boolean | (boolean | null)) 来源：`C1`。
- `GETCOLUMNSSEARCH` 边界：page (属性: 可选，类型: integer，默认值: 1，有效范围:≥1)
                          limit (属性: 可选，类型: integer，默认值: 20，有效范围:≥1)
                          column_name (属性: 可选，类型: string | (string | null))
                          column_alias (属性: 可选，类型: string | (string | null))
                          is_banned (属性: 可选，类型: boolean | (boolean | null)) 来源：`C2`。
- `POSTCOLUMNADD` 边界：column_name (属性: 必选，类型: string)
                       column_alias (属性: 必选，类型: string)
                       is_banned (属性: 必选，类型: boolean)
                       column_type (属性: 必选，类型: string)
                       metadata_json (属性: 必选，类型: object) 来源：`C3`。
- `POSTCOLUMNUPDATE` 边界：id (属性: 必选，类型: integer, 有效范围: 已存在的列id)
                          column_name (属性: 可选，类型: string | (string | null)，有效范围: 长度≥1)
                          column_alias (属性: 可选，类型: string | (string | null)，有效范围: 长度≥1)
                          is_banned (属性: 可选，类型: boolean | (boolean | null))
                          column_type (属性: 可选，类型: string | (string | null)，有效范围: 长度≥1)
                          metadata_json (属性: 可选，类型: object) 来源：`C4`。
- `DELETECOLUMNID` 边界：id (属性: 必选，类型: integer, 有效范围: 已存在的列id) 来源：`C5`。
- `DELETECOLUMNS` 边界：无，来源：`C6`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 基础可达组合
  - `R1.1` 参与基：`B1`、`B2`。
  - `R1.2` 组合方式：`B1`（全量列表）与 `B2`（条件查询）可独立调用，作为列管理组合链路入口。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`GETCOLUMNSALL + GETCOLUMNSSEARCH`。

- `R2` id 依赖组合
  - `R2.1` 参与基：`B1 + B4`、`B1 + B5`、`B2 + B4`、`B2 + B5`、`B3 + B4`、`B3 + B5`。
  - `R2.2` 组合方式：`B4/B5` 必须依赖有效 `id`；`id` 来源限定为同轮 `B1` 返回列表选定记录、同轮 `B2` 返回列表选定记录，或同轮 `B3` 新增成功返回值。
  - `R2.3` 输出能力：`C4 + C5`。
  - `R2.4` 边界绑定：`GETCOLUMNSALL + GETCOLUMNSSEARCH + POSTCOLUMNADD + POSTCOLUMNUPDATE + DELETECOLUMNID`。

- `R3` 增改查闭环组合
  - `R3.1` 参与基：`B3 + B2`、`B3 + B4 + B2`。
  - `R3.2` 组合方式：先通过 `B3` 建立新增列，再通过 `B4`（可选）执行更新，最终用 `B2` 按 `column_name/column_alias/is_banned` 回查验证，形成“新增-更新-查询”闭环。
  - `R3.3` 输出能力：`C2 + C3 + C4`。
  - `R3.4` 边界绑定：`POSTCOLUMNADD + POSTCOLUMNUPDATE + GETCOLUMNSSEARCH`。

- `R4` 删除收敛组合
  - `R4.1` 参与基：`B1 + B5`、`B2 + B5`、`B1 + B6`、`B2 + B6`、`B3 + B5`、`B3 + B5 + B6`。
  - `R4.2` 组合方式：单列删除走 `B5`，批量清理走 `B6`；同轮同时出现 `B5` 与 `B6` 时，`B6` 必须作为该轮最后一步，用于收敛残留数据。
  - `R4.3` 输出能力：`C5 + C6`。
  - `R4.4` 边界绑定：`GETCOLUMNSALL + GETCOLUMNSSEARCH + POSTCOLUMNADD + DELETECOLUMNID + DELETECOLUMNS`。

- `R5` 随机有效组合约束
  - `R5.1` 参与基：`{B1, B2, B3, B4, B5, B6}` 的随机子集。
  - `R5.2` 组合方式：合法随机组合必须满足：
  - `R5.2.a` 含 `B4/B5` 时，必须存在同轮可追溯 `id` 来源。
  - `R5.2.b` 含 `B6` 时，`B6` 必须最后执行。
  - `R5.2.c` 同轮含 `B3 + B4/B5` 时，`B4/B5` 的目标优先绑定当轮 `B3` 新建记录或当轮查询选中记录，不允许无来源漂移。
  - `R5.2.d` 同轮执行 `B5` 后，不得继续对同一 `id` 调用 `B4/B5`。
  - `R5.3` 输出能力：在约束下实现对 `C1~C6` 的高覆盖随机组合。
  - `R5.4` 边界绑定：`GETCOLUMNSALL + GETCOLUMNSSEARCH + POSTCOLUMNADD + POSTCOLUMNUPDATE + DELETECOLUMNID + DELETECOLUMNS`。

- `R6` 无效组合判定
  - `R6.1` 参与基：`B4`、`B5`、`B6` 或任意包含这些依赖基的组合。
  - `R6.2` 组合方式：无 `id` 来源直接调用 `B4/B5`、`B6` 非末位执行、删除后继续操作已删除目标，均判定为无效组合。
  - `R6.3` 失效结论：`N1`。
  - `R6.4` 边界绑定：`POSTCOLUMNUPDATE + DELETECOLUMNID + DELETECOLUMNS + GETCOLUMNSALL + GETCOLUMNSSEARCH + POSTCOLUMNADD`。

## 5. 验证（Verification）

- `V1` 规则预校验：通过条件：每轮组合满足 `R1~R5` 且不触发 `R6`；失败原因：依赖链、顺序或目标约束不成立；证据：规则判定日志。

- `V2` 参数边界校验：通过条件：请求参数与请求体字段满足边界（含 `id/page/limit` 约束与字段类型）；失败原因：字段缺失、类型错误、越界；证据：请求快照与字段校验报告。

- `V3` id 来源追踪校验：通过条件：`B4/B5` 使用的 `id` 可追溯到同轮 `B1/B2/B3` 输出；失败原因：`id` 来源不明、跨轮串用或目标漂移；证据：`id` 溯源映射表。

- `V4` 调用结果校验：通过条件：有效组合中参与基调用状态码均为 `2000`；失败原因：接口调用失败或返回码异常；证据：接口响应日志。

- `V5` 删除顺序校验：通过条件：含 `B6` 组合满足“`B6` 末位执行”，且删除后无非法续调；失败原因：顺序颠倒或删除后继续操作目标；证据：执行步骤序列。

- `V6` 能力导出校验：通过条件：`C1~C6` 均至少由一条合法规则导出；失败原因：存在能力未被有效组合覆盖；证据：`R -> C` 导出覆盖报告。

- `V7` 随机覆盖校验：通过条件：随机执行覆盖全部有效组合类型，且 `B1~B6` 至少各执行一次；失败原因：覆盖不全或组合分布偏斜；证据：组合命中统计。

- `V8` 稳定性校验：通过条件：多轮随机执行中有效组合成功率为 `100%`；失败原因：依赖漂移、顺序漂移或数据污染；证据：轮次成功率与失败归因报告。

- `V9` 结论判定：通过条件：`V1~V8` 全部通过；失败原因：任一验证项失败；证据：汇总判定单（通过/失败、原因、证据路径）。
