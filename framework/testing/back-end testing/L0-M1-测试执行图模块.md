# 测试执行图模块:TestExecutionDiagram

@framework

## 1. 能力声明（Capability Statement）

- `C1` 测试创建/启动执行任务能力：测试系统在后台生成并启动一个新的执行任务的能力。
- `C2` 测试查询执行记录列表能力：测试获取历史的或正在运行的执行任务列表的能力，常支持分页以及根据状态、时间等条件进行过滤筛选。
- `C3` 测试删除执行记录能力：测试清理或取消执行任务的能力。
- `C4` 测试获取指定任务的完成状态/结果能力：测试查询某个特定任务是否已经完成，或者拉取该任务执行完毕后的最终结果数据的能力。
- `C5` 测试查询单次执行的详情或事件日志能力：测试获取该次任务的详细信息的能力，常获取该任务在运行过程中的步骤流转、状态变更事件或详细日志，用于前端展示执行图的实时进度或排查错误。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：本模块不负责无效参数/无效组合场景下具体错误码语义的一致性。

## 2. 边界定义（Boundary / 参数）

- `POSTEXECUTE` 边界：graph_id (属性: 必选，类型: string) 
                     data_src (属性: 必选，类型: list[ExecuteInputParamsDTO]，有效范围: 至少 1 项)
                     data_src[].node_id (属性: 必选，类型: string)
                     data_src[].file_id (属性: 必选，类型: string)
                     data_src[].file_name (属性: 必选，类型: string) 来源：`C1`。
- `GETEXECUTE` 边界：limit (属性: 可选，类型: integer, 默认值: 10，有效范围:[1,100])
                    page (属性: 可选，类型: integer，默认值: 1，有效范围:≥1)
                    file_name (属性: 可选，类型: string | (string | null)，有效范围：默认所有文件) 
                    graph_name (属性: 可选，类型: string | (string | null)，有效范围：默认所有图) 
                    state (属性: 可选，类型: string | (string | null)，默认值:all) 
                    graph_id (属性: 可选，类型: string | (string | null)，有效范围：默认所有图) 
                    device_name (属性: 可选，类型: string | (string | null)，有效范围：默认所有设备) 来源：`C2`。
- `DELETEEXECUTE` 边界：execute_id_list (属性: 必选，类型: list[string], 有效范围: 已存在的 execute_id 集合且非空) 来源：`C3`。
- `GETEXECUTECOMPLETEDID` 边界：execute_id (属性: 必选，类型: string, 有效范围: 已存在的execute_id) 来源：`C4`。
- `GETEXECUTETID` 边界：execute_id (属性: 必选，类型: string, 有效范围: 已存在的execute_id) 来源：`C5`。

## 3. 最小可行基（Minimum Viable Bases）
    
- `B1` 测试创建/启动执行任务基：
       由

         请求体：graph_id

                data_src[].node_id

                data_src[].file_name

       构成的测试创建/启动执行任务结构。来源：`C1 + POSTEXECUTE`。

- `B2` 测试查询执行记录列表基：
       由
         调用接口: GET /api/execute
         参数：limit 
               page
               file_name 
               graph_name
               state
               graph_id
               device_name
         请求体：无
         调用结果：调用成功状态码返回2000，调用失败状态码返回!2000
       构成的测试查询执行记录列表结构。来源：`C2 + GETEXECUTE`。

- `B3` 测试删除执行记录基：
       由
         调用接口: DELETE /api/execute
         参数：无
         请求体：execute_id_list
         调用结果：调用成功状态码返回2000，调用失败状态码返回!2000
       构成的测试删除执行记录结构。来源：`C3 + DELETEEXECUTE`。


- `B4` 测试获取指定任务的完成状态/结果基：
       由
         调用接口: GET /api/execute/completed/{execute_id}
         参数：execute_id

       构成的测试获取指定任务的完成状态/结果结构。来源：`C4 + GETEXECUTECOMPLETEDID`。

- `B5` 测试查询单次执行的详情或事件日志基：
       由
         调用接口: GET /api/execute/{execute_id}
         参数：execute_id
         请求体：无

       构成的测试查询单次执行的详情或事件日志结构。来源：`C5 + GETEXECUTETID`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 基边界有效原则（单元测试）
  - `R1.1` 参与基：`B1`、`B2`、`B3`、`B4`、`B5`。
  - `R1.2` 组合方式：每个基独立调用并满足第 2 节边界有效条件；对依赖 execute_id 的基，使用预置有效标识样例进行单基验证。
  - `R1.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R1.4` 边界绑定：`POSTEXECUTE + GETEXECUTE + DELETEEXECUTE + GETEXECUTECOMPLETEDID + GETEXECUTETID`。

- `R2` 基边界无效原则（单元测试）
  - `R2.1` 参与基：`B1`、`B2`、`B3`、`B4`、`B5`。
  - `R2.2` 组合方式：每个基均独立调用，并人为构造“单 API 非法参数”（缺字段、类型错误、越界、无效/空标识）触发边界无效。
  - `R2.3` 失效结论：`N1`。
  - `R2.4` 边界绑定：`POSTEXECUTE + GETEXECUTE + DELETEEXECUTE + GETEXECUTECOMPLETEDID + GETEXECUTETID`。

- `R3` 随机有效组合原则（混沌集成测试）
  - `R3.1` 参与基：`{B1, B2, B3, B4, B5}` 的随机子集。
  - `R3.2` 组合方式：在 `R1` 已通过且已剔除单 API 非法参数样例前提下，合法随机组合必须满足：
  - `R3.2.a` 含 `B3/B4/B5` 时，`execute_id` 来源必须可追溯到同轮 `B1` 返回值或同轮 `B2` 列表选定记录。
  - `R3.2.b` 含 `B3` 时，`execute_id_list` 非空且列表元素均可追溯到同轮有效来源。
  - `R3.2.c` 同轮含删除 `B3` 与查询 `B4/B5` 时，先查后删，且删除作为该目标链路末步。
  - `R3.3` 输出能力：在约束下实现对 `C1~C5` 的高覆盖随机有效组合。
  - `R3.4` 边界绑定：`POSTEXECUTE + GETEXECUTE + DELETEEXECUTE + GETEXECUTECOMPLETEDID + GETEXECUTETID`。

- `R4` 无效组合原则（混沌集成测试）
  - `R4.1` 参与基：`B3`、`B4`、`B5` 及其随机组合实例。
  - `R4.2` 组合方式：在各基边界参数有效前提下，若出现“无来源直调依赖基、先删后查、跨轮串用失效 `execute_id`、删除后继续查询已删目标”等情况，判定为无效组合。
  - `R4.3` 失效结论：`N1`。
  - `R4.4` 边界绑定：`DELETEEXECUTE + GETEXECUTECOMPLETEDID + GETEXECUTETID + POSTEXECUTE + GETEXECUTE`。

- `R5` 分层与判定优先级原则（补充）
  - `R5.1` 参与基：`B1`、`B2`、`B3`、`B4`、`B5` 及其组合。
  - `R5.2` 组合方式：执行顺序固定为“先 `R1/R2`（单元）后 `R3/R4`（混沌集成）”；仅当 `R1/R2` 通过时才进入组合测试；冲突样例按无效优先（`R4 > R3`）。
  - `R5.3` 输出能力：`C1 + C2 + C3 + C4 + C5` 的判定链路稳定可复验。
  - `R5.4` 边界绑定：`POSTEXECUTE + GETEXECUTE + DELETEEXECUTE + GETEXECUTECOMPLETEDID + GETEXECUTETID`。

- `R6` 覆盖与稳定性补充原则（补充）
  - `R6.1` 参与基：`{B1, B2, B3, B4, B5}` 的有效组合集合。
  - `R6.2` 组合方式：固定 `seed`、固定轮次（建议 `N>=30`）执行混沌测试；要求每个 `B*` 至少一次有效命中，且关键链路“创建→列表→详情/完成态→删除”可稳定复现。
  - `R6.3` 输出能力：`C1 + C2 + C3 + C4 + C5` 的覆盖充分性与稳定性。
  - `R6.4` 边界绑定：`POSTEXECUTE + GETEXECUTE + DELETEEXECUTE + GETEXECUTECOMPLETEDID + GETEXECUTETID`。

## 5. 验证（Verification）

- `V1` 基边界有效原则验证（对应 `R1`）：通过条件：`B1~B5` 在边界有效样例下均返回 `2000`；失败原因：任一样例返回 `!=2000`；证据：单 API 有效样例请求/响应对照表。

- `V2` 基边界无效原则验证（对应 `R2`）：通过条件：`B1~B5` 在单 API 非法参数样例下均返回 `!=2000`；失败原因：任一样例返回 `2000`；证据：单 API 无效样例请求/响应对照表。

- `V3` 随机有效组合原则验证（对应 `R3`）：通过条件：命中的有效组合均满足 `R3.2` 约束，且全部返回 `2000`；失败原因：有效组合约束未满足或任一调用返回 `!=2000`；证据：有效组合命中统计、标识溯源报告与响应日志。

- `V4` 无效组合原则验证（对应 `R4`）：通过条件：在各基边界有效前提下，触发 `R4.2` 的组合均返回 `!=2000`；失败原因：任一无效组合返回 `2000`；证据：无效组合样例集、规则命中记录与响应日志。

- `V5` 分层与门槛验证（对应 `R5`）：通过条件：执行严格遵循“先 `V1/V2` 后 `V3/V4`”，且仅当 `V1/V2` 通过才进入组合测试；失败原因：阶段串扰、样例污染或门槛失效；证据：测试流水线执行序列与样例分层清单。

- `V6` 能力导出与问题分层校验：通过条件：`C1~C5` 均有有效规则导出链，且失败样例可区分为“结构定义问题”与“实例配置问题”；失败原因：能力未覆盖或归因不可分层；证据：`R -> C` 覆盖报告与失败分类报告。

- `V7` 覆盖率验证（对应 `R6`）：通过条件：有效组合类型命中完整，且每个 `B*` 至少被有效执行一次；失败原因：组合类型缺失或基长期未命中；证据：组合命中统计与覆盖率报告。

- `V8` 稳定性与顺序闭环验证（对应 `R6`）：通过条件：固定 `seed` 的多轮随机执行中“创建→列表→详情/完成态→删除”关键链路顺序稳定且有效组合成功率为 `100%`；失败原因：顺序漂移、偶发失败或数据污染；证据：轮次执行序列、成功率统计与失败归因报告。

- `V9` 结论判定：通过条件：`V1~V8` 全部通过；失败原因：任一验证项失败；证据：汇总判定单（通过/失败、原因、证据路径）。
