# 测试执行图模块:TestExecutionDiagram

@framework

## 1. 能力声明（Capability Statement）

- `C1` 测试创建/启动执行任务能力：测试系统在后台生成并启动一个新的执行任务的能力。
- `C2` 测试查询执行记录列表能力：测试获取历史的或正在运行的执行任务列表的能力，常支持分页以及根据状态、时间等条件进行过滤筛选。
- `C3` 测试删除执行记录能力：测试清理或取消执行任务的能力。
- `C4` 测试获取指定任务的完成状态/结果能力：测试查询某个特定任务是否已经完成，或者拉取该任务执行完毕后的最终结果数据的能力。
- `C5` 测试查询单次执行的详情或事件日志能力：测试获取该次任务的详细信息的能力，常获取该任务在运行过程中的步骤流转、状态变更事件或详细日志，用于前端展示执行图的实时进度或排查错误。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：不负责底层具体业务逻辑的实际运算与处理。

## 2. 边界定义（Boundary / 参数）

- `POSTEXECUTE` 边界：graph_id (属性: 必选，类型: string) 
                     data_src (属性: 必选，类型: list[ExecuteInputParamsDTO]，有效范围: 至少 1 项)
                     data_src[].node_id (属性: 必选，类型: string)
                     data_src[].file_id (属性: 必选，类型: string)
                     data_src[].file_name (属性: 必选，类型: string) 来源：`C1`。
- `GETEXECUTE` 边界：limit (属性: 可选，类型: integer, 默认值: 10，有效范围:[1,100])
                    page (属性: 可选，类型: integer，默认值: 1，有效范围:≥1)
                    file_name (属性: 可选，类型: string | (string | null)，有效范围：默认所有文件) 
                    graph_name (属性: 可选，类型: string | (string | null)，有效范围：默认所有图) 
                    state (属性: 可选，类型: string | (string | null)，默认值:all) 
                    graph_id (属性: 可选，类型: string | (string | null)，有效范围：默认所有图) 
                    device_name (属性: 可选，类型: string | (string | null)，有效范围：默认所有设备) 来源：`C2`。
- `DELETEEXECUTE` 边界：execute_id_list (属性: 必选，类型: list[string], 有效范围: 已存在的 execute_id 集合且非空) 来源：`C3`。
- `GETEXECUTECOMPLETEDID` 边界：execute_id (属性: 必选，类型: string, 有效范围: 已存在的execute_id) 来源：`C4`。
- `GETEXECUTETID` 边界：execute_id (属性: 必选，类型: string, 有效范围: 已存在的execute_id) 来源：`C5`

## 3. 最小可行基（Minimum Viable Bases）
    
- `B1` 测试创建/启动执行任务基：
       由
         调用接口: POST /api/execute
         参数：无
         请求体：graph_id
                data_src
                data_src[].node_id
                data_src[].file_id
                data_src[].file_name
         调用结果：调用成功状态码返回2000，且响应中应包含执行记录id（execute_id）；调用失败状态码返回!2000
       构成的测试创建/启动执行任务结构。来源：`C1 + POSTEXECUTE`。

- `B2` 测试查询执行记录列表基：
       由
         调用接口: GET /api/execute
         参数：limit 
               page
               file_name 
               graph_name
               state
               graph_id
               device_name
         请求体：无
         调用结果：调用成功状态码返回2000，调用失败状态码返回!2000
       构成的测试查询执行记录列表结构。来源：`C2 + GETEXECUTE`。

- `B3` 测试删除执行记录基：
       由
         调用接口: DELETE /api/execute
         参数：无
         请求体：execute_id_list
         调用结果：调用成功状态码返回2000，调用失败状态码返回!2000
       构成的测试删除执行记录结构。来源：`C3 + DELETEEXECUTE`。


- `B4` 测试获取指定任务的完成状态/结果基：
       由
         调用接口: GET /api/execute/completed/{execute_id}
         参数：execute_id
         请求体：无
         调用结果：调用成功状态码返回2000，调用失败状态码返回!2000
       构成的测试获取指定任务的完成状态/结果结构。来源：`C4 + GETEXECUTECOMPLETEDID`。

- `B5` 测试查询单次执行的详情或事件日志基：
       由
         调用接口: GET /api/execute/{execute_id}
         参数：execute_id
         请求体：无
         调用结果：调用成功状态码返回2000，调用失败状态码返回!2000
       构成的测试查询单次执行的详情或事件日志结构。来源：`C5 + GETEXECUTETID`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 基础起点组合
  - `R1.1` 参与基：`B1`、`B2`。
  - `R1.2` 组合方式：`B1`（创建执行）与 `B2`（列表查询）可独立调用，作为后续 `execute_id` 依赖链的入口。
  - `R1.3` 输出能力：`C1`、`C2`。
  - `R1.4` 边界绑定：`POSTEXECUTE + GETEXECUTE`。

- `R2` execute_id 依赖组合
  - `R2.1` 参与基：`B3`、`B4`、`B5`。
  - `R2.2` 组合方式：`B4/B5` 使用 path `execute_id`，`B3` 使用 body `execute_id_list`；三者 id 必须来自同轮 `B1` 返回值或同轮 `B2` 选定记录。
  - `R2.3` 输出能力：`C3 + C4 + C5`。
  - `R2.4` 边界绑定：`DELETEEXECUTE + GETEXECUTECOMPLETEDID + GETEXECUTETID`。

- `R3` 可观测执行链路组合
  - `R3.1` 参与基：`B1 + B2 + B5`、`B1 + B2 + B4`、`B1 + B2 + B5 + B4`。
  - `R3.2` 组合方式：先创建执行，再在列表检出目标任务，再查询事件与完成态，形成单任务多视角追踪链路。
  - `R3.3` 输出能力：`C1 + C2 + C4 + C5`。
  - `R3.4` 边界绑定：`POSTEXECUTE + GETEXECUTE + GETEXECUTETID + GETEXECUTECOMPLETEDID`。

- `R4` 删除收尾闭环组合
  - `R4.1` 参与基：`B1 + B2 + B5 + B4 + B3` 或任意包含 `B3` 的合法组合。
  - `R4.2` 组合方式：`B3` 必须最后执行；`execute_id_list` 必须非空，且必须来自本轮上下文中已确认的目标任务集。
  - `R4.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R4.4` 边界绑定：`POSTEXECUTE + GETEXECUTE + GETEXECUTETID + GETEXECUTECOMPLETEDID + DELETEEXECUTE`。

- `R5` 随机有效组合约束
  - `R5.1` 参与基：`{B1, B2, B3, B4, B5}` 的随机子集。
  - `R5.2` 组合方式：合法组合必须满足：
  - `R5.2.a` 含 `B4/B5` 时，必须有同轮有效 `execute_id` 来源。
  - `R5.2.b` 含 `B3` 时，`execute_id_list` 必须非空且 `B3` 最后执行。
  - `R5.2.c` 同轮 `B4` 与 `B5` 共存时，必须针对同一 `execute_id`。
  - `R5.2.d` 同轮若含 `B3` 且含 `B4/B5`，则删除列表必须覆盖该轮已查询目标 id。
  - `R5.3` 输出能力：在约束下实现对 `C1~C5` 的高覆盖随机组合。
  - `R5.4` 边界绑定：`POSTEXECUTE + GETEXECUTE + DELETEEXECUTE + GETEXECUTECOMPLETEDID + GETEXECUTETID`。

- `R6` 无效组合判定
  - `R6.1` 参与基：`B3`、`B4`、`B5` 或任意包含这些依赖基的组合。
  - `R6.2` 组合方式：无 id 来源直接调用 `B3/B4/B5`，删除早于查询执行，或跨轮串用 id，均判定为无效组合。
  - `R6.3` 失效结论：`N1`。
  - `R6.4` 边界绑定：`DELETEEXECUTE + GETEXECUTECOMPLETEDID + GETEXECUTETID + POSTEXECUTE + GETEXECUTE`。

## 5. 验证（Verification）

- `V1` 规则预校验：通过条件：组合满足 `R1~R5` 且不触发 `R6`；失败原因：依赖、顺序或来源约束不成立；证据：规则判定日志。

- `V2` 参数形态校验：通过条件：`B4/B5.execute_id` 为非空字符串，`B3.execute_id_list` 为非空 `list[string]`；失败原因：参数缺失或形态错误；证据：请求构造快照。

- `V3` 来源追踪校验：通过条件：`B3/B4/B5` 使用的 id 均可追溯到同轮 `B1` 返回值或 `B2` 选定记录；失败原因：跨轮串用或来源不明；证据：id 来源映射表。

- `V4` 调用结果校验：通过条件：有效组合中的 `B1~B5` 调用状态码均为 `2000`；失败原因：接口调用失败或返回码异常；证据：响应日志。

- `V5` 顺序闭环校验：通过条件：含 `B3` 组合满足“查询在前、删除在后”；失败原因：删除顺序违规；证据：步骤执行序列。

- `V6` 能力导出校验：通过条件：`R1~R5` 至少各命中一次，且 `C1~C5` 均有对应导出链；失败原因：能力未被规则导出；证据：`R -> C` 覆盖报告。

- `V7` 随机覆盖校验：通过条件：随机执行覆盖全部有效组合类型，且每个 `B*` 至少被执行一次；失败原因：覆盖不全；证据：组合命中统计。

- `V8` 稳定性校验：通过条件：多轮随机执行中有效组合成功率为 `100%`；失败原因：依赖漂移、顺序漂移、环境污染；证据：轮次成功率与失败归因报告。

- `V9` 结论判定：通过条件：`V1~V8` 全部通过；失败原因：任一验证项失败；证据：汇总判定单（通过/失败、原因、证据路径）。
