# 测试文件上传模块:TestFileUpload

@framework

## 1. 能力声明（Capability Statement）

- `C1` 测试文件上传能力：测试将用户本地的文件数据发送并保存到服务器上的能力。
- `C2` 测试查询所有文件列表能力：测试获取服务器上已经上传的文件列表的能力。
- `C3` 测试查询单个特定文件能力：测试系统会根据特定文件的哈希值能返回单个特定文件的详细属性或内容的能力。
- `C4` 测试删除文件能力：测试系统会根据传入指定的 ID 把对应的文件从服务器中彻底删除的能力。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：本模块不负责无效参数/无效组合场景下具体错误码语义的一致性。

## 2. 边界定义（Boundary / 参数）

- `POSTFILE` 边界：file (属性: 必选，类型: string($binary), 有效范围: 格式txt|csv 编码utf-8) 
                  table_name (属性: 可选，类型: string | (string | null))
                  device_name (属性: 必选，类型: string，有效范围: 已存在的device_name)
                  skip（属性: 可选，类型: integer, 有效范围:[`0`,`上传文件总行数-1`]）来源：`C1`。
- `GETFILE` 边界：page (属性: 可选，类型: integer，默认值: 1，有效范围:≥1)
                  limit (属性: 可选，类型: integer, 默认值: 20，有效范围:≥1)
                  table_name (属性: 可选，类型: string | (string | null))
                  user（属性: 可选，类型: string | (string | null)）
                  device_name（属性: 可选，类型: string | (string | null)）来源：`C2`。
- `GETSINGLEFILE` 边界：hash_table_name (属性: 必选，类型: string, 有效范围: 已存在的hash_table_name) 来源：`C3`。
- `DELETEFILE` 边界：id (属性: 必选，类型: string, 有效范围: 已存在的文件 id) 来源：`C4`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 基边界有效原则（单元测试）
  - `R1.1` 参与基：`B1`、`B2`、`B3`、`B4`。
  - `R1.2` 组合方式：每个基独立调用并满足第 2 节边界有效条件；对依赖 hash_table_name/id 的基，使用预置有效标识样例进行单基验证。
  - `R1.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R1.4` 边界绑定：`POSTFILE + GETFILE + GETSINGLEFILE + DELETEFILE`。

- `R2` 基边界无效原则（单元测试）
  - `R2.1` 参与基：`B1`、`B2`、`B3`、`B4`。
  - `R2.2` 组合方式：每个基独立调用，并构造“单 API 非法参数”（缺字段、类型错误、越界、无效标识）触发边界无效。
  - `R2.3` 失效结论：`N1`。
  - `R2.4` 边界绑定：`POSTFILE + GETFILE + GETSINGLEFILE + DELETEFILE`。

- `R3` 随机有效组合原则（混沌集成测试）
  - `R3.1` 参与基：`{B1, B2, B3, B4}` 的随机子集。
  - `R3.2` 组合方式：在 `R1` 已通过且剔除单 API 非法参数样例前提下，合法随机组合需满足：
  - `R3.2.a` 含 `B3/B4` 时，依赖标识仅可来源于同轮 `B1` 返回值或同轮 `B2` 结果集选定记录。
  - `R3.2.b` 同轮含 `B3 + B4` 时必须先查后删，且删除为该目标链路末步。
  - `R3.2.c` 不允许跨轮复用已删除/失效标识，不允许标识目标漂移。
  - `R3.3` 输出能力：在约束下实现 `C1~C4` 的高覆盖有效组合。
  - `R3.4` 边界绑定：`POSTFILE + GETFILE + GETSINGLEFILE + DELETEFILE`。

- `R4` 无效组合原则（混沌集成测试）
  - `R4.1` 参与基：`B3`、`B4` 及其随机组合实例。
  - `R4.2` 组合方式：在各基边界参数有效前提下，若出现“无来源直调、先删后查、删除后继续查询已删目标、跨轮串用失效标识”等情况，判定为无效组合。
  - `R4.3` 失效结论：`N1`。
  - `R4.4` 边界绑定：`GETSINGLEFILE + DELETEFILE + POSTFILE + GETFILE`。

- `R5` 分层与判定优先级原则（补充）
  - `R5.1` 参与基：`B1`、`B2`、`B3`、`B4` 及其组合。
  - `R5.2` 组合方式：执行顺序固定为“先 `R1/R2`（单元）后 `R3/R4`（混沌集成）”；仅当 `R1/R2` 通过时进入组合测试；冲突样例按无效优先（`R4 > R3`）。
  - `R5.3` 输出能力：`C1 + C2 + C3 + C4` 的判定链路稳定可复验。
  - `R5.4` 边界绑定：`POSTFILE + GETFILE + GETSINGLEFILE + DELETEFILE`。

- `R6` 覆盖与稳定性补充原则（补充）
  - `R6.1` 参与基：`{B1, B2, B3, B4}` 的有效组合集合。
  - `R6.2` 组合方式：固定 `seed`、固定轮次（建议 `N>=30`）执行混沌测试；要求每个 `B*` 至少一次有效命中，且关键链路“上传→列表→单查→删除”可稳定复现。
  - `R6.3` 输出能力：`C1 + C2 + C3 + C4` 的覆盖充分性与稳定性。
  - `R6.4` 边界绑定：`POSTFILE + GETFILE + GETSINGLEFILE + DELETEFILE`。

## 5. 验证（Verification）

- `V1` 基边界有效原则验证（对应 `R1`）：通过条件：`B1~B4` 在边界有效样例下均返回 `2000`；失败原因：任一样例返回 `!=2000`；证据：单 API 有效样例请求/响应对照表。

- `V2` 基边界无效原则验证（对应 `R2`）：通过条件：`B1~B4` 在单 API 非法参数样例下均返回 `!=2000`；失败原因：任一样例返回 `2000`；证据：单 API 无效样例请求/响应对照表。

- `V3` 随机有效组合原则验证（对应 `R3`）：通过条件：命中的有效组合均满足 `R3.2` 约束，且全部返回 `2000`；失败原因：有效组合约束未满足或任一调用返回 `!=2000`；证据：有效组合命中统计、标识溯源报告与响应日志。

- `V4` 无效组合原则验证（对应 `R4`）：通过条件：在各基边界有效前提下，触发 `R4.2` 的组合均返回 `!=2000`；失败原因：任一无效组合返回 `2000`；证据：无效组合样例集、规则命中记录与响应日志。

- `V5` 分层与门槛验证（对应 `R5`）：通过条件：执行严格遵循“先 `V1/V2` 后 `V3/V4`”，且仅当 `V1/V2` 通过才进入组合测试；失败原因：阶段串扰、样例污染或门槛失效；证据：测试流水线执行序列与样例分层清单。

- `V6` 能力导出与问题分层校验：通过条件：`C1~C4` 均有有效规则导出链，且失败样例可区分为“结构定义问题”与“实例配置问题”；失败原因：能力未覆盖或归因不可分层；证据：`R -> C` 覆盖报告与失败分类报告。

- `V7` 覆盖率验证（对应 `R6`）：通过条件：有效组合类型命中完整，且每个 `B*` 至少被有效执行一次；失败原因：组合类型缺失或基长期未命中；证据：组合命中统计与覆盖率报告。

- `V8` 稳定性与顺序闭环验证（对应 `R6`）：通过条件：固定 `seed` 的多轮随机执行中关键链路顺序稳定且有效组合成功率为 `100%`；失败原因：顺序漂移、偶发失败或数据污染；证据：轮次执行序列、成功率统计与失败归因报告。

- `V9` 结论判定：通过条件：`V1~V8` 全部通过；失败原因：任一验证项失败；证据：汇总判定单（通过/失败、原因、证据路径）。
