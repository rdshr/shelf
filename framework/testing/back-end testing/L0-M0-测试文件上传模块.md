# 测试文件上传模块:TestFileUpload

@framework

## 1. 能力声明（Capability Statement）

- `C1` 测试文件上传能力：测试将用户本地的文件数据发送并保存到服务器上的能力。
- `C2` 测试查询所有文件列表能力：测试获取服务器上已经上传的文件列表的能力。
- `C3` 测试查询单个特定文件能力：测试系统会根据特定文件的哈希值能返回单个特定文件的详细属性或内容的能力。
- `C4` 测试删除文件能力：测试系统会根据传入指定的 ID 把对应的文件从服务器中彻底删除的能力。

### 非职责声明（Non-Responsibility Statement）

- `N1` 非职责声明：不负责文件内容的深度业务解析与校验。

## 2. 边界定义（Boundary / 参数）

- `POSTFILE` 边界：file (属性: 必选，类型: string($binary), 有效范围: 格式txt|csv 编码utf-8) 
                  table_name (属性: 可选，类型: string | (string | null))
                  device_name (属性: 必选，类型: string，有效范围: 已存在的device_name)
                  skip（属性: 可选，类型: integer, 有效范围:[`0`,`上传文件总行数-1`]）来源：`C1`。
- `GETFILE` 边界：page (属性: 可选，类型: integer，默认值: 1，有效范围:≥1)
                  limit (属性: 可选，类型: integer, 默认值: 20，有效范围:≥1)
                  table_name (属性: 可选，类型: string | (string | null))
                  user（属性: 可选，类型: string | (string | null)）
                  device_name（属性: 可选，类型: string | (string | null)）来源：`C2`。
- `GETSINGLEFILE` 边界：hash_table_name (属性: 必选，类型: string, 有效范围: 已存在的hash_table_name) 来源：`C3`。
- `DELETEFILE` 边界：hash_table_name (属性: 必选，类型: string, 有效范围: 已存在的hash_table_name) 来源：`C4`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 基础可达组合
  - `R1.1` 参与基：`B1`、`B2`。
  - `R1.2` 组合方式：`B1`（上传）与 `B2`（列表查询）可独立调用，作为后续依赖链的起点。
  - `R1.3` 输出能力：`C1`、`C2`。
  - `R1.4` 边界绑定：`POSTFILE + GETFILE`。

- `R2` 标识依赖组合
  - `R2.1` 参与基：`B1 + B3`、`B1 + B4`、`B2 + B3`、`B2 + B4`。
  - `R2.2` 组合方式：`B3` 依赖 `hash_table_name`，`B4` 依赖 `id`；两者标识必须来自同轮 `B1` 返回值或同轮 `B2` 选定记录。
  - `R2.3` 输出能力：`C3`、`C4`。
  - `R2.4` 边界绑定：`GETSINGLEFILE + DELETEFILE + GETFILE + POSTFILE`。

- `R3` 可观测链路组合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：先上传，再在列表中检出，再查询单文件详情，形成单文件可追踪链路。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`POSTFILE + GETFILE + GETSINGLEFILE`。

- `R4` 生命周期闭环组合
  - `R4.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R4.2` 组合方式：在完成查询断言后执行删除，`B4` 必须作为该轮最后一步，形成“上传-查询-删除”闭环。
  - `R4.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R4.4` 边界绑定：`POSTFILE + GETFILE + GETSINGLEFILE + DELETEFILE`。

- `R5` 随机有效组合约束
  - `R5.1` 参与基：`{B1, B2, B3, B4}` 的随机子集。
  - `R5.2` 组合方式：合法组合必须满足：
  - `R5.2.a` 含 `B3` 时，必须有同轮有效 `hash_table_name` 来源。
  - `R5.2.b` 含 `B4` 时，必须有同轮有效 `id` 来源，且 `B4` 最后执行。
  - `R5.2.c` 同轮 `B3` 与 `B4` 共存时，必须先查询后删除。
  - `R5.3` 输出能力：在约束下实现对 `C1~C4` 的高覆盖随机组合。
  - `R5.4` 边界绑定：`POSTFILE + GETFILE + GETSINGLEFILE + DELETEFILE`。

- `R6` 无效组合判定
  - `R6.1` 参与基：`B3`、`B4` 或任意包含 `B3/B4` 的组合。
  - `R6.2` 组合方式：无标识来源直接调用 `B3/B4`，或删除早于查询执行，均为无效组合。
  - `R6.3` 失效结论：`N1`。
  - `R6.4` 边界绑定：`GETSINGLEFILE + DELETEFILE + POSTFILE + GETFILE`。
 
## 5. 验证（Verification）

- `V1` 规则预校验：通过条件：每轮组合满足 `R1~R5` 且不触发 `R6`；失败原因：组合含无效依赖或顺序违规；证据：组合生成记录与规则判定日志。

- `V2` 标识来源校验：通过条件：`B3` 的 `hash_table_name`、`B4` 的 `id` 可追溯到同轮 `B1/B2`；失败原因：标识缺失、跨轮复用或来源不明；证据：请求参数快照与上下文映射表。

- `V3` 调用结果校验：通过条件：有效组合中各基调用状态码均为 `2000`；失败原因：接口调用失败或返回码异常；证据：接口响应日志。

- `V4` 顺序闭环校验：通过条件：含 `B4` 组合满足“查询在前、删除在后”；失败原因：先删后查或删除后继续使用同轮目标；证据：步骤执行序列。

- `V5` 能力导出校验：通过条件：`R1~R5` 至少各命中一次，且 `C1~C4` 均有对应导出链；失败原因：能力未被任何有效规则导出；证据：`R -> C` 映射覆盖报告。

- `V6` 随机覆盖校验：通过条件：随机执行覆盖全部有效组合类型，且每个 `B*` 至少被执行一次；失败原因：覆盖偏斜或长期未命中关键组合；证据：组合命中统计。

- `V7` 稳定性校验：通过条件：多轮随机执行中有效组合成功率为 `100%`；失败原因：偶发依赖缺失、顺序漂移或数据污染；证据：轮次成功率与失败归因报告。

- `V8` 结论判定：通过条件：`V1~V7` 全部通过；失败原因：任一验证项失败；证据：汇总判定单（通过/失败、原因、证据路径）。
