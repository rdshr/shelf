# 文件库范围查询编排模块:ReviewWorkbenchFileLibrarySkeletonAndScopeQueryOrchestration

@framework

本模块定义文件库处理场景前半段是怎么一步一步接起来的，也就是“先进入统一平台中的文件库场景，再看范围树，确定当前范围，必要时通过范围树局部工具位做刷新、上传或局部目录动作，然后进入固定列表、设置查询、进入文件清单工作面、看到清单结果和反馈”。它位于 `review_workbench` 的编排层，关心的是文件库场景内部这些部分如何配合，而不是具体有哪些目录、字段或统计项。那些具体内容应在项目配置里被实例化，更具体的承载方式再落到 Code 层。

本模块建立在 `review_workbench.L0.M1~M5`、`review_workbench.L0.M10` 与外部领域 `review_workbench_backend.L0.M1` 这些领域原子之上，并默认承接 `review_workbench.L1.M0` 已经整理好的统一平台场景挂接结果。对于“范围树怎样确定当前范围，再带出固定列表、查询条件、文件清单工作面和统计结果”这件事，它向下复用外部领域 `review_workbench_backend.L0.M1` 的查询与结果集契约，不在这里重新定义请求和返回结构。

## 1. 能力声明（Capability Statement）
- `C1` 文件库场景承接能力：把统一平台中的文件库场景承接到范围树、固定列表和反馈区上。
- `C2` 范围树到当前范围与局部工具位的导出能力：让树上的当前节点和当前路径能够稳定说明“现在在文件库的哪个范围里”，并让范围树局部工具位附着当前范围。
- `C3` 当前范围到固定列表页签的连接能力：让当前范围确定之后，能稳定进入文件库固定列表页签。
- `C4` 固定列表页签到查询的连接能力：让固定列表页签里的查询条件和筛选条件形成清楚关系。
- `C5` 查询到文件清单工作面的连接能力：让查询之后能稳定进入文件清单工作面。
- `C6` 文件清单工作面到反馈的连接能力：让文件清单工作面里的结果、状态列、动作入口、摘要、空态或状态变化保持可解释。
- `C7` 查询后端契约消费能力：让当前范围和筛选条件能稳定消费查询请求与结果返回契约。
- `N1` 非职责声明：不负责处理页签、不负责提交审核流程、不负责历史差异查看，也不负责具体树控件实现。

## 2. 边界定义（Boundary / 参数）
- `SCENE_SCOPE` 文件库场景边界：文件库场景、范围树、固定列表和反馈区允许采用不同承载形式，但四者必须共同落在统一平台挂接出的同一场景承载面中。来源：`C1`。
- `CURRENT_SCOPE` 当前范围边界：范围树的节点结构、层级展开和当前节点允许变化，但任一时刻都必须能导出唯一当前范围、当前路径与局部工具位作用范围。来源：`C2 + C3`。
- `QUERY_SCOPE` 查询边界：固定列表页签、查询条件和筛选摘要允许变化，但查询必须落在当前范围下，且不能展开为脱离范围的自由查询结构。来源：`C4`。
- `FILE_WORKSPACE_SCOPE` 文件清单工作面边界：查询后的文件清单、文件状态列、动作入口和空态允许变化，但结果成立后必须共同落在同一文件清单工作面中。来源：`C5 + C6`。
- `RESULT_SCOPE` 结果反馈边界：摘要、统计、空态和状态提示允许采用不同反馈组合，但查询结果变化后必须至少有一处持续可观察反馈。来源：`C6`。
- `QUERY_BACKEND_CONTRACT_SCOPE` 查询后端契约消费边界：当前范围和查询条件允许映射为不同后端契约字段，但二者都必须稳定绑定到查询请求与结果返回。来源：`C7`。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 文件库场景与范围树基：`L0.M1[R1,R2]`。来源：`C1 + C2 + SCENE_SCOPE + CURRENT_SCOPE`。
- `B2` 固定列表页签与查询基：`L0.M2[R3]`。来源：`C3 + C4 + QUERY_SCOPE`。
- `B3` 文件清单工作面基：`L0.M10[R1,R2,R3,R4]`。来源：`C5 + C6 + FILE_WORKSPACE_SCOPE + RESULT_SCOPE`。
- `B4` 查询后端契约消费基：`review_workbench_backend.L0.M1[R1,R2,R3,R4]`。来源：`C7 + QUERY_BACKEND_CONTRACT_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 先有当前范围与局部工具位，再有固定列表和查询
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：文件库固定列表页签、范围树局部工具位和筛选条件必须建立在当前范围之上；不能把筛选区写成和当前范围无关的自由表单，也不能把范围树局部工具位写成与当前范围脱钩的独立按钮组。
  - `R1.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R1.4` 边界绑定：`SCENE_SCOPE/CURRENT_SCOPE/QUERY_SCOPE`。
- `R2` 查询之后必须进入文件清单工作面并看到反馈
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：每组筛选条件成立后，都应该能在文件库固定列表页签里进入文件清单工作面，并看到文件结果、状态列、动作入口、摘要、统计变化、空态说明或状态提示；不能只在内部状态里变化。
  - `R2.3` 输出能力：`C4 + C5 + C6`。
  - `R2.4` 边界绑定：`QUERY_SCOPE/FILE_WORKSPACE_SCOPE/RESULT_SCOPE`。
- `R3` 固定列表查询必须消费后端契约
  - `R3.1` 参与基：`B1 + B2 + B4`。
  - `R3.2` 组合方式：当前范围、固定列表页签和筛选条件，不仅界面上要讲得清楚，还要能稳定变成查询请求和列表结果返回；否则这套编排没法复用。
  - `R3.3` 输出能力：`C3 + C4 + C7`。
  - `R3.4` 边界绑定：`CURRENT_SCOPE/QUERY_SCOPE/QUERY_BACKEND_CONTRACT_SCOPE`。
- `R4` 文件库范围查询前半段闭环
  - `R4.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R4.2` 组合方式：文件库场景承接、范围树、当前范围、范围树局部工具位、固定列表页签、筛选、文件清单工作面、后端契约消费和汇总反馈要共同组成“进入文件库场景 -> 选树节点 -> 确认当前范围 -> 必要时使用范围树局部工具位 -> 进入固定列表页签 -> 设筛选 -> 发起查询 -> 进入文件清单工作面 -> 看到清单结果与反馈”的完整前半段。
  - `R4.3` 输出能力：`C1 + C2 + C3 + C4 + C5 + C6 + C7`。
  - `R4.4` 边界绑定：`SCENE_SCOPE/CURRENT_SCOPE/QUERY_SCOPE/FILE_WORKSPACE_SCOPE/QUERY_BACKEND_CONTRACT_SCOPE/RESULT_SCOPE`。
- `R5` 无效方案
  - `R5.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R5.2` 组合方式：以下方案无效：只有筛选条件，没有范围树、当前范围或固定列表页签；范围树旁有局部工具位，但无法说明当前作用范围；查询完成后没有文件清单工作面承接；只有统计数字，没有当前文件清单来源；树变了但当前范围讲不清；界面上有范围和筛选，但接不上稳定查询请求。
  - `R5.3` 失效结论：`N1`。
  - `R5.4` 边界绑定：`SCENE_SCOPE/CURRENT_SCOPE/QUERY_SCOPE/FILE_WORKSPACE_SCOPE/RESULT_SCOPE/QUERY_BACKEND_CONTRACT_SCOPE`。

## 5. 验证（Verification）
- `V1` 场景承接有效性：文件库场景、范围树、固定列表页签和反馈区都必须共同挂接在统一平台的同一场景承载面中。
- `V2` 范围导出有效性：任一筛选条件或范围树局部工具位都必须能追溯到当前范围。
- `V3` 文件清单工作面可解释性：筛选成立后，必须能进入可解释的文件清单工作面，并看到文件结果、状态列、动作入口或空态。
- `V4` 结果反馈可见性有效性：摘要、统计、空态或状态提示必须至少有一处发生可解释变化。
- `V5` 查询后端契约消费有效性：当前范围和查询条件必须稳定绑定到查询请求与结果返回。
- `V6` 最小必要性：移除任一 `B*` 后，`V1~V5` 至少一项失效。
- `V7` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
