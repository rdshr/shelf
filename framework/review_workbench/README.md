# review_workbench 领域总览

本文件是 `framework/review_workbench` 的领域级总览入口。它不替代各个 `L0/L1/L2/L3` 模块正文，而是回答三个跨模块问题：

1. 这个领域当前的模块树如何分层
2. 后续新增场景时应优先改哪一层
3. 什么情况才值得继续新增 `L1` 或 `L0`

`review_workbench` 当前不再被理解成“单一审查处理工作台”，而是被理解成：

`统一平台 + 平台内具体场景 + 场景模板 + 共享原子`

## 1. 当前分层

### `L0` 共享原子层

`L0` 负责定义跨多个场景稳定复用的前端共享结构原子。它回答的是：

- 平台壳层怎么成立
- 场景导航怎么成立
- 工作集和工作面怎么切换
- 对象列表、对象动作和查看入口怎么成立
- 全局反馈怎么持续可见
- 范围改动、审核结论输入、历史序列回看这些跨实例稳定前端结构怎么成立

这里的 `L0` 默认为前端邻域真相源：
- 它负责前端承接、前端表达、前端工作面与前端可观察反馈。
- 它不直接负责请求字段、返回字段、动作编码或历史差异编码。
- 这些服务侧语义统一外移到 `review_workbench_backend`。

当前主要模块包括：

- `L0-M0` 统一审查平台壳层与场景导航原子模块
- `L0-M1` 范围树与局部概览原子模块
- `L0-M2` 查询条件与筛选摘要原子模块
- `L0-M3` 工作集与工作面切换原子模块
- `L0-M4` 对象列表与查看入口共享底座原子模块
  说明：这是共享底座原子，本身不直接代表文件库、审核或追溯某一具体场景；当前主要被 `L0-M8`、`L0-M9`、`L0-M10` 继续复用。
- `L0-M5` 全局反馈与状态汇总原子模块
- `L0-M7` 文件与目录变更原子模块
- `L0-M8` 审核结论与结果回流原子模块
- `L0-M9` 历史序列与差异回看原子模块
- `L0-M10` 文件清单工作面原子模块
- `L0-M11` 分块预览与处理原子模块
- `L0-M12` 来源定位与原位回看原子模块
- `L0-M13` 审核工作栏与队列概览原子模块

其中有三类前端原子需要特别注意边界：
- `L0-M7` 负责文件与目录改动在前端如何承接、如何解释位置与影响，不负责结构改动请求 / 返回契约。
  说明：它不是悬空专有原子，而是建立在 `L0-M1` 的范围上下文、`L0-M4` 的对象底座和 `L0-M5` 的回流反馈底座之上，再把“文件与目录改动”沉成领域专有支路。
- `L0-M8` 负责审核结论输入与前端可观察回流，不负责审核动作请求 / 结果编码。
- `L0-M9` 负责历史序列、差异回看与元信息在前端如何承接，不负责历史请求 / 返回契约。
  说明：它不是悬空历史查看原子，而是建立在 `L0-M4` 的对象 / 差异入口底座与 `L0-M5` 的元信息 / 回看反馈底座之上，再把历史序列与差异回看沉成领域专有支路。
- `L0-M12` 负责来源定位与原位回看在前端如何承接，不负责查看请求 / 返回契约。
  说明：它不是悬空来源查看原子，而是建立在 `L0-M4` 的对象 / 查看锚点底座与 `L0-M5` 的反馈底座之上，再把来源定位与原位回看沉成领域专有支路。

其中和审核型场景直接相关的两个前端原子需要分清：
- `L0-M8` 负责“审核结论输入与结果回流”这条最小审核语义。
- `L0-M13` 负责“审核工作栏、当前焦点摘要、待审 / 已审概览与局部输入承接”这类稳定侧栏工作台结构。
- 两者应组合使用，不应互相吞并。

### `L1` 场景模板层

`L1` 负责定义一类场景如何由 `L0` 原子组合成稳定主线。它回答的是：

- 平台如何从导航进入具体场景
- 某一类场景的前半段和后半段如何闭环
- 某类支撑支路如何回接主线

当前主要模块包括：

- `L1-M0` 平台导航到场景挂接编排模块
- `L1-M1` 文件库工作集与处理页签编排模块
- `L1-M2` 文件库范围结构变更与查询衔接编排模块
- `L1-M3` 文件库范围查询编排模块
- `L1-M4` 审核型场景编排模块
- `L1-M5` 追溯型场景编排模块

当前 `L1` 层默认按“前端结构 + 外部后端契约消费”共同编排：
- 文件库范围查询主链主要组合 `L0-M1`、`L0-M2`、`L0-M10`，并复用 `review_workbench_backend.L0-M1`。
- 文件库处理后半段主要组合 `L0-M3`、`L0-M11`、`L0-M12`，并复用 `review_workbench_backend.L0-M2` 与 `L0-M4`。
  说明：这里的 `L0-M12` 默认继续复用 `L0-M4/L0-M5`，不要把来源回看支路误写成脱离共享对象底座与反馈底座的并行结构。
- 文件库范围结构改动支路主要组合 `L0-M1`、`L0-M7`，并复用 `review_workbench_backend.L0-M3`。
  说明：这里的 `L0-M7` 默认继续复用 `L0-M1/L0-M4/L0-M5`，不要把范围改动支路误写成脱离共享对象底座与反馈底座的并行结构。
- 审核型主链主要组合 `L0-M3`、`L0-M4`、`L0-M5`、`L0-M8`、`L0-M13`，并复用 `review_workbench_backend.L0-M1`、`L0-M2`、`L0-M4`。
- 追溯型主链主要组合 `L0-M4`、`L0-M5`、`L0-M9`，并复用 `review_workbench_backend.L0-M1`、`L0-M2`、`L0-M5`。
  说明：这里的 `L0-M9` 默认继续复用 `L0-M4/L0-M5`，不要把历史回看支路误写成脱离共享对象底座与反馈底座的并行结构。

### `L2` 平台内具体场景层

`L2` 负责定义统一平台中的具体场景实例。它回答的是：

- 当前平台里具体有哪些场景
- 每个场景如何承接统一平台与对应模板
- 不同具体场景分别导出什么能力

当前主要模块包括：

- `L2-M0` 文件库处理场景模块
- `L2-M1` 提交审核场景模块
- `L2-M2` 历史追溯场景模块

### `L3` 统一平台层

`L3` 负责定义统一平台本身如何成立，并如何挂接 `L2` 具体场景。它回答的是：

- 平台如何进入
- 当前场景如何被挂接到统一主工作区
- 场景切换、平台工具位和平台反馈如何保持成立

当前主要模块包括：

- `L3-M0` 统一审查平台场景模块

## 2. 默认扩展规则

后续新增场景时，默认按下面顺序判断：

### 规则 A：优先新增 `L2`

如果新需求只是平台里多了一个具体场景，但它本质上仍属于现有模板之一，那么默认新增 `L2` 模块，不先改 `L1` 或 `L0`。

典型例子：

- 新增一个“抽检审核场景”，但它仍然属于审核型主线
- 新增一个“版本回放场景”，但它仍然属于追溯型主线

这类变化优先落在 `L2`。

### 规则 B：只有出现新的稳定场景类型时，才新增 `L1`

如果新场景无法自然归入文件库处理型、审核型、追溯型等已有模板，而且这种差异是跨实例稳定复用的，那么才新增新的 `L1` 模板模块。

判断标准不是“当前项目觉得不一样”，而是：

- 是否形成新的稳定主线
- 是否会被多个未来场景复用
- 是否不是现有 `L1` 通过组合就能覆盖

### 规则 C：只有出现新的共享结构原子时，才新增 `L0`

如果多个 `L1` 模板都开始重复依赖某个新的稳定结构，而这个结构又不能被现有 `L0` 原子表达，那么才允许新增 `L0`。

判断标准是：

- 是否跨多个场景模板复用
- 是否是结构原子，而不是功能清单
- 是否不是某个 `L1/L2` 模块内部的实例化差异

## 3. 修改判定规则

为了避免未来继续把平台结构改乱，后续修改时优先按下面方式判断：

### 情况 1：只是当前场景字段变了

例如新增字段、筛选项、列表列、统计项、审核意见字段、历史展示字段。

处理方式：

- 优先改 `project.toml` 或后续实例化配置
- 不优先改 framework

### 情况 2：只是多了一个同类型场景

例如新增一个新的审核场景、历史场景、对象处理场景。

处理方式：

- 优先新增 `L2`

### 情况 3：出现了新的场景主线

例如新场景既不是文件库处理型，也不是审核型，也不是追溯型，而且会稳定复用。

处理方式：

- 新增 `L1`

### 情况 4：多个模板都在重复借同一种新结构

例如多个 `L1` 都开始重复定义新的共享对象关系、共享承接面、共享契约结构。

处理方式：

- 新增或重构 `L0`

## 4. 当前作者入口建议

如果后续继续改 `review_workbench`，默认阅读顺序建议是：

1. 先读本文件，判断当前需求应落在哪一层
2. 再读 `L3-M0`，确认平台层入口语义
3. 如果是具体场景，先找对应 `L2`
4. 如果 `L2` 不够，再看是否应落到某个现有 `L1`
5. 只有在多个 `L1` 都出现共享缺口时，才回到 `L0`

## 5. 当前总原则

`review_workbench` 后续扩展时，默认新增具体场景，不默认新增底层结构。

换句话说：

- 优先扩 `L2`
- 谨慎扩 `L1`
- 更谨慎扩 `L0`

只有这样，统一平台壳、具体场景、场景模板和共享原子之间的单向收敛关系才会稳定。

## 6. 边界与验证写法约定

为了让 `review_workbench` 的 framework 文档更适合作为后续 Config / Code / Evidence 的稳定上游，后续新增或修改模块时，默认采用下面这组写法约定：

- 边界项优先写“允许变化的维度 + 必须成立的最小结构条件”，不要只写“需要稳定”。
- 边界项优先回答“什么可以变、变到哪里仍然合法、最少要保留什么结构”，不要退回对象名词清单。
- 涉及当前焦点的边界，优先写成“唯一当前场景 / 唯一当前范围 / 唯一当前工作面”这类可判定条件。
- 涉及来源、归属、定位的边界，优先写成“必须说明来源 / 归属 / 锚点 / 落点”，而不是泛写“可追溯”。
- 涉及反馈的边界，优先写成“状态变化后至少有一处持续可观察反馈”，不要只写“有反馈”。
- 验证项应直接复用边界语义，优先写成“必须能识别什么、必须说明什么、至少出现什么变化”，避免和边界口径脱节。
- 验证项默认要能区分“结构没成立”与“实例值不同”这两类问题，因此尽量避免把具体现实例值写进 `V*`。
- 若某个模块新增了新的边界写法或验证写法，优先先看这组约定是否也要同步更新到 `README`。
### 外部后端契约依赖
`review_workbench` 当前把后端契约真相外移到独立领域 `review_workbench_backend`。也就是说：
- `review_workbench` 继续负责前端邻域中的原子、编排与场景。
- `review_workbench_backend` 负责统一审查平台服务侧的请求 / 返回 / 回流后端契约。
- `review_workbench` 中若提到“消费后端契约”，默认是复用 `review_workbench_backend.L0.M0~M5`，而不是在本目录重复维护一份后端契约真相。
- 对于审核与追溯这类场景，`review_workbench` 里的 `L0-M8`、`L0-M9` 只负责前端承接与前端可观察结构；对应的动作请求、历史请求、差异返回等契约语义，由 `review_workbench_backend` 中对应模块承担。
- 对于文件库这类场景，`L0-M7`、`L0-M12` 只负责前端结构变更承接与来源回看承接；对应的结构改动请求 / 失效回流、查看请求 / 返回契约分别由 `review_workbench_backend.L0-M3`、`L0-M2` 承担。
