# 文件库工作集、处理页签与来源回看编排模块:ReviewWorkbenchFileLibraryWorksetResultActionAndSourceOrchestration

@framework

本模块定义文件库处理场景后半段是怎么接起来的，也就是“从文件清单工作面里打开文件项 -> 进入处理页签中的分块预览与处理 -> 做纠错、分类、确认等块级动作 -> 必要时进入来源定位与原位回看 -> 看到反馈回来”。它位于 `review_workbench` 的编排层，关心的是文件库处理页签怎样从文件清单工作面打开、分块预览与块处理怎样承接、来源怎样原位回看、反馈怎样回到当前工作上下文，而不是具体有哪些列、有哪些状态或用哪个接口。

本模块建立在 `review_workbench.L0.M3~M5`、`review_workbench.L0.M11`、`review_workbench.L0.M12` 与外部领域 `review_workbench_backend.L0.M2`、`review_workbench_backend.L0.M4` 这些领域原子之上，并默认承接 `review_workbench.L1.M3` 已经整理好的文件库文件清单工作面上下文和打开来源；而这些领域原子继续显式承接 `frontend.L1.M4`、`frontend.L1.M9`、`frontend.L1.M12` 和 `frontend.L1.M15`。也就是说，本模块不直接重写前端页签、分块预览工作面、来源查看面和反馈壳，而是把下层已经稳定的前端领域原子与外部后端契约编排成“打开分块预览处理页签 -> 查看来源 / 执行动作 -> 收到反馈”的后半段主链。

对于“处理页签怎样打开、来源定位怎样请求、块级动作怎样提交、结果怎样回流”这件事，它向下复用外部领域 `review_workbench_backend.L0.M2` 的工作面打开与查看契约，以及 `review_workbench_backend.L0.M4` 的对象动作与结果回流契约，不在这里重定义请求和返回结构；而“来源查看面如何承接、怎样保持原位回看上下文可解释”则继续由 `L0-M12` 负责。

这里的 `L0-M12` 不是悬空的来源查看子系统，而是继续建立在 `L0-M4` 的对象来源 / 查看锚点底座与 `L0-M5` 的来源反馈底座之上，再被编排进文件库处理页签上下文中的来源回看支路。

## 1. 能力声明（Capability Statement）
- `C1` 文件库处理页签切换能力：支持文件库固定列表页签与多个处理页签在同一工作台里切换，并通过已有工作页签返回之前打开过的处理内容。
- `C2` 分块预览与处理承接能力：支持当前处理页签稳定承接分块预览、块序列、当前块和块级处理动作。
- `C3` 块级处理动作回流能力：支持纠错、分类、确认、标记、跳转和批量处理，并把结果返回到当前处理上下文。
- `C4` 来源定位与原位回看能力：支持从当前对象或当前块进入来源文件、定位到原位并保持来源上下文可解释。
- `C5` 持续反馈能力：支持处理后同步更新块状态、局部统计和全局汇总。
- `C6` 后端契约消费能力：支持处理页签打开、来源定位、块级动作和结果回流稳定消费后端契约。
- `N1` 非职责声明：不负责定义具体业务动作名，不负责提交审核流程，不负责历史差异回放，也不负责具体预览控件实现。

## 2. 边界定义（Boundary / 参数）
- `WORKSET_SCOPE` 工作集边界：固定列表页签、处理页签和返回既有页签的方式允许变化，但任一时刻必须存在可解释当前工作页签，且切换与关闭后的回退结果必须可解释。来源：`C1`。
- `DETAIL_SCOPE` 处理内容边界：当前文件、当前块、块处理信息、空态和加载态允许变化，但处理内容必须落在当前处理页签与当前对象上下文中，并能识别当前文件与当前块。来源：`C2`。
- `ACTION_SCOPE` 动作边界：块级动作、批量动作和查看动作允许采用不同动作集合，但每个动作都必须附着当前处理页签中的明确对象、当前块或对象集合。来源：`C3`。
- `SOURCE_SCOPE` 来源边界：来源回看允许通过当前块、片段或源文件对象进入，但任一次来源回看都必须能追溯到当前处理页签中的明确来源。来源：`C4`。
- `FEEDBACK_SCOPE` 反馈边界：动作执行后的状态变化允许落到处理页签、固定列表页签、局部统计或全局汇总，但动作完成后必须至少有一处持续可观察反馈。来源：`C5`。
- `BACKEND_CONTRACT_SCOPE` 后端契约消费边界：处理页签打开、来源定位、动作提交和汇总回流允许映射为不同后端契约字段，但每类动作都必须稳定绑定到对应请求与返回。来源：`C6`。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 工作页签集合基：`L0.M3[R1,R3]`。来源：`C1 + WORKSET_SCOPE`。
- `B2` 分块预览与处理基：`L0.M11[R1,R2]`。来源：`C2 + DETAIL_SCOPE`。
- `B3` 块级动作基：`L0.M11[R2,R3]`。来源：`C3 + ACTION_SCOPE`。
- `B4` 来源定位与原位回看基：`L0.M12[R1,R2,R3]`。来源：`C4 + SOURCE_SCOPE`。
- `B5` 反馈基：`L0.M5[R2,R3] + L0.M11[R3] + L0.M12[R3]`。来源：`C5 + FEEDBACK_SCOPE`。
- `B6` 后端契约消费基：`review_workbench_backend.L0.M2[R1,R2] + review_workbench_backend.L0.M4[R1,R3]`。来源：`C6 + BACKEND_CONTRACT_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 文件库处理页签必须承接打开来源
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：文件库处理页签必须属于当前工作页签集合，并且要能说明自己是从文件清单工作面中的哪个文件项打开的；已存在于工作集中的处理页签可以作为返回之前打开过的处理内容的入口，但该页签首次打开时仍必须来自文件清单工作面中的明确文件项来源；不能把处理页签写成无来源的孤立页面。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`WORKSET_SCOPE/DETAIL_SCOPE/SOURCE_SCOPE`。
- `R2` 块级动作必须挂在分块预览中的当前块结构上
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：任何块级处理动作都必须作用在当前处理页签中的当前块、某一组块或当前对象之上；不能出现脱离分块预览与处理上下文的黑盒操作。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`DETAIL_SCOPE/ACTION_SCOPE`。
- `R3` 来源回看必须挂在当前块或当前对象上
  - `R3.1` 参与基：`B2 + B4`。
  - `R3.2` 组合方式：任何来源定位与原位回看都必须从当前处理页签中的当前对象或当前块进入，打开后的来源面和原位定位也要能说明当前来源对象、当前块与当前处理页签；不能出现脱离处理页签内容的孤立来源预览。
  - `R3.3` 输出能力：`C2 + C4`。
  - `R3.4` 边界绑定：`DETAIL_SCOPE/SOURCE_SCOPE`。
- `R4` 动作之后必须有反馈
  - `R4.1` 参与基：`B3 + B5`。
  - `R4.2` 组合方式：动作执行后，处理页签、固定列表页签、局部统计、全局汇总或状态提示里至少要有一处能看出变化，并且能追到当前页签、当前对象与当前块。
  - `R4.3` 输出能力：`C3 + C5`。
  - `R4.4` 边界绑定：`ACTION_SCOPE/FEEDBACK_SCOPE`。
- `R5` 文件库处理页签、来源回看和动作必须消费后端契约
  - `R5.1` 参与基：`B2 + B3 + B4 + B6`。
  - `R5.2` 组合方式：分块预览与处理不能只停留在界面层，还要对应稳定的打开请求与结果返回；来源定位也要对应稳定的来源请求和来源返回；块级动作也要对应稳定的动作请求和动作结果，否则这套后半段没法复用。
  - `R5.3` 输出能力：`C2 + C3 + C4 + C6`。
  - `R5.4` 边界绑定：`DETAIL_SCOPE/ACTION_SCOPE/SOURCE_SCOPE/BACKEND_CONTRACT_SCOPE`。
- `R6` 文件库处理场景后半段闭环
  - `R6.1` 参与基：`B1 + B2 + B3 + B4 + B5 + B6`。
  - `R6.2` 组合方式：文件清单工作面中的打开动作、处理页签切换、分块预览与处理、来源定位与原位回看、块级动作、后端契约消费和反馈回流要共同组成“在文件清单工作面中打开文件项 -> 进入处理页签中的分块预览与处理 -> 查看来源 / 做块级动作 -> 收到结果 -> 看到反馈”的完整后半段。
  - `R6.3` 输出能力：`C1 + C2 + C3 + C4 + C5 + C6`。
  - `R6.4` 边界绑定：`WORKSET_SCOPE/DETAIL_SCOPE/ACTION_SCOPE/SOURCE_SCOPE/FEEDBACK_SCOPE/BACKEND_CONTRACT_SCOPE`。
- `R7` 无效方案
  - `R7.1` 参与基：`B1 + B2 + B3 + B4 + B5 + B6`。
  - `R7.2` 组合方式：以下方案无效：只有标签切换，没有处理页签；只有分块预览没有当前块；只有块级动作没有块归属；来源回看可以打开，但无法说明来自哪个当前块或不能定位到源文件原位；动作执行后没有任何反馈；界面上有预览、来源和动作，但接不上稳定服务返回。
  - `R7.3` 失效结论：`N1`。
  - `R7.4` 边界绑定：`WORKSET_SCOPE/DETAIL_SCOPE/ACTION_SCOPE/SOURCE_SCOPE/FEEDBACK_SCOPE/BACKEND_CONTRACT_SCOPE`。

## 5. 验证（Verification）
- `V1` 工作面承接有效性：当前处理页签必须能追溯到文件清单工作面中的打开来源，并解释当前对象上下文。
- `V2` 块级动作归属有效性：处理页签里的任一动作都必须有明确当前块或对象归属。
- `V3` 来源定位有效性：任一来源回看都必须能追溯到当前块或当前对象，并在来源面中定位到原位。
- `V4` 反馈回流有效性：任一块级动作执行后，必须至少有一处持续可观察反馈出现可解释变化。
- `V5` 切换返回有效性：切换或关闭页签后，当前工作面必须可解释。
- `V6` 后端契约消费有效性：处理页签、来源定位和块级动作回流必须稳定绑定到打开返回、来源返回、动作请求、动作结果和汇总返回。
- `V7` 最小必要性：移除任一 `B*` 后，`V1~V6` 至少一项失效。
- `V8` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
