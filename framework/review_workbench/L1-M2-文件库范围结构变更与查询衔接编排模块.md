# 文件库范围结构变更与查询衔接编排模块:ReviewWorkbenchFileLibraryScopeStructureMutationAndQueryOrchestration

@framework

本模块定义文件库处理场景前半段里“怎样在当前范围下做范围改动，以及这些改动怎样回到查询和局部反馈上”这条链路。它位于 `review_workbench` 的编排层，关心的是上传、新建目录、删除、移动和重命名怎样建立在文件库当前范围树节点上，并在改动完成后把树、当前范围、局部信息和查询重新接稳，而不是关心具体目录字段、接口路径或按钮实现。

本模块建立在 `review_workbench.L0.M1`、`review_workbench.L0.M2`、`review_workbench.L0.M7` 与外部领域 `review_workbench_backend.L0.M3` 这些领域原子之上，并默认承接 `review_workbench.L1.M3` 已经整理好的当前场景、范围树、当前范围和查询上下文；而其中的范围树、查询条件这些领域原子继续显式承接 `frontend.L1.M0`、`frontend.L1.M1`、`frontend.L1.M2`、`frontend.L1.M3` 和 `frontend.L1.M4`。

这里的 `L0.M7` 不是悬空的专有改动原子，而是继续建立在 `L0.M1` 的当前范围上下文、`L0.M4` 的对象底座和 `L0.M5` 的反馈底座之上，再把文件与目录改动收束成领域专有结构。也就是说，本模块不直接重写上传控件、树控件或查询摘要基础，而是把下层已经稳定的领域原子编排成“当前范围下改动 -> 树与范围回流 -> 查询重新稳定”的文件库前半段支路。

对于“范围改动怎样发给后端契约、树、当前范围和查询怎样回流”这件事，它向下复用外部领域 `review_workbench_backend.L0.M3` 的范围结构变更与查询失效契约，以及 `review_workbench.L0.M7` 的文件与目录变更原子结构，不在这里重新定义请求和返回结构。

## 1. 能力声明（Capability Statement）
- `C1` 当前范围下发起范围改动的能力：支持在文件库当前范围树节点和当前范围下发起上传、新建目录、删除、移动与重命名。
- `C2` 范围改动回流能力：支持范围改动后同步更新范围树、当前范围状态和局部信息。
- `C3` 查询重新稳定能力：支持范围改动后让当前筛选、结果命中和查询摘要重新稳定。
- `C4` 后端契约消费能力：支持范围改动稳定消费后端契约中的改动请求、改动返回与查询失效回流。
- `N1` 非职责声明：不负责结果区业务处理、不负责多场景切换、不负责具体树控件和上传控件实现。

## 2. 边界定义（Boundary / 参数）
- `MUTATION_SCOPE` 改动边界：上传、新建目录、删除、移动与重命名允许在不同对象和节点上发生，但每次改动都必须附着文件库当前范围树节点和当前范围。来源：`C1`。
- `RETURN_SCOPE` 回流边界：改动后当前路径、当前范围状态和焦点回退策略允许变化，但改动完成后必须能解释当前工作范围与当前焦点落点。来源：`C2`。
- `TREE_SCOPE` 树刷新边界：改动后范围树、局部信息和受影响节点允许采用不同刷新策略，但改动完成后必须至少有一处出现可解释更新。来源：`C2`。
- `QUERY_SCOPE` 查询边界：改动后筛选摘要、结果命中和查询上下文允许刷新、失效或重算，但必须能解释当前查询为何变化以及当前查询状态是否仍成立。来源：`C3`。
- `BACKEND_CONTRACT_SCOPE` 后端契约消费边界：范围改动请求、改动返回和查询失效信息允许映射为不同后端契约字段，但三者都必须稳定绑定到后端契约请求与返回。来源：`C4`。
- `TRACE_SCOPE` 追踪边界：改动对象、当前范围和改动后落点允许变化，但任一次改动都必须能追溯“改的是谁、发生在哪、改后落到哪”。来源：`C1 + C2`。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 当前范围树改动基：`L0.M1[R1] + L0.M7[R1]`。来源：`C1 + MUTATION_SCOPE + TRACE_SCOPE`。
- `B2` 树与当前范围回流基：`L0.M1[R3] + L0.M7[R5]`。来源：`C2 + RETURN_SCOPE + TREE_SCOPE`。
- `B3` 查询重新稳定基：`L0.M2[R2,R3] + review_workbench_backend.L0.M3[R3]`。来源：`C3 + QUERY_SCOPE`。
- `B4` 改动后端契约消费基：`review_workbench_backend.L0.M3[R1,R2,R3] + L0.M7[R5]`。来源：`C4 + BACKEND_CONTRACT_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 范围改动必须附着当前范围
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：任一上传、新建目录、删除、移动或重命名都必须能说明“它发生在文件库当前范围树哪个节点以及哪个当前范围下”；不能把范围改动写成脱离树节点和当前范围的孤立动作。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`MUTATION_SCOPE/TRACE_SCOPE`。
- `R2` 树更新必须和当前范围一起回流
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：范围改动后，范围树更新不能和当前范围状态回退、局部信息变化分离；否则用户无法解释现在树上处于哪个节点、当前范围还是不是原来的范围。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`MUTATION_SCOPE/RETURN_SCOPE/TREE_SCOPE/TRACE_SCOPE`。
- `R3` 范围改动之后必须重新稳定查询
  - `R3.1` 参与基：`B2 + B3`。
  - `R3.2` 组合方式：上传、新建目录、删除、移动或重命名之后，当前筛选摘要、结果命中和查询上下文至少要有一处发生可解释更新；不能只改树，不让查询重新稳定。
  - `R3.3` 输出能力：`C2 + C3`。
  - `R3.4` 边界绑定：`TREE_SCOPE/QUERY_SCOPE`。
- `R4` 范围改动编排必须消费后端契约
  - `R4.1` 参与基：`B1 + B3 + B4`。
  - `R4.2` 组合方式：文件库当前范围树和当前范围下的范围改动，不仅界面上要讲得清楚，还要能稳定变成服务请求、树返回、范围状态回流与查询失效信息；否则这套编排没法复用。
  - `R4.3` 输出能力：`C1 + C3 + C4`。
  - `R4.4` 边界绑定：`MUTATION_SCOPE/RETURN_SCOPE/QUERY_SCOPE/BACKEND_CONTRACT_SCOPE`。
- `R5` 文件库范围改动前半段支路闭环
  - `R5.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R5.2` 组合方式：当前范围树下的范围改动、树更新、当前范围状态回流、查询重新稳定和后端契约消费要共同组成“进入文件库范围树 -> 确认当前范围 -> 发起范围改动 -> 看到树与范围状态更新 -> 查询重新稳定”的完整前半段支路。
  - `R5.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R5.4` 边界绑定：`MUTATION_SCOPE/RETURN_SCOPE/TREE_SCOPE/QUERY_SCOPE/BACKEND_CONTRACT_SCOPE/TRACE_SCOPE`。
- `R6` 范围改动完成后必须回到当前范围下的查询上下文
  - `R6.1` 参与基：`B2 + B3`。
  - `R6.2` 组合方式：范围改动完成后，当前范围状态、筛选摘要、结果命中或查询上下文必须重新稳定到“可继续当前范围处理”的状态；不能只更新树和范围状态，却让文件库列表上下文断开或变成不可解释的旧查询状态。
  - `R6.3` 输出能力：`C2 + C3`。
  - `R6.4` 边界绑定：`RETURN_SCOPE/TREE_SCOPE/QUERY_SCOPE`。
- `R7` 无效方案
  - `R7.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R7.2` 组合方式：以下方案无效：范围改动能做，但不属于当前范围树节点；树节点变了但当前范围讲不清；查询结果和筛选摘要保持旧状态；界面上有删除、移动和重命名，但接不上稳定服务请求、范围状态回流与查询失效回流；树和范围状态更新了，但无法回到当前范围下继续查询或处理。
  - `R7.3` 失效结论：`N1`。
  - `R7.4` 边界绑定：`MUTATION_SCOPE/RETURN_SCOPE/TREE_SCOPE/QUERY_SCOPE/BACKEND_CONTRACT_SCOPE/TRACE_SCOPE`。

## 5. 验证（Verification）
- `V1` 改动附着有效性：任一范围改动都必须能追溯到文件库当前范围树节点和当前范围。
- `V2` 范围回流有效性：范围改动后，当前范围状态、焦点回退和树刷新都必须能被解释。
- `V3` 查询重稳有效性：范围改动后，筛选摘要、结果命中或查询上下文必须至少有一处重新稳定。
- `V4` 后端契约消费有效性：范围改动编排必须稳定绑定到改动请求、改动返回和查询失效回流。
- `V5` 查询回接有效性：范围改动完成后，必须能回到当前范围下继续查询或处理，不能停留在不可解释的断开状态。
- `V6` 最小必要性：移除任一 `B*` 后，`V1~V5` 至少一项失效。
- `V7` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
