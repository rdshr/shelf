# 文件清单工作面原子模块:ReviewWorkbenchFileListWorkspaceAtoms

@framework

本模块定义 `review_workbench` 中文件库场景最小的文件清单工作面结构。推导链为：`framework/review_workbench/L0-M4-对象列表与查看入口共享底座原子模块.md` + `framework/review_workbench/L0-M5-全局反馈与状态汇总原子模块.md` -> 本模块。也就是说，文件库场景中间那块“文件名、归属、状态、动作、空态反馈”工作面，不单独发明另一套表格真相，而是在通用对象列表、动作入口与反馈结构之上，收束成“文件资源清单工作面”这组领域原子。

本模块位于 `review_workbench` 的原子层。它只负责把文件库里的中间清单工作面表达清楚，让文件资源如何被列出、如何展示主标识与归属、如何展示处理状态、如何进入动作与查看入口，以及如何表达空态或结果反馈，形成稳定结构；它不在这里定义具体字段名，不定义项目实例里的列配置，也不替代处理页签本身。

## 1. 能力声明（Capability Statement）
- `C1` 文件清单集合表达能力：将文件资源集合、结果表格与空态组织成稳定结构。
- `C2` 文件主标识与归属列表达能力：将文件名、文件类型、来源范围、所属目录或其他归属信息组织成稳定列结构。
- `C3` 文件处理状态列表达能力：将分类状态、审核状态、未解决项、处理进度或其他文件状态信息组织成稳定列结构。
- `C4` 文件动作列与查看入口表达能力：将行级动作、批量动作与查看入口组织成稳定动作结构。
- `C5` 文件清单反馈态表达能力：将清单空态、筛选空态、结果摘要和状态提示组织成稳定反馈结构。
- `N1` 非职责声明：不定义具体业务列名，不定义后端 schema，不直接替代处理页签或工作集切换。

## 2. 边界定义（Boundary / 参数）
- `FILESET` 文件清单集合边界：文件数量、排序和结果态形式允许变化，但结果成立时必须存在可识别文件集合，空态时必须能解释当前无结果状态。
- `FILE_IDENTITY` 文件主标识与归属边界：归属范围、目录归属和主展示字段允许变化，但每个文件都必须具备稳定主标识，并能解释当前来源范围。
- `FILE_STATUS` 文件状态边界：分类、审核、未解决项和其他状态字段允许变化，但每个状态都必须附着明确文件行。
- `FILE_ACTION` 文件动作边界：行级动作、批量动作和查看入口允许变化，但每个动作都必须附着文件行或文件集合。
- `FILE_FEEDBACK` 文件清单反馈边界：空态、摘要和状态提示组合允许变化，但结果变化后必须至少有一处反馈能够解释当前文件清单结果态。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 文件清单集合基：`L0.M4[R1]`。来源：`C1 + FILESET`。
- `B2` 文件主标识与归属列基：`L0.M4[R1,R2]`。来源：`C2 + FILE_IDENTITY`。
- `B3` 文件状态列基：`L0.M4[R2] + L0.M5[R1]`。来源：`C3 + FILE_STATUS`。
- `B4` 文件动作与查看入口基：`L0.M4[R2,R3]`。来源：`C4 + FILE_ACTION`。
- `B5` 文件清单反馈态基：`L0.M4[R1] + L0.M5[R1,R3]`。来源：`C5 + FILE_FEEDBACK`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 文件清单必须先有文件集合再有工作面列结构
  - `R1.1` 参与基：`B1 + B2 + B3`。
  - `R1.2` 组合方式：文件清单工作面必须先说明当前有哪些文件资源被列出，再说明这些文件如何以主标识、归属列和状态列被承接；不能只有一个抽象表格容器而没有文件集合与列语义。
  - `R1.3` 输出能力：`C1 + C2 + C3`。
  - `R1.4` 边界绑定：`FILESET/FILE_IDENTITY/FILE_STATUS`。
- `R2` 文件主标识、归属列和状态列必须附着同一文件行
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：文件名、归属信息和处理状态必须共同附着在同一文件行结构上；不能把文件状态写成脱离文件行的独立摘要，也不能把归属列写成无法追溯文件来源的游离信息。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`FILE_IDENTITY/FILE_STATUS`。
- `R3` 文件动作与查看入口必须附着文件行或文件集合
  - `R3.1` 参与基：`B1 + B4`。
  - `R3.2` 组合方式：文件动作列、批量动作和查看入口必须附着在明确的文件行或文件集合之上；不能定义成脱离当前清单结果的孤立按钮。
  - `R3.3` 输出能力：`C1 + C4`。
  - `R3.4` 边界绑定：`FILESET/FILE_ACTION`。
- `R4` 文件清单反馈必须附着当前清单结果态
  - `R4.1` 参与基：`B1 + B5`。
  - `R4.2` 组合方式：清单空态、筛选空态、摘要和状态提示必须附着在当前文件清单结果态上；不能出现有反馈但无法说明当前是在什么文件清单状态下看到它。
  - `R4.3` 输出能力：`C1 + C5`。
  - `R4.4` 边界绑定：`FILESET/FILE_FEEDBACK`。
- `R5` 文件清单工作面闭环
  - `R5.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R5.2` 组合方式：文件集合、主标识与归属列、状态列、动作入口和清单反馈必须共同组成“列出文件 -> 识别文件归属与状态 -> 对文件执行动作 / 进入查看 -> 看到当前清单反馈”的稳定工作面闭环。
  - `R5.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R5.4` 边界绑定：`FILESET/FILE_IDENTITY/FILE_STATUS/FILE_ACTION/FILE_FEEDBACK`。
- `R6` 无效方案
  - `R6.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R6.2` 组合方式：以下方案无效：只有抽象结果列表没有文件资源语义；只有文件名没有归属或状态列；动作按钮存在但无法说明作用于哪一个文件行；存在空态或状态提示但无法说明对应哪一个当前清单结果。
  - `R6.3` 失效结论：`N1`。
  - `R6.4` 边界绑定：`FILESET/FILE_IDENTITY/FILE_STATUS/FILE_ACTION/FILE_FEEDBACK`。

## 5. 验证（Verification）
- `V1` 文件清单可识别性：结果成立时必须存在可识别文件集合、文件行与主要列结构。
- `V2` 文件行附着有效性：任一归属列或状态列都必须能追溯到明确文件行。
- `V3` 文件动作归属有效性：任一动作或查看入口都必须能说明附着的文件行或文件集合。
- `V4` 文件清单反馈可解释性：任一空态、摘要或状态提示都必须能说明当前对应的文件清单结果态。
- `V5` 最小必要性：移除任一 `B*` 后，`V1~V4` 至少一项失效。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
