# 统一审查平台场景模块:ReviewWorkbenchUnifiedPlatformScenarios

@framework

本模块定义 `review_workbench` 在统一平台层如何把下层平台编排结构组织成一个可挂接多个工作场景的统一审查平台。它关注的不是某个具体场景里的范围树、审核表单或历史差异块，而是用户如何进入平台、切换当前场景、让场景挂接到统一主工作区、持续使用顶部全局工具位，并持续看到平台级状态与反馈。

本模块位于 `review_workbench` 的统一平台层。向下它依赖 `L1-M0` 这条平台挂接主链，并向下承接 `L2-M0`、`L2-M1`、`L2-M2` 这类具体场景模块作为统一平台中的可挂接场景实例。向上它为项目配置、代码实现和验证视图提供“统一平台壳 + 可插拔场景 + 平台级工具位”这类更贴近真实产品组织方式的场景入口。

`review_workbench` 的具体业务场景不再在本模块里混成一条主线，而是分别落到独立场景模块中：文件库处理场景使用 `L2-M0`，提交审核场景使用 `L2-M1`，历史追溯场景使用 `L2-M2`。本模块只强调统一平台如何稳定承接这些场景，而不反向定义场景内部编排真相。

关于 `review_workbench` 的模块树分层、默认扩展顺序以及“新增场景优先落 `L2`、仅在出现新的稳定模板时才上提 `L1`、仅在出现新的共享结构原子时才上提 `L0`”这类跨模块规则，统一说明见同目录下的 `README.md`。

## 1. 能力声明（Capability Statement）
- `C1` 统一平台进入能力：支持用户进入统一审查平台并识别可切换的一级场景。
- `C2` 场景挂接能力：支持把当前激活场景稳定挂接到统一主工作区中，并让平台中的具体场景实例保持唯一当前归属。
- `C3` 顶部全局工具位持续可用能力：支持全局搜索、主题切换、用户入口和其他平台级工具位在场景切换前后保持稳定可用。
- `C4` 场景切换可解释能力：支持在切换场景后稳定说明当前正在使用哪个场景，以及切换后的工作落点。
- `C5` 平台级反馈持续可见能力：支持平台级状态、汇总和切换反馈持续可观察。
- `C6` 多场景平台闭环能力：支持把平台进入、场景切换、顶部工具位、场景挂接与平台反馈接成统一且可扩展的平台主线。
- `N1` 非职责声明：不直接替代具体场景编排层，不直接定义范围树、审核结论、历史差异字段或具体界面实现。

## 2. 边界定义（Boundary / 参数）
- `ENTRY_SCOPE` 平台入口边界：平台允许从统一入口进入，但进入后必须能识别场景入口集合与唯一当前激活场景；不允许进入后处于无场景落点的悬空状态。来源：`C1`。
- `SCENE_INSTANCE_SCOPE` 场景实例边界：统一平台允许挂接不同具体场景，但被挂接对象必须是 `review_workbench` 中已声明、满足平台内具体场景层约束的 `L2` 场景模块；不允许把未声明页面流或临时场景伪装成平台场景实例。来源：`C2 + C6`。
- `MOUNT_SCOPE` 场景挂接边界：被激活场景的承载结构允许变化，但当前场景必须挂接在统一主工作区中，且不能脱离平台外壳独立存在。来源：`C2`。
- `GLOBAL_TOOL_SCOPE` 顶部全局工具位边界：平台级工具位种类允许因实例而变化，但每个工具位都必须保持稳定位置、平台级语义和跨场景可用性。来源：`C3`。
- `SWITCH_SCOPE` 场景切换边界：场景切换前后的当前场景、回退场景和切换结果允许变化，但切换后必须能解释唯一当前工作落点。来源：`C4`。
- `PLATFORM_FEEDBACK_SCOPE` 平台反馈边界：平台级状态变化允许通过状态条、全局汇总或提示区表达，但状态变化后必须至少有一处持续可观察反馈。来源：`C5`。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 平台进入与场景选择基：`L1.M0[R1]`。来源：`C1 + ENTRY_SCOPE`。
- `B2` 合法场景实例集合基：`L2.M0[B1] + L2.M1[B1] + L2.M2[B1]`。来源：`C2 + C6 + SCENE_INSTANCE_SCOPE`。
- `B3` 顶部全局工具位持续可用基：`L1.M0[R2]`。来源：`C2 + C3 + MOUNT_SCOPE + GLOBAL_TOOL_SCOPE + SWITCH_SCOPE`。
- `B4` 场景切换与落点解释基：`L1.M0[R3]`。来源：`C4 + SWITCH_SCOPE + MOUNT_SCOPE`。
- `B5` 平台反馈基：`L1.M0[R4]`。来源：`C5 + PLATFORM_FEEDBACK_SCOPE`。
- `B6` 平台闭环基：`L1.M0[R5]`。来源：`C6 + ENTRY_SCOPE + SCENE_INSTANCE_SCOPE + MOUNT_SCOPE + GLOBAL_TOOL_SCOPE + SWITCH_SCOPE + PLATFORM_FEEDBACK_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R0` 统一平台只能挂接合法场景实例集合中的成员
  - `R0.1` 参与基：`B1 + B2`。
  - `R0.2` 组合方式：统一平台进入后，当前可切换场景和当前被挂接场景都必须取自合法场景实例集合；任一被挂接对象都必须能定位到集合中的唯一成员，不允许在平台层临时拼出并行场景实例。
  - `R0.3` 输出能力：`C1 + C2 + C6`。
  - `R0.4` 边界绑定：`ENTRY_SCOPE/SCENE_INSTANCE_SCOPE`。
- `R1` 统一平台必须先进入再挂接场景
  - `R1.1` 参与基：`B1 + B2 + B3`。
  - `R1.2` 组合方式：完整平台链必须先进入统一平台并识别当前场景，再在主工作区挂接当前激活场景，并保持顶部平台工具位持续可用；不能跳过平台入口，直接进入没有平台上下文的具体场景。
  - `R1.3` 输出能力：`C1 + C2 + C3 + C6`。
  - `R1.4` 边界绑定：`ENTRY_SCOPE/SCENE_INSTANCE_SCOPE/MOUNT_SCOPE/GLOBAL_TOOL_SCOPE`。
- `R2` 场景切换只能在统一平台中发生
  - `R2.1` 参与基：`B2 + B3 + B4 + B6`。
  - `R2.2` 组合方式：场景切换、挂接与回退必须共同服务于统一平台中的当前工作落点，同时顶部全局工具位仍保持平台级语义；不能把它们组织成脱离平台外壳的自由场景跳转。
  - `R2.3` 输出能力：`C2 + C3 + C4 + C6`。
  - `R2.4` 边界绑定：`SCENE_INSTANCE_SCOPE/MOUNT_SCOPE/GLOBAL_TOOL_SCOPE/SWITCH_SCOPE`。
- `R3` 平台反馈必须随场景切换持续可见
  - `R3.1` 参与基：`B5 + B6`。
  - `R3.2` 组合方式：进入平台、切换场景或场景挂接失败之后，平台反馈必须持续带出状态条、全局汇总或平台提示区里的可解释变化；不能让状态只停留在内部状态。
  - `R3.3` 输出能力：`C5 + C6`。
  - `R3.4` 边界绑定：`PLATFORM_FEEDBACK_SCOPE/SWITCH_SCOPE`。
- `R4` 统一平台主线闭环
  - `R4.1` 参与基：`B1 + B2 + B3 + B4 + B5 + B6`。
  - `R4.2` 组合方式：进入统一平台、识别可切换场景、保持顶部全局工具位可用、挂接当前场景，并通过平台反馈确认当前状态，要共同组成“进入平台 -> 选场景 -> 持续使用顶部工具位 -> 挂接场景 -> 持续可见平台反馈”的完整平台主线；不能把任一段拆成脱离前后文的孤立页面或孤立操作。
  - `R4.3` 输出能力：`C1 + C2 + C3 + C4 + C5 + C6`。
  - `R4.4` 边界绑定：`ENTRY_SCOPE/SCENE_INSTANCE_SCOPE/MOUNT_SCOPE/GLOBAL_TOOL_SCOPE/SWITCH_SCOPE/PLATFORM_FEEDBACK_SCOPE`。
- `R5` 无效方案
  - `R5.1` 参与基：`B1 + B2 + B3 + B4 + B5 + B6`。
  - `R5.2` 组合方式：以下方案无效：只有统一平台壳没有场景入口；只有具体场景没有平台入口；进入具体场景后顶部搜索和主题切换丢失平台级语义；切换场景后当前落点不可解释；平台存在状态变化但没有任何持续可观察反馈；把未声明的并行页面流伪装成平台场景实例；把不同具体场景的内部主线直接混成本模块的唯一主线。
  - `R5.3` 失效结论：`N1`。
  - `R5.4` 边界绑定：`ENTRY_SCOPE/SCENE_INSTANCE_SCOPE/MOUNT_SCOPE/GLOBAL_TOOL_SCOPE/SWITCH_SCOPE/PLATFORM_FEEDBACK_SCOPE`。

## 5. 验证（Verification）
- `V1` 平台入口有效性：任一平台进入后，都必须能识别可切换场景集合与唯一当前激活场景。
- `V2` 顶部全局工具位有效性：任一场景被挂接后，顶部全局搜索、主题切换、用户入口等平台级工具位仍必须具备稳定位置与平台语义。
- `V3` 场景挂接有效性：任一当前场景都必须能挂接到统一主工作区，并解释当前场景归属。
- `V4` 场景切换有效性：任一场景切换后都必须能解释当前工作落点。
- `V5` 平台反馈可见性有效性：任一平台级状态变化之后，必须至少有一处持续可观察反馈给出可解释变化。
- `V6` 平台承接有效性：统一平台中的场景挂接主链必须能追溯到 `L1-M0`，且当前挂接场景必须能定位到合法场景实例集合中的唯一成员，而不是在本模块中重新拼出并行场景真相源。
- `V7` 最小必要性：移除任一 `B*` 后，`V1~V6` 至少一项失效。
- `V8` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
