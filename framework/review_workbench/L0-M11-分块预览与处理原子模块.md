# 分块预览与处理原子模块:ReviewWorkbenchFilePreviewAndChunkProcessingAtoms

@framework

本模块定义 `review_workbench` 中文件库场景最小的分块预览与处理结构。推导链为：`framework/review_workbench/L0-M3-工作集与工作面切换原子模块.md` + `framework/review_workbench/L0-M4-对象列表与查看入口共享底座原子模块.md` + `framework/review_workbench/L0-M5-全局反馈与状态汇总原子模块.md` -> 本模块。也就是说，文件库场景中的“打开文件 -> 进入分块预览 -> 查看分块 -> 在当前块上做纠错或确认”这组结构，不单独发明一套新的工作面真相，而是在工作面切换、对象承接与反馈结构之上，收束成分块预览与处理这组领域原子。

本模块位于 `review_workbench` 的原子层。它负责把分块预览、分块序列、当前块、块级处理动作和块级反馈表达清楚，回答“当前正在预览哪个块”“该块属于哪个当前文件”“可以对该块做什么”“动作后如何回到当前处理上下文”这几个最小问题；它不定义具体预览控件，不限定 PDF、Word 或图片等具体实现，也不在这里展开来源定位本身。

## 1. 能力声明（Capability Statement）
- `C1` 分块预览工作面表达能力：将当前文件、当前块预览承接面和预览上下文组织成稳定结构。
- `C2` 分块序列表达能力：将块序列、块顺序、块边界和当前块组织成稳定结构。
- `C3` 当前块处理动作表达能力：将纠错、分类、确认、标记等块级动作组织成稳定结构。
- `C4` 当前块反馈表达能力：将块级状态变化、处理提示和局部反馈组织成稳定结构。
- `N1` 非职责声明：不定义具体文件格式预览控件，不定义来源文件定位逻辑，不替代文件清单工作面。

## 2. 边界定义（Boundary / 参数）
- `PREVIEW_SCOPE` 分块预览边界：预览承接面形式和预览上下文允许变化，但任一时刻都必须能识别当前文件与当前块预览上下文。
- `CHUNK_SCOPE` 分块边界：块数量、顺序和边界形式允许变化，但任一分块序列都必须能识别唯一当前块及其顺序关系。
- `CHUNK_ACTION_SCOPE` 块级动作边界：动作类型和动作集合允许变化，但每个块级动作都必须附着当前块或当前块集合，并能解释动作结果。
- `CHUNK_FEEDBACK_SCOPE` 块级反馈边界：反馈内容和位置允许变化，但动作执行后必须至少有一处反馈能够解释当前块状态变化。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 分块预览工作面基：`L0.M3[R1] + L0.M4[R1]`。来源：`C1 + PREVIEW_SCOPE`。
- `B2` 分块序列基：`L0.M4[R1,R2]`。来源：`C2 + CHUNK_SCOPE`。
- `B3` 块级动作基：`L0.M4[R2]`。来源：`C3 + CHUNK_ACTION_SCOPE`。
- `B4` 块级反馈基：`L0.M5[R2,R3]`。来源：`C4 + CHUNK_FEEDBACK_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 分块预览必须带出当前块
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：任何分块预览工作面都必须能说明当前块属于哪个文件、文件被怎样分块以及当前正在处理哪个块；不能只有预览容器而没有当前块语义。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`PREVIEW_SCOPE/CHUNK_SCOPE`。
- `R2` 块级动作必须附着当前块
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：纠错、分类、确认、标记等块级动作必须附着当前块或当前块集合，不能写成脱离当前块上下文的黑盒操作。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`CHUNK_SCOPE/CHUNK_ACTION_SCOPE`。
- `R3` 块级动作之后必须有块级反馈
  - `R3.1` 参与基：`B3 + B4`。
  - `R3.2` 组合方式：块级动作执行后，至少一处局部反馈区必须能说明当前块状态如何变化；不能只在内部状态里变化。
  - `R3.3` 输出能力：`C3 + C4`。
  - `R3.4` 边界绑定：`CHUNK_ACTION_SCOPE/CHUNK_FEEDBACK_SCOPE`。
- `R4` 分块预览与处理闭环
  - `R4.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R4.2` 组合方式：分块预览、分块序列、当前块、块级动作和块级反馈必须共同组成“进入分块预览 -> 识别当前块 -> 对当前块执行处理 -> 看到块级反馈”的稳定闭环。
  - `R4.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R4.4` 边界绑定：`PREVIEW_SCOPE/CHUNK_SCOPE/CHUNK_ACTION_SCOPE/CHUNK_FEEDBACK_SCOPE`。
- `R5` 无效方案
  - `R5.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R5.2` 组合方式：以下方案无效：只有分块预览没有当前块；只有分块列表没有当前块预览；块级动作存在但无法说明作用于哪一个块；动作执行后没有任何块级反馈。
  - `R5.3` 失效结论：`N1`。
  - `R5.4` 边界绑定：`PREVIEW_SCOPE/CHUNK_SCOPE/CHUNK_ACTION_SCOPE/CHUNK_FEEDBACK_SCOPE`。

## 5. 验证（Verification）
- `V1` 分块预览有效性：任一时刻都必须能识别当前文件、当前块预览与唯一当前块。
- `V2` 块级动作归属有效性：任一块级动作都必须能追溯到当前块或当前块集合。
- `V3` 块级反馈可见性：任一块级动作之后，必须至少有一处局部反馈发生可解释变化。
- `V4` 最小必要性：移除任一 `B*` 后，`V1~V3` 至少一项失效。
- `V5` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
