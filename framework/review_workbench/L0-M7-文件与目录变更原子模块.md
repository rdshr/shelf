# 文件与目录变更原子模块:ReviewWorkbenchFileAndDirectoryMutationAtoms

@framework

本模块定义审查处理工作台中的“文件与目录改动前端承接”结构原子。它回答“当前范围下哪些文件与目录可以被前端识别为可改动对象”“这次改动在前端承接中到底在改什么”以及“改动后树、焦点和局部信息如何在前端回流”这三个最小问题。它不负责查询条件、不负责列表审查，也不负责具体接口路径，只负责把文件与目录改动沉成稳定前端原子。

本模块位于 `review_workbench` 的领域原子层，但它和 `review_workbench.L0.M0~M5` 的定位不同：前者更多是在 `frontend` 通用原子之上收束前端领域结构，而本模块定义的是工作台领域特有的“文件 / 目录对象身份 + 前端改动承接 + 前端回流影响”结构。

不过它并不是悬空自成体系的专有原子，而是明确建立在现有 `review_workbench` 共享原子之上：其中 `L0-M1` 提供当前范围树、当前范围和局部工具位的稳定上下文，`L0-M4` 提供文件 / 目录对象、对象动作与查看入口的共享对象底座，`L0-M5` 提供改动后的局部 / 全局反馈与状态回流底座。本模块是在这些既有原子之上，把“文件与目录改动”这条领域专有支路单独沉成稳定结构，而不是另起一套并行前端真相源。

它与 `review_workbench_backend.L0.M3` 一起构成范围改动支路的领域专有底座：`L0.M7` 负责说明“前端识别改什么、前端承接怎么改、前端回流后影响什么”，`review_workbench_backend.L0.M3` 负责说明“这些改动怎样稳定映射成后端契约请求、树更新与查询失效回流”。这两层应保持领域专有，而不是被前端通用原子吸收。

## 1. 能力声明（Capability Statement）

- `C1` 文件与目录对象前端表达能力：将文件、目录、父子关系与稳定对象身份组织成稳定前端对象结构。
- `C2` 改动动作前端承接能力：将上传、新建目录、删除、移动与重命名组织成稳定前端改动结构。
- `C3` 位置与影响前端表达能力：将原位置、目标位置、影响范围与受影响节点组织成稳定前端位置影响结构。
- `C4` 回流前端承接能力：将改动后的树更新、焦点回退、局部信息变化与可观察反馈组织成稳定前端回流结构。
- `N1` 非职责声明：不负责查询表单、不负责结果列结构、不负责具体服务协议与权限细则。

## 2. 边界定义（Boundary / 参数）

- `OBJECT` 对象边界：对象类型和展示信息允许变化，但每个文件或目录都必须具备稳定对象标识，且改动前后可追踪为同一对象。
- `CONTAINER` 容器边界：父子节点数量、层级深度和挂接方式允许变化，但任一对象都必须能解释当前父节点与层级位置，并能回接到当前范围树中的对应节点。
- `PLACEMENT` 位置边界：原位置、目标位置和父节点允许变化，但移动前后都必须能同时说明旧位置与新位置。
- `MUTATION_VIEW` 改动边界：上传、新建目录、删除、移动和重命名的前端动作类型允许变化，但每次改动都必须具备明确目标对象、当前范围 / 父节点上下文与最小前端改动承接组合。
- `IMPACT` 影响边界：受影响节点、局部信息和关联状态的范围允许变化，但每次改动后都必须能解释至少一组受影响节点或状态变化。
- `RETURN` 回流边界：焦点回退路径、树刷新方式和反馈位置允许变化，但改动完成后必须能解释当前焦点落点，并至少出现一处可观察回流。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 文件 / 目录对象与范围挂接基：`L0.M1[R1,R2] + L0.M4[R1,R2]`。由当前范围树、当前范围、文件 / 目录对象身份与父子关系共同构成的对象挂接结构。来源：`C1 + OBJECT + CONTAINER`。
- `B2` 改动动作承接基：`L0.M1[R2] + L0.M1[R3] + L0.M4[R2,R3]`。由当前范围、局部工具位、目标对象、父节点上下文与最小前端改动入口共同构成的改动承接结构。来源：`C2 + OBJECT + CONTAINER + MUTATION_VIEW`。
- `B3` 改动回流基：`L0.M1[R1,R3] + L0.M5[R1,R2,R3]`。由位置变化、范围树更新、焦点回退、局部信息变化与可观察反馈共同构成的回流结构。来源：`C3 + C4 + PLACEMENT + IMPACT + RETURN`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 改动必须附着稳定对象关系
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：任一文件与目录改动都必须能说明“改的是谁、当前挂在哪、这次准备怎么改”；不能只有一个动作名而没有稳定对象关系。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`OBJECT/CONTAINER/MUTATION_VIEW`。

- `R2` 位置变化必须可解释
  - `R2.1` 参与基：`B1 + B2 + B3`。
  - `R2.2` 组合方式：上传、新建目录、移动、删除或重命名之后，系统必须能解释对象原来在哪、现在在哪、哪些节点因此受影响；移动尤其必须同时说明旧位置与新位置。
  - `R2.3` 输出能力：`C1 + C2 + C3`。
  - `R2.4` 边界绑定：`OBJECT/PLACEMENT/IMPACT/RETURN`。

- `R3` 重命名与移动不得改写对象身份
  - `R3.1` 参与基：`B1 + B2`。
  - `R3.2` 组合方式：重命名只改变显示语义，移动只改变挂接位置，这两类改动都不得把同一对象伪装成另一个对象；对象身份必须稳定可追踪。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`OBJECT/PLACEMENT/MUTATION_VIEW`。

- `R4` 删除必须解释回退与影响
  - `R4.1` 参与基：`B2 + B3`。
  - `R4.2` 组合方式：删除文件或目录后，前端承接结构必须能解释当前焦点回到哪个节点、哪些局部信息失效或重算、树上哪些节点发生了变化；不能只剩一个抽象“删除成功”，也不能把服务返回字段直接写回本模块。
  - `R4.3` 输出能力：`C2 + C3 + C4`。
  - `R4.4` 边界绑定：`MUTATION_VIEW/IMPACT/RETURN`。

- `R5` 文件与目录改动原子闭环
  - `R5.1` 参与基：`B1 + B2 + B3`。
  - `R5.2` 组合方式：对象关系、改动动作与回流承接必须共同形成“识别对象 -> 发起改动 -> 更新位置与影响 -> 落回当前焦点”的稳定结构。
  - `R5.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R5.4` 边界绑定：`OBJECT/CONTAINER/PLACEMENT/MUTATION_VIEW/IMPACT/RETURN`。

- `R6` 无效方案
  - `R6.1` 参与基：`B1 + B2 + B3`。
  - `R6.2` 组合方式：以下方案无效：只有上传按钮没有落点；只有目录名修改没有稳定对象身份；移动后只能看见新位置看不见旧位置；删除后当前焦点不可解释；改动成功但树和局部信息没有任何可观察回流；`L0-M7` 自己重写了一套脱离 `L0-M1/L0-M4/L0-M5` 的并行对象或反馈底座；把后端结构改动请求字段或失效回流字段直接写回本模块。
  - `R6.3` 失效结论：`N1`。
  - `R6.4` 边界绑定：`OBJECT/CONTAINER/PLACEMENT/MUTATION_VIEW/IMPACT/RETURN`。

## 5. 验证（Verification）

- `V1` 对象关系可识别性：任一文件或目录都必须能识别对象身份、当前父节点与当前挂接关系。
- `V2` 变更动作有效性：任一上传、新建目录、删除、移动或重命名都必须能说明目标对象与最小改动载荷。
- `V3` 位置影响一致性：任一范围改动都必须能同时解释位置变化、至少一组受影响节点与当前焦点回流。
- `V4` 身份稳定性：移动与重命名后对象身份必须保持稳定。
- `V5` 共享原子承接有效性：文件 / 目录改动的对象挂接、动作入口和回流反馈必须能分别追溯到 `L0-M1`、`L0-M4` 和 `L0-M5`，而不是在本模块中重写并行底座。
- `V6` 最小必要性：移除任一 `B*` 后，`V1~V5` 至少一项失效。
- `V7` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
