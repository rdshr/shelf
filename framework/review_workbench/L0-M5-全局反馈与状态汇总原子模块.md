# 全局反馈与状态汇总原子模块:ReviewWorkbenchGlobalFeedbackAndStatusSummaryAtoms

@framework

本模块定义 `review_workbench` 领域最小的全局反馈与状态汇总结构。推导链为：`framework/frontend/L0-M0-承载与绑定原子模块.md` + `framework/frontend/L1-M2-展示与容器原子模块.md` + `framework/frontend/L1-M4-标记与反馈原子模块.md` -> 本模块。也就是说，工作台中的状态条、全局汇总和持续反馈，不单独重写一套全局提示基础，而是把前端已有的承载壳、展示容器和标记反馈约束成“工作台级反馈”这组领域原子。

本模块定义审查处理工作台中的全局反馈与状态汇总结构原子。它回答“工作台当前整体状态如何被持续表达”以及“处理结果如何被回流到用户可观察区域”这两个最小问题。它不负责结果表格本身，只负责把状态条、全局计数、反馈提示与汇总结构沉成稳定反馈原子。

## 1. 能力声明（Capability Statement）

- `C1` 全局汇总表达能力：将全局计数、总体状态与持续汇总信息组织成稳定结构。
- `C2` 反馈提示表达能力：将处理成功、失败、空态、进行中等反馈组织成稳定表达。
- `C3` 状态持续承接能力：将工作台级反馈挂接在稳定位置并持续反映当前状态。
- `N1` 非职责声明：不负责定义具体动作，不负责范围树与查询条件。

## 2. 边界定义（Boundary / 参数）

- `GLOBALSUMMARY` 全局汇总边界：计数项、总体状态和汇总项数量允许变化，但必须至少保留一处可解释的工作台级汇总承接。
- `GLOBALFEEDBACK` 全局反馈边界：成功、失败、空态、处理中和提示信息的组合允许变化，但状态变化发生后必须至少导出一处可观察反馈。
- `STATUSBAR` 状态条边界：反馈承接位置允许采用不同工作台级区域，但必须存在稳定承接位置，且不能随单次动作丢失。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 全局汇总基：frontend.L1.M2[R1,R2] + frontend.L1.M4[R1]。由展示容器、汇总文本和状态标记约束成全局计数、汇总项与总体状态结构。来源：`C1 + GLOBALSUMMARY`。
- `B2` 全局反馈基：frontend.L1.M4[R1,R2]。由成功、失败、空态与处理中反馈约束成全局反馈结构。来源：`C2 + GLOBALFEEDBACK`。
- `B3` 状态条承接基：frontend.L0.M0[R1,R2] + frontend.L1.M2[R2]。由稳定承载面和展示容器约束成工作台级状态条位置与持续状态承接关系。来源：`C3 + STATUSBAR`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 汇总必须有稳定位置
  - `R1.1` 参与基：`B1 + B3`。
  - `R1.2` 组合方式：在 frontend 的承载壳和展示容器结构上，全局汇总必须挂接到稳定的状态条或全局反馈区，不能只以漂浮数字存在。
  - `R1.3` 输出能力：`C1 + C3`。
  - `R1.4` 边界绑定：`GLOBALSUMMARY/STATUSBAR`。

- `R2` 反馈必须进入全局承接
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：在 frontend 的标记反馈和承载结构上，处理成功、失败、空态或进行中反馈必须进入稳定承接位置，不能停留在不可观察状态。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`GLOBALFEEDBACK/STATUSBAR`。

- `R3` 全局反馈原子闭环
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：全局汇总、反馈提示与状态条承接必须共同形成“有状态变化 -> 有全局表达 -> 有稳定承接位置”的反馈结构。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`GLOBALSUMMARY/GLOBALFEEDBACK/STATUSBAR`。

- `R4` 无效方案
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：以下方案无效：跳过 frontend 承载壳与反馈结构直接声明工作台全局状态；只有汇总数字没有承接位置；只有提示消息没有全局状态语义；反馈出现后无法持续表达当前工作台状态。
  - `R4.3` 失效结论：`N1`。
  - `R4.4` 边界绑定：`GLOBALSUMMARY/GLOBALFEEDBACK/STATUSBAR`。

## 5. 验证（Verification）

- `V1` 全局汇总可识别性：工作台中必须至少存在一处可识别全局汇总，能够说明全局计数或总体状态。
- `V2` 反馈可观察性：成功、失败、空态或处理中状态发生后，必须至少有一处可观察反馈。
- `V3` 稳定承接性：反馈与汇总都必须挂接在稳定工作台级位置，且不能随单次动作丢失。
- `V4` 前端承接有效性：全局汇总、反馈提示和状态条承接必须能追溯到 frontend 的承载绑定、展示容器和标记反馈结构。
- `V5` 最小必要性：移除任一 `B*` 后，`V1~V4` 至少一项失效。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
