# 审核工作栏与队列概览原子模块:ReviewWorkbenchReviewSidebarAndQueueOverviewAtoms

@framework

本模块定义 `review_workbench` 领域最小的“审核工作栏与队列概览”前端结构。它回答“审核型场景左侧工作栏如何承接当前审核对象”“审核说明与提交入口如何稳定附着当前审核焦点”以及“待审核 / 已审核列表与局部队列概览如何被持续表达”这三个最小问题。它不负责查询契约、不负责审核动作请求编码，也不替代中部审核工作面中的只读预览或审核结果回流本身。

本模块位于 `review_workbench` 的领域原子层。它不重写前端基础容器、对象列表或全局反馈原子，而是在 `L0-M4` 的对象承接能力与 `L0-M5` 的反馈承接能力之上，把审核型场景里稳定复用的左侧工作栏、当前对象摘要、未选择态、审核输入区、提交入口和待审 / 已审列表沉成共享结构，为提交审核、终审、抽检和复核等审核型场景提供统一底座。

对于“审核结论输入本身怎样成立、审核完成后对象状态怎样回流”这件事，本模块默认继续承接 `L0-M8`；本模块关注的是这些审核语义在左侧工作栏和局部队列概览里如何被稳定承载，而不是重新定义审核结论字段或结果编码。

## 1. 能力声明（Capability Statement）
- `C1` 审核工作栏承接能力：支持在审核型场景中稳定承接左侧审核工作栏与局部工作区。
- `C2` 当前审核对象摘要能力：支持显示当前审核对象、当前状态、必要上下文以及未选择态。
- `C3` 审核说明输入与提交入口承接能力：支持审核说明、结论输入和提交动作入口稳定附着当前审核对象。
- `C4` 审核队列概览能力：支持待审核、已审核和其他局部审核队列以列表、分区或摘要形式稳定成立。
- `C5` 审核工作栏局部回流能力：支持审核提交后在工作栏或队列概览中看到状态、计数、提示或选择态变化。
- `N1` 非职责声明：不直接替代中部对象列表，不直接定义审核动作请求 / 返回字段，也不直接替代全局反馈区。

## 2. 边界定义（Boundary / 参数）
- `REVIEW_SIDEBAR_SCOPE` 审核工作栏边界：审核型场景的左侧承载布局、局部区块顺序和展示方式允许变化，但必须存在稳定审核工作栏承接面。来源：`C1`。
- `CURRENT_REVIEW_OBJECT_SCOPE` 当前审核对象边界：当前对象可以是文件、提交项或其他审核对象，但任一时刻都必须能解释唯一当前审核焦点，或明确处于未选择态。来源：`C2`。
- `DECISION_ENTRY_SCOPE` 结论输入入口边界：说明输入、结论类型和提交入口允许变化，但每次输入或提交都必须附着当前审核对象，未选择态下必须能解释为什么不可提交。来源：`C3`。
- `QUEUE_OVERVIEW_SCOPE` 队列概览边界：待审核、已审核和其他局部队列可以表现为列表、摘要块或计数区，但每一项都必须能解释当前审核队列归属，并且列表项必须能追溯到对应审核对象。来源：`C4`。
- `SIDEBAR_FEEDBACK_SCOPE` 工作栏反馈边界：审核后的状态变化、提示、计数或当前选择态变化允许落在工作栏中的不同区域，但审核动作完成后必须至少有一处持续可观察反馈。来源：`C5`。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 审核工作栏承接基：`L0.M4[R1,R2] + L0.M5[R3]`。来源：`C1 + REVIEW_SIDEBAR_SCOPE`。
- `B2` 当前审核对象摘要基：`L0.M4[R1,R2] + L0.M5[R1]`。来源：`C2 + CURRENT_REVIEW_OBJECT_SCOPE`。
- `B3` 审核输入与提交入口基：`L0.M8[R1,R2] + L0.M4[R3]`。来源：`C3 + DECISION_ENTRY_SCOPE`。
- `B4` 审核队列概览基：`L0.M4[R1,R2,R3] + L0.M5[R1,R3]`。来源：`C4 + QUEUE_OVERVIEW_SCOPE`。
- `B5` 工作栏局部回流基：`L0.M8[R3] + L0.M5[R2,R3]`。来源：`C5 + SIDEBAR_FEEDBACK_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 审核工作栏必须导出当前审核焦点
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：审核型场景中的左侧工作栏必须能说明当前审核对象是谁，或明确说明当前处于未选择态；不能只有一个输入区容器，而没有当前审核焦点语义。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`REVIEW_SIDEBAR_SCOPE/CURRENT_REVIEW_OBJECT_SCOPE`。
- `R2` 审核说明输入与提交入口必须附着当前审核对象
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：说明输入、结论输入和提交入口都必须附着当前审核对象；在未选择态下可以保持不可提交，但必须能解释当前不可提交原因；不能出现脱离审核对象归属的孤立提交按钮。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`CURRENT_REVIEW_OBJECT_SCOPE/DECISION_ENTRY_SCOPE`。
- `R3` 待审核与已审核列表必须附着审核工作栏上下文
  - `R3.1` 参与基：`B1 + B4`。
  - `R3.2` 组合方式：待审核、已审核和其他局部队列概览必须作为审核工作栏中的局部分区、局部列表或局部摘要存在，并能说明其审核队列归属；若以列表承接，则每个列表项都必须能说明对应审核对象来源；不能出现脱离审核工作栏上下文的游离数字或游离分区标题。
  - `R3.3` 输出能力：`C1 + C4`。
  - `R3.4` 边界绑定：`REVIEW_SIDEBAR_SCOPE/QUEUE_OVERVIEW_SCOPE`。
- `R4` 审核动作完成后工作栏必须有局部回流
  - `R4.1` 参与基：`B3 + B5`。
  - `R4.2` 组合方式：审核动作完成后，当前对象状态、队列计数、提示信息或当前选择态至少要有一处在审核工作栏内持续可观察地变化；不能只在中部列表或内部状态中变化而工作栏无感。
  - `R4.3` 输出能力：`C3 + C5`。
  - `R4.4` 边界绑定：`DECISION_ENTRY_SCOPE/SIDEBAR_FEEDBACK_SCOPE`。
- `R5` 审核工作栏与队列概览原子闭环
  - `R5.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R5.2` 组合方式：审核工作栏承接、当前审核对象摘要、说明输入与提交入口、待审核 / 已审核列表和局部回流必须共同组成“识别当前审核焦点 -> 在工作栏中查看待审 / 已审队列 -> 输入说明 / 给出结论 -> 提交动作 -> 在工作栏中看到状态与队列变化”的稳定结构；其中动作请求与结果编码继续由 `review_workbench_backend` 承担。
  - `R5.3` 输出能力：`C1 + C2 + C3 + C4 + C5`。
  - `R5.4` 边界绑定：`REVIEW_SIDEBAR_SCOPE/CURRENT_REVIEW_OBJECT_SCOPE/DECISION_ENTRY_SCOPE/QUEUE_OVERVIEW_SCOPE/SIDEBAR_FEEDBACK_SCOPE`。
- `R6` 无效方案
  - `R6.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R6.2` 组合方式：以下方案无效：只有审核说明输入框没有当前审核对象；只有当前对象标签没有输入区或提交入口；存在待审核 / 已审核标题但无法说明队列归属；审核动作完成后工作栏内没有任何局部回流；把审核动作请求字段或服务返回字段直接写回本模块。
  - `R6.3` 失效结论：`N1`。
  - `R6.4` 边界绑定：`REVIEW_SIDEBAR_SCOPE/CURRENT_REVIEW_OBJECT_SCOPE/DECISION_ENTRY_SCOPE/QUEUE_OVERVIEW_SCOPE/SIDEBAR_FEEDBACK_SCOPE`。

## 5. 验证（Verification）
- `V1` 审核工作栏可识别性：审核型场景中必须存在可识别审核工作栏承接面，并能说明当前审核焦点或未选择态。
- `V2` 输入归属有效性：任一审核说明输入或提交入口都必须能说明附着的当前审核对象，或明确处于不可提交的未选择态。
- `V3` 队列概览可解释性：任一待审核、已审核或其他局部队列概览都必须能说明审核队列归属；若以列表承接，则每个列表项都必须能说明对应审核对象来源。
- `V4` 局部回流可见性：任一审核动作完成后，工作栏内必须至少有一处持续可观察反馈出现可解释变化。
- `V5` 共享原子有效性：本模块必须能被提交审核、终审、抽检或复核等审核型场景复用，而不是被误写成某一个具体场景的唯一真相源。
- `V6` 最小必要性：移除任一 `B*` 后，`V1~V5` 至少一项失效。
- `V7` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
