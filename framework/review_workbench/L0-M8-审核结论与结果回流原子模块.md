# 审核结论与结果回流原子模块:ReviewWorkbenchReviewDecisionAndResultFlowAtoms

@framework

本模块定义 `review_workbench` 领域最小的“审核结论输入与可观察回流”结构。它回答“当前审核对象如何作为结论输入的附着对象”“审核结论输入如何在前端稳定表达”以及“审核完成后状态如何回流到可观察区域”这三个最小问题。它不负责范围树、不负责文件块处理细节，也不负责历史差异本身。

本模块位于 `review_workbench` 的领域原子层。它不重写前端的基础表单或反馈原子，也不承担后端契约请求 / 返回本身；它把当前审核对象的结论附着关系、审核结论输入、审核意见呈现和结果回流收束成稳定的前端领域结构，为提交审核、终审、抽检等审核型场景提供共享底座。凡是“结论怎样映射成动作请求、结果怎样映射成后端契约回流”的问题，统一交给外部领域 `review_workbench_backend.L0.M4`。

对于“待审核 / 已审核列表怎样在侧栏中承接、当前审核焦点怎样在工作栏中切换、局部队列概览怎样成立”这件事，本模块默认不负责，统一交给 `L0-M13`；本模块只负责结论输入与结果回流本身。

## 1. 能力声明（Capability Statement）
- `C1` 当前审核对象附着能力：将提交项、审核对象和审核上下文组织成稳定的“当前结论附着对象”结构。
- `C2` 审核结论输入表达能力：将通过、驳回、退回、补充意见等审核结论输入组织成稳定前端输入结构。
- `C3` 审核结果可观察回流能力：将审核完成后的状态变化、统计变化和提示反馈组织成稳定前端回流结构。
- `N1` 非职责声明：不负责具体审核规则、不负责权限细则、不负责历史差异查看。

## 2. 边界定义（Boundary / 参数）
- `REVIEW_OBJECT` 审核对象边界：提交项种类和审核上下文允许变化，但任一审核结论输入都必须附着稳定当前审核对象身份；待审 / 已审列表与工作栏焦点切换不在本边界中定义。 
- `DECISION_INPUT_SCOPE` 审核结论输入边界：结论类型、附加意见形式和前端输入承载方式允许变化，但每次结论表达都必须具备可解释结论类型与最小输入组合。
- `OBSERVABLE_FLOW_SCOPE` 结果回流边界：状态变化、统计变化和反馈位置允许变化，但审核完成后必须至少有一处可观察回流能够解释对象状态变化。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 当前审核对象基：`L0.M4[R1,R2]`。来源：`C1 + REVIEW_OBJECT`。
- `B2` 审核结论输入基：`L0.M4[R2] + L0.M5[R2]`。来源：`C2 + DECISION_INPUT_SCOPE`。
- `B3` 审核回流基：`L0.M5[R1,R3]`。来源：`C3 + OBSERVABLE_FLOW_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 审核结论输入必须附着当前审核对象
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：任何审核结论输入都必须能说明作用于哪个当前提交项或当前审核对象，不能只有结论词而没有对象归属；这里约束的是前端输入表达如何附着当前对象，而不是服务动作请求如何编码；至于当前对象怎样从待审 / 已审队列中被选中，由 `L0-M13` 单独承担。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`REVIEW_OBJECT/DECISION_INPUT_SCOPE`。
- `R2` 审核完成后必须有结果回流
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：审核完成后，提交项状态、统计或提示反馈至少要有一处能说明变化，不能只在内部状态里完成；这里约束的是前端哪些区域必须持续可观察，而不是服务返回字段如何命名。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`DECISION_INPUT_SCOPE/OBSERVABLE_FLOW_SCOPE`。
- `R3` 审核结论与回流原子闭环
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：当前审核对象附着关系、审核结论输入和结果回流必须共同形成“识别当前对象 -> 给结论 -> 看到回流”的稳定前端审核结构；其中动作请求与结果编码由 `review_workbench_backend.L0.M4` 单独承担，而工作栏与审核队列承接由 `L0-M13` 单独承担。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`REVIEW_OBJECT/DECISION_INPUT_SCOPE/OBSERVABLE_FLOW_SCOPE`。
- `R4` 无效方案
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：以下方案无效：只有审核按钮没有当前审核对象；只有结论词没有可解释输入承接；只有结论记录没有可观察回流；审核完成后无法说明对象状态变化；把工作栏队列切换直接写回本模块；把服务动作请求字段直接写回本模块。
  - `R4.3` 失效结论：`N1`。
  - `R4.4` 边界绑定：`REVIEW_OBJECT/DECISION_INPUT_SCOPE/OBSERVABLE_FLOW_SCOPE`。

## 5. 验证（Verification）
- `V1` 当前审核对象可识别性：任一审核结论输入都必须能追溯到明确当前审核对象身份。
- `V2` 审核结论输入有效性：任一审核结论表达都必须能说明结论类型与最小输入组合。
- `V3` 结果回流可见性：审核完成后必须至少有一处持续可观察反馈出现可解释变化。
- `V4` 最小必要性：移除任一 `B*` 后，`V1~V3` 至少一项失效。
- `V5` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
