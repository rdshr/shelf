# 对象动作与结果回流后端契约原子模块:ReviewWorkbenchBackendObjectActionAndResultFlowServiceContracts

@framework

本模块定义 `review_workbench_backend` 中与“对象动作提交及其结果回流”直接相关的最小稳定后端契约。它回答“当前场景下对哪个对象或对象集合做了什么动作、动作最少要携带什么载荷、结果最少如何表达成功失败，以及动作完成后哪些对象、摘要、范围或刷新要求需要稳定回流”。

本模块默认建立在 `review_workbench_backend.L0.M0` 之上，延续统一后端契约骨架来承接动作请求、动作结果与结果回流。

## 1. 能力声明（Capability Statement）
- `C1` 动作请求整理能力。
- `C2` 动作载荷表达能力。
- `C3` 动作结果表达能力。
- `C4` 动作影响回流能力。
- `N1` 非职责声明：不负责查询列表本身、不负责历史序列。

## 2. 边界定义（Boundary / 参数）
- `ACTION_REQUEST_SCOPE` 动作请求边界：任一动作请求都必须说明当前场景、动作类型、目标对象与当前进入上下文。
- `ACTION_PAYLOAD_SCOPE` 动作载荷边界：任一动作请求可以带意见、原因、参数、标签或确认选项，但只能承载本次动作成立所需的最小载荷。
- `ACTION_RESULT_SCOPE` 动作结果边界：任一动作返回都必须说明当前动作结果状态与最小解释信息。
- `ACTION_EFFECT_SCOPE` 动作影响边界：任一动作返回只要已经影响当前工作上下文，就必须至少回流一组可解释影响。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 动作请求基：来源：`C1 + ACTION_REQUEST_SCOPE`。
- `B2` 动作载荷基：来源：`C2 + ACTION_PAYLOAD_SCOPE`。
- `B3` 动作结果基：来源：`C3 + ACTION_RESULT_SCOPE`。
- `B4` 动作影响基：来源：`C4 + ACTION_EFFECT_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 任一动作请求都必须附着明确动作类型与目标对象
  - `R1.1` 参与基：`L0.M0[B1,B2] + B1`。
  - `R1.2` 组合方式：任一动作请求都必须说明自己属于哪个当前场景、要执行什么动作、动作作用于哪个对象、对象集合或当前工作项，以及从哪个当前上下文进入。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`ACTION_REQUEST_SCOPE`。
- `R2` 动作载荷只能承载动作所需最小语义
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：动作载荷只能用于支撑本次动作成立，不能把查询条件、查看上下文、历史差异内容或结构改动语义混入动作载荷。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`ACTION_REQUEST_SCOPE/ACTION_PAYLOAD_SCOPE`。
- `R3` 任一动作返回都必须表达明确结果状态并回流影响
  - `R3.1` 参与基：`B3 + B4`。
  - `R3.2` 组合方式：任一动作返回都必须说明本次动作是成功、失败、部分成功、拒绝执行还是需要补充信息，并在影响当前工作上下文时至少回流一组对象变化、摘要变化、刷新要求或失效提示。
  - `R3.3` 输出能力：`C3 + C4`。
  - `R3.4` 边界绑定：`ACTION_RESULT_SCOPE/ACTION_EFFECT_SCOPE`。

## 5. 验证（Verification）
- `V1` 动作请求有效性：任一动作请求都必须能说明当前场景、动作类型、目标对象与进入上下文。
- `V2` 动作载荷有效性：任一动作载荷都必须只承载本次动作成立所需的最小语义。
- `V3` 动作结果有效性：任一动作返回都必须能说明稳定结果状态。
- `V4` 动作影响回流有效性：任一影响当前工作上下文的动作结果都必须至少回流一组可解释影响。
- `V5` 最小必要性：移除任一 `B*` 后，`V1~V4` 至少一项失效。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
