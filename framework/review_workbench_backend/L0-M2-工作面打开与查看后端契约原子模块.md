# 工作面打开与查看后端契约原子模块:ReviewWorkbenchBackendWorksetOpenAndInspectionServiceContracts

@framework

本模块定义 `review_workbench_backend` 中与“对象打开、工作面承接、查看与定位恢复”直接相关的最小稳定后端契约。它回答“从对象列表或历史节点打开当前对象时，请求最少要带什么，以及查看内容、锚点、来源定位和恢复信息怎样稳定返回”。

本模块默认建立在 `review_workbench_backend.L0.M0` 之上，延续统一后端契约骨架来承接对象打开、工作面承接、查看请求与查看返回。

## 1. 能力声明（Capability Statement）
- `C1` 对象打开请求整理能力：把当前对象、进入来源、工作面类型与进入上下文整理成稳定请求。
- `C2` 工作面承接返回能力：把打开后的对象承接面、上下文身份和恢复信息整理成稳定返回。
- `C3` 查看请求整理能力：把来源查看、审核查看、历史查看等查看目标与定位上下文整理成稳定请求。
- `C4` 查看与定位恢复返回能力：把查看承接内容、锚点和定位恢复信息整理成稳定返回。
- `N1` 非职责声明：不直接定义查询结果集、动作提交、结构改动或历史差异组织方式。

## 2. 边界定义（Boundary / 参数）
- `OPEN_REQUEST_SCOPE` 打开请求边界：任一打开请求都必须说明从哪个列表、节点或上下文打开哪个对象以及要进入哪类工作面。
- `WORKSET_BINDING_SCOPE` 工作面承接边界：打开后的对象承接面允许因场景不同而变化，但必须能说明当前工作面身份与对象归属。
- `INSPECTION_REQUEST_SCOPE` 查看请求边界：任一查看请求都必须说明当前对象、查看类型与定位上下文。
- `INSPECTION_RESPONSE_SCOPE` 查看返回边界：任一查看返回都必须说明承接内容、锚点与最小恢复信息。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 打开请求基：来源：`C1 + OPEN_REQUEST_SCOPE`。
- `B2` 工作面承接基：来源：`C2 + WORKSET_BINDING_SCOPE`。
- `B3` 查看请求基：来源：`C3 + INSPECTION_REQUEST_SCOPE`。
- `B4` 查看返回基：来源：`C4 + INSPECTION_RESPONSE_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 打开工作面必须带对象来源上下文
  - `R1.1` 参与基：`L0.M0[B1,B2] + B1 + B2`。
  - `R1.2` 组合方式：打开详情面、审核面、处理页签或历史面时，都必须说明来自哪个对象列表、节点或前序上下文，并在返回里承接当前工作面身份。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`OPEN_REQUEST_SCOPE/WORKSET_BINDING_SCOPE`。
- `R2` 查看请求必须附着明确对象与定位上下文
  - `R2.1` 参与基：`B3 + B4`。
  - `R2.2` 组合方式：任一查看请求都要说明当前对象、当前查看类型和定位上下文；查看返回也要明确承接内容、锚点和恢复信息，不能只回无法落位的内容片段。
  - `R2.3` 输出能力：`C3 + C4`。
  - `R2.4` 边界绑定：`INSPECTION_REQUEST_SCOPE/INSPECTION_RESPONSE_SCOPE`。

## 5. 验证（Verification）
- `V1` 打开请求有效性：任一工作面打开都必须能说明对象来源、当前对象与工作面类型。
- `V2` 工作面承接有效性：任一打开返回都必须能说明当前工作面身份与对象归属。
- `V3` 查看契约有效性：任一查看请求与返回都必须能说明对象、定位上下文与恢复信息。
- `V4` 最小必要性：移除任一 `B*` 后，`V1~V3` 至少一项失效。
- `V5` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
