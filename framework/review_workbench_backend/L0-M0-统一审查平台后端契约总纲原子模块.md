# 统一审查平台后端契约总纲原子模块:ReviewWorkbenchBackendUnifiedPlatformServiceContractBases

@framework

本模块定义 `review_workbench_backend` 在后端契约侧所有契约共同依赖的最小公共骨架。它不回答“查询怎么写”“动作怎么提交”“历史差异怎么返回”，而是回答这些不同类型的后端契约交互最低限度必须如何表达请求上下文、目标对象、结果状态与回流影响，才能保持跨场景一致、可组合、可验证。

本模块位于 `review_workbench_backend` 的后端契约原子层，只保留统一审查平台在后端契约层最稳定、最不应重复定义的公共语义，不直接规定 HTTP 方法、路由、Header、鉴权方式、SDK 形态或存储模型。这些属于 `Implementation Config` 与代码实现层；本模块负责的是领域层“后端契约最少要说明什么”。

本模块也不替代查询契约、工作面打开与查看契约、对象动作契约、范围结构改动契约和历史差异契约。那些模块回答的是具体后端契约语义；本模块回答的是这些具体契约在进入 `review_workbench` 之后共同遵守的统一表达骨架。

## 1. 能力声明（Capability Statement）
- `C1` 统一请求上下文表达能力：把当前场景、进入来源、当前范围与调用上下文整理成稳定请求骨架。
- `C2` 统一目标对象表达能力：把单对象、对象集合、范围节点、工作面对象或历史节点整理成稳定目标骨架。
- `C3` 统一结果状态表达能力：把成功、失败、部分成功、无结果与拒绝处理等结果整理成稳定返回骨架。
- `C4` 统一回流影响表达能力：把对象变化、范围变化、摘要变化、刷新要求与失效提示整理成稳定影响骨架。
- `N1` 非职责声明：不直接定义查询参数、工作面载荷、查看内容结构、动作载荷、历史差异内容或协议实现细节。

## 2. 边界定义（Boundary / 参数）
- `REQUEST_CONTEXT_SCOPE` 请求上下文边界：任一后端契约请求都允许因场景实例不同而携带不同上下文，但必须至少能说明“当前属于哪个场景”“从哪个入口或上一步进入”“当前绑定在哪个范围或工作上下文中”，不允许脱离平台场景上下文孤立发起。
- `TARGET_SCOPE` 目标对象边界：任一后端契约请求允许面向单对象、对象集合、范围节点、工作面对象或历史节点，但必须明确目标类型、稳定标识与必要从属关系，不允许只有动作名或请求名而没有目标语义。
- `RESULT_STATUS_SCOPE` 结果状态边界：任一后端契约返回允许是成功、失败、部分成功、空结果或拒绝处理，但必须明确当前结果状态与最小解释信息，不允许只返回无法落地理解的匿名布尔结果。
- `EFFECTS_SCOPE` 回流影响边界：任一后端契约返回允许带对象变化、范围变化、摘要变化、刷新要求或失效提示中的不同组合，但只要结果会影响平台当前工作上下文，就必须至少回流一组可解释影响，不允许变化只停留在后端契约内部。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 请求上下文基：回答“任一后端契约请求最少要带哪些场景、入口、范围或工作上下文信息”。来源：`C1 + REQUEST_CONTEXT_SCOPE`。
- `B2` 目标对象基：回答“任一后端契约请求最少要说明什么目标对象、目标集合或目标节点”。来源：`C2 + TARGET_SCOPE`。
- `B3` 结果状态基：回答“任一后端契约返回最少要说明什么结果状态与解释语义”。来源：`C3 + RESULT_STATUS_SCOPE`。
- `B4` 回流影响基：回答“任一后端契约返回最少要如何说明对象变化、范围变化、摘要变化、刷新要求或失效提示”。来源：`C4 + EFFECTS_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 任一后端契约请求都必须附着统一请求上下文
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：无论是查询、打开工作面、查看、动作、结构改动还是历史请求，都必须能够说明自己属于 `review_workbench` 中的哪个场景、从哪个入口或前序工作上下文进入，以及当前附着在哪个范围、对象上下文或工作面上下文中；不能把后端契约请求组织成脱离平台语义的匿名调用。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`REQUEST_CONTEXT_SCOPE`。
- `R2` 任一后端契约请求都必须附着明确目标对象
  - `R2.1` 参与基：`B2`。
  - `R2.2` 组合方式：任一后端契约请求都必须说明其目标是单对象、对象集合、范围节点、工作面对象还是历史节点，并且目标必须具备稳定标识与必要从属关系；不能只有“刷新”“提交”“查看”“打开”这类动作意图，而没有明确目标语义。
  - `R2.3` 输出能力：`C2`。
  - `R2.4` 边界绑定：`TARGET_SCOPE`。
- `R3` 任一后端契约返回都必须表达明确结果状态
  - `R3.1` 参与基：`B3`。
  - `R3.2` 组合方式：任一后端契约返回都必须说明当前结果是成功、失败、部分成功、空结果还是拒绝处理，并带有最小解释语义；不能只返回一个调用完成信号，而不说明平台当前工作上下文应如何理解该结果。
  - `R3.3` 输出能力：`C3`。
  - `R3.4` 边界绑定：`RESULT_STATUS_SCOPE`。
- `R4` 任一会影响工作台状态的后端契约返回都必须回流影响信息
  - `R4.1` 参与基：`B3 + B4`。
  - `R4.2` 组合方式：只要后端契约结果会影响对象状态、范围结构、局部摘要、平台反馈、查询结果或工作面内容，就必须至少回流一组可解释影响；不能让前端只能依赖猜测或全量重载来恢复状态。
  - `R4.3` 输出能力：`C3 + C4`。
  - `R4.4` 边界绑定：`RESULT_STATUS_SCOPE/EFFECTS_SCOPE`。
- `R5` 总纲只能定义共性，不能吞并具体后端契约语义
  - `R5.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R5.2` 组合方式：本模块只允许定义跨查询、工作面打开、查看、动作、结构改动和历史请求共同成立的公共骨架；一旦某类能力开始需要说明自己的特有请求字段、特有返回结构或特有校验关系，就必须下沉到独立后端契约原子模块，而不能继续堆叠在本模块中。
  - `R5.3` 输出能力：`N1`。
  - `R5.4` 边界绑定：`REQUEST_CONTEXT_SCOPE/TARGET_SCOPE/RESULT_STATUS_SCOPE/EFFECTS_SCOPE`。

## 5. 验证（Verification）
- `V1` 请求上下文有效性：任一后端契约请求都必须能说明当前场景、进入来源与当前工作上下文。
- `V2` 目标对象有效性：任一后端契约请求都必须能说明目标对象、目标集合或目标节点的稳定语义。
- `V3` 结果状态有效性：任一后端契约返回都必须能说明成功、失败、部分成功、空结果或拒绝处理中的一种稳定状态。
- `V4` 回流影响有效性：任一会改变当前工作台状态的后端契约返回都必须至少回流一组可解释影响。
- `V5` 总纲边界有效性：本模块只验证统一后端契约骨架是否成立；若出现查询、动作、结构改动或历史差异的专有语义，应由对应后端契约原子模块独立验证。
- `V6` 最小必要性：移除任一 `B*` 后，`V1~V5` 至少一项失效。
- `V7` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
