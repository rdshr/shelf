# 范围结构变更与查询失效后端契约原子模块:ReviewWorkbenchBackendStructureMutationAndQueryInvalidationServiceContracts

@framework

本模块定义 `review_workbench_backend` 中与“范围结构改动及其查询失效回流”直接相关的最小稳定后端契约。它回答“上传、新建目录、删除、移动与重命名请求最少要带什么，以及树更新、范围状态回退、局部信息变化和查询失效怎样稳定回流”。

本模块默认建立在 `review_workbench_backend.L0.M0` 之上，延续统一后端契约骨架来承接结构改动请求、改动返回与查询失效回流。

## 1. 能力声明（Capability Statement）
- `C1` 结构改动请求整理能力。
- `C2` 树更新与范围状态回流能力。
- `C3` 查询失效与重算回流能力。
- `N1` 非职责声明：不负责对象打开、审核动作和历史差异。

## 2. 边界定义（Boundary / 参数）
- `STRUCTURE_REQUEST_SCOPE` 改动请求边界：改动请求必须说明目标对象、当前范围、父节点或目标位置与最小载荷。
- `STRUCTURE_RESPONSE_SCOPE` 改动返回边界：改动返回必须说明树更新、位置变化或当前范围状态回退中的必要组合。
- `QUERY_INVALIDATION_SCOPE` 查询失效边界：改动后必须能说明哪些查询摘要、结果命中或局部统计需要刷新、失效或重算。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 改动请求基：来源：`C1 + STRUCTURE_REQUEST_SCOPE`。
- `B2` 改动返回基：来源：`C2 + STRUCTURE_RESPONSE_SCOPE`。
- `B3` 查询失效基：来源：`C3 + QUERY_INVALIDATION_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 任一结构改动都必须附着当前范围与目标位置
  - `R1.1` 参与基：`L0.M0[B1,B2] + B1`。
  - `R1.2` 组合方式：上传、新建目录、删除、移动与重命名都必须能说明当前范围、目标对象、父节点或目标位置，不能变成脱离当前范围的匿名改动。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`STRUCTURE_REQUEST_SCOPE`。
- `R2` 结构改动成功后必须回流树更新或范围状态变化
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：只要改动成立，就必须回流树更新、位置变化或当前范围状态回退中的必要组合，不能只返回一个成功标记。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`STRUCTURE_REQUEST_SCOPE/STRUCTURE_RESPONSE_SCOPE`。
- `R3` 结构改动后必须说明查询如何失效或重算
  - `R3.1` 参与基：`B2 + B3`。
  - `R3.2` 组合方式：结构改动一旦影响当前查询上下文，就必须稳定说明哪些查询摘要、结果命中或局部统计需要刷新、失效或重算。
  - `R3.3` 输出能力：`C2 + C3`。
  - `R3.4` 边界绑定：`STRUCTURE_RESPONSE_SCOPE/QUERY_INVALIDATION_SCOPE`。

## 5. 验证（Verification）
- `V1` 改动请求有效性：任一结构改动都必须能说明当前范围、目标对象和目标位置。
- `V2` 改动返回有效性：任一结构改动返回都必须能说明树更新、位置变化或范围状态回退中的必要组合。
- `V3` 查询失效有效性：任一影响查询上下文的结构改动都必须能说明哪些查询需要刷新、失效或重算。
- `V4` 最小必要性：移除任一 `B*` 后，`V1~V3` 至少一项失效。
- `V5` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
