# 历史序列与差异后端契约原子模块:ReviewWorkbenchBackendHistorySequenceAndDiffServiceContracts

@framework

本模块定义 `review_workbench_backend` 中与“历史序列、历史节点与差异返回”直接相关的最小稳定后端契约。它回答“历史请求最少要带什么对象 / 节点 / 版本关系，以及历史序列、节点详情、差异内容和元信息怎样稳定返回”。

本模块默认建立在 `review_workbench_backend.L0.M0` 之上，延续统一后端契约骨架来承接历史请求、历史返回与差异返回。

## 1. 能力声明（Capability Statement）
- `C1` 历史请求整理能力。
- `C2` 历史序列表达能力。
- `C3` 历史节点与版本关系表达能力。
- `C4` 差异内容与元信息表达能力。
- `N1` 非职责声明：不负责动作提交和结构改动。

## 2. 边界定义（Boundary / 参数）
- `HISTORY_REQUEST_SCOPE` 历史请求边界：任一历史请求都必须说明当前对象、历史节点或前后版本关系。
- `HISTORY_SEQUENCE_SCOPE` 历史序列边界：历史返回必须能说明序列、当前节点或节点详情中的必要组合。
- `VERSION_RELATION_SCOPE` 版本关系边界：差异查看必须能说明明确前后版本关系。
- `DIFF_SCOPE` 差异边界：历史返回必须能说明差异内容与至少一组可解释元信息。

## 3. 最小可行基（Minimum Viable Bases）
- `B1` 历史请求基：来源：`C1 + HISTORY_REQUEST_SCOPE`。
- `B2` 序列返回基：来源：`C2 + HISTORY_SEQUENCE_SCOPE`。
- `B3` 版本关系基：来源：`C3 + VERSION_RELATION_SCOPE`。
- `B4` 差异内容基：来源：`C4 + DIFF_SCOPE`。

## 4. 基组合原则（Base Combination Principles）
- `R1` 历史请求必须附着明确对象与节点
  - `R1.1` 参与基：`L0.M0[B1,B2] + B1`。
  - `R1.2` 组合方式：任一历史请求都必须说明当前对象、当前节点或前后版本关系，不能只发起匿名历史调用。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`HISTORY_REQUEST_SCOPE`。
- `R2` 历史返回必须稳定承接序列、节点与差异
  - `R2.1` 参与基：`B2 + B3 + B4`。
  - `R2.2` 组合方式：历史返回必须能说明历史序列、节点详情、前后版本关系、差异内容和至少一组可解释元信息中的必要组合，不能只回无法追溯来源的历史文本。
  - `R2.3` 输出能力：`C2 + C3 + C4`。
  - `R2.4` 边界绑定：`HISTORY_SEQUENCE_SCOPE/VERSION_RELATION_SCOPE/DIFF_SCOPE`。

## 5. 验证（Verification）
- `V1` 历史请求有效性：任一历史请求都必须能说明对象、节点或版本关系。
- `V2` 历史返回有效性：任一历史返回都必须能说明序列、节点和差异内容中的必要组合。
- `V3` 差异可解释性：任一差异内容都必须能说明明确版本关系并至少带一组元信息。
- `V4` 最小必要性：移除任一 `B*` 后，`V1~V3` 至少一项失效。
- `V5` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
