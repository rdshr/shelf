# 多渠道个人助手框架标准模块:MultiChannelPersonalAssistantFramework

@framework

本模块定义消息总线驱动的多渠道个人助手共同结构语言。它只负责渠道回路、代理工作回路和后台延续回路如何共同闭合；具体产品人格、启用渠道、提供商选择、权限实例值与工作区文案应落到 `Product Spec` 与 `Implementation Config`，不得反向写进本模块。

## 1. 能力声明（Capability Statement）

- `C1` 多渠道回路能力：把不同承载面的消息出入统一收敛到稳定主回路。
- `C2` 回合工作能力：把即时代理工作统一收敛到稳定回合结构。
- `C3` 后台延续能力：把脱离当前回合的后台延续统一收敛到稳定侧回路。
- `C4` 非能力项：不负责具体渠道名单、提供商名单、产品人格真相与领域知识内容。

## 2. 边界定义（Boundary / 参数）

- `MAINLOOP` 主回路边界：前台入站、前台出站和后台回投都必须共享同一主回路语义。来源：`C1 + C3`。
- `CHANNELDIFF` 渠道差异边界：渠道差异只能留在渠道回路层，不得改写当前回合回路和后台延续回路。来源：`C1 + C2`。
- `SESSIONCLOSE` 会话闭合边界：任一响应或后台结果都必须回到唯一会话主轴。来源：`C1 + C2 + C3`。
- `CONTEXTLOCAL` 上下文局部边界：任一工作链只能读取其装载时声明过的账页和册页。来源：`C2 + C3`。
- `ASYNCRETURN` 延续回投边界：任一后台侧回路都必须通过显式回投链并回主回路。来源：`C3`。
- `TRACECHAIN` 追踪边界：任一消息和任务路径都必须能从来源回放到终点。来源：`C1 + C2 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 渠道消息回路基：L1.M0[R3]。来源：`C1 + MAINLOOP + CHANNELDIFF + SESSIONCLOSE + TRACECHAIN`。
  说明：前台多渠道收发消息这一整套外层回路，用这一个基承接。
- `B2` 当前回合回路基：L1.M1[R3]。来源：`C2 + SESSIONCLOSE + CONTEXTLOCAL + TRACECHAIN`。
  说明：一轮代理工作从装载到结果落回记录，这一整套回路用这个基承接。
- `B3` 后台延续回路基：L1.M2[R3]。来源：`C3 + ASYNCRETURN + SESSIONCLOSE + TRACECHAIN`。
  说明：那些不在当前回合里立刻做完、而是在后台继续并最终回来的回路，用这个基承接。

## 4. 基组合原则（Base Combination Principles）

- `R1` 即时主回路闭合
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：先固定渠道消息回路，再把当前回合回路挂到同一主回路上，形成即时代理工作闭环。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`MAINLOOP/CHANNELDIFF/SESSIONCLOSE/CONTEXTLOCAL/TRACECHAIN`。
- `R2` 侧回路闭合
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：把后台延续回路挂到当前回合回路的侧边，使后台工作能脱离即时回合又能重新并回主回路。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`SESSIONCLOSE/CONTEXTLOCAL/ASYNCRETURN/TRACECHAIN`。
- `R3` 全框架闭合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：将渠道消息回路、当前回合回路和后台延续回路共同绑定到同一主回路与同一会话闭合规则上，形成完整多渠道个人助手框架。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`MAINLOOP/CHANNELDIFF/SESSIONCLOSE/CONTEXTLOCAL/ASYNCRETURN/TRACECHAIN`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：渠道差异侵入当前回合回路、回合结果脱离唯一会话主轴或后台侧回路无法并回主回路的组合无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`MAINLOOP/CHANNELDIFF/SESSIONCLOSE/CONTEXTLOCAL/ASYNCRETURN/TRACECHAIN`。

## 5. 验证（Verification）

- `V1` 主回路有效性：渠道消息回路必须满足 `MAINLOOP/CHANNELDIFF` 约束。
- `V2` 回合有效性：当前回合回路必须满足 `SESSIONCLOSE/CONTEXTLOCAL` 约束。
- `V3` 延续有效性：后台延续回路必须满足 `ASYNCRETURN/TRACECHAIN` 约束。
- `V4` 全局闭合性：主回路、当前回合回路和后台侧回路必须共同闭合到同一会话主轴和同一追踪链。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
