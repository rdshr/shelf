# 接入插口与执行插槽原子模块:PortAndExecutionSlotAtoms

@framework

本模块定义多渠道助手框架中的接入插口、工作腔位和触发窗格结构。它只负责回答“外部从哪里接进来、内部在哪些腔位里工作、延续从哪个窗格被唤醒”，不负责具体厂商、协议库和实例值。

## 1. 能力声明（Capability Statement）

- `C1` 接入插口抽象能力：定义外部进入和离开系统时的稳定插口结构。
- `C2` 工作腔位抽象能力：定义系统内部承载工作链时的稳定腔位结构。
- `C3` 触发窗格抽象能力：定义计划、唤醒和恢复落点时的稳定窗格结构。
- `C4` 非能力项：不负责具体渠道名单、提供商名单、权限实例值和桥接实现细节。

## 2. 边界定义（Boundary / 参数）

- `BIND` 绑定位边界：每个插口、腔位或窗格都必须落在一个明确绑定位上，不得悬空存在。来源：`C1 + C2 + C3`。
- `GATE` 门控边界：每个插口和腔位都必须有显式允许条件或拒绝条件，不得裸露开放。来源：`C1 + C2`。
- `HANDOFF` 交接边界：相邻插口、腔位与窗格之间交接的载荷外形必须一致，不得半途变形。来源：`C1 + C2 + C3`。
- `TRIGGER` 触发边界：每个触发窗格只能对应一种声明过的唤醒来源或计划形式。来源：`C3`。
- `RESUME` 恢复边界：每条工作链离开当前腔位后都必须有显式停止位、重试位或恢复位。来源：`C2 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 渠道插口基：由入站口、出站口和绑定位面构成。来源：`C1 + BIND + GATE + HANDOFF`。
  说明：系统跟外部对接时，先要分清楚从哪进、从哪出、固定挂在哪。
- `B2` 工作槽位基：由推理腔、动作腔和延续腔构成。来源：`C2 + BIND + GATE + HANDOFF`。
  说明：系统内部干活的位置要先分出来，分别承接不同工作段。
- `B3` 触发窗口基：由计划窗、唤醒窗和恢复窗构成。来源：`C3 + TRIGGER + RESUME + BIND`。
  说明：后台相关的时间动作，要落在一组明确窗口里。

## 4. 基组合原则（Base Combination Principles）

- `R1` 插口成立
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先固定入站口、出站口和绑定位面，形成可受门控的接入插口。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`BIND/GATE/HANDOFF`。
- `R2` 腔位成立
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：在已绑定的接入插口之后挂接推理腔、动作腔和延续腔，形成稳定工作面。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`BIND/GATE/HANDOFF`。
- `R3` 窗格闭合
  - `R3.1` 参与基：`B2 + B3`。
  - `R3.2` 组合方式：将计划窗、唤醒窗和恢复窗挂到延续腔之后，形成可唤醒、可暂停、可恢复的延续面。
  - `R3.3` 输出能力：`C2 + C3`。
  - `R3.4` 边界绑定：`BIND/HANDOFF/TRIGGER/RESUME`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：绑定位悬空、门控缺失、交接外形断裂或触发后无恢复窗的组合无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`BIND/GATE/HANDOFF/TRIGGER/RESUME`。

## 5. 验证（Verification）

- `V1` 插口有效性：通道插口必须满足 `BIND/GATE/HANDOFF` 约束。
- `V2` 腔位有效性：工作腔位必须满足 `BIND/GATE/HANDOFF` 约束。
- `V3` 窗格有效性：触发窗格必须满足 `TRIGGER/RESUME` 约束。
- `V4` 交接闭合性：插口、腔位和窗格之间必须能以同形载荷完成稳定交接。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
