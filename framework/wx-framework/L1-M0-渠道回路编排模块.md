# 渠道回路编排模块:ChannelLoopOrchestration

@framework

本模块把消息部件、会话主轴和出入口位置件接成一条渠道线。它只负责 4 件事：收消息段怎么接、发消息段怎么接、消息挂到哪条会话线上、不同渠道怎么隔开；不负责回复内容本身。

## 1. 能力声明（Capability Statement）

- `C1` 收消息能力：把外部消息稳定接进系统。
- `C2` 发消息能力：把系统结果稳定送回目标承载面。
- `C3` 渠道隔离能力：把不同渠道和不同会话的活动线隔开。
- `C4` 非能力项：不负责推理生成逻辑、会话记忆策略和后台计划值。

## 2. 边界定义（Boundary / 参数）

- `INGRESS` 收入口边界：同一外部消息只能经一个声明过的入站口进入系统。来源：`C1 + C3`。
- `EGRESS` 发出口边界：同一出站结果只能落到一个声明过的目标口，不得一结果多投。来源：`C2 + C3`。
- `CARRIER` 承载边界：消息进入渠道线后不得丢失主标识、会话挂点和目标挂点。来源：`C1 + C2`。
- `ISOLATE` 隔离边界：不同渠道和不同会话不得共用同一活动线。来源：`C1 + C3`。
- `BRIDGE` 侧链接边界：桥接段只能挂在渠道线侧边，不得冒充主线。来源：`C2 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 渠道收消息段基：L0.M0[R2] + L0.M2[R1]。来源：`C1 + INGRESS + CARRIER`。
  说明：`L0.M0[R2]` 给出已挂线消息，`L0.M2[R1]` 给出入站口，两者接上后才有一段真正能收消息的渠道段。
- `B2` 渠道发消息段基：L0.M0[R3] + L0.M2[R2]。来源：`C2 + EGRESS + CARRIER + BRIDGE`。
  说明：`L0.M0[R3]` 给出可继续接出的消息链，`L0.M2[R2]` 给出出站口，两者接上后才有一段真正能发消息的渠道段。
- `B3` 渠道会话挂点基：L0.M0[R2] + L0.M1[R1]。来源：`C1 + C2 + ISOLATE`。
  说明：`L0.M0[R2]` 让消息能挂线，`L0.M1[R1]` 给出会话主轴，两者接上后消息才知道自己属于哪条会话线。
- `B4` 渠道分区基：L0.M1[R1] + L0.M2[R1] + L0.M2[R2]。来源：`C3 + ISOLATE + INGRESS + EGRESS`。
  说明：独立会话主轴配上一组独立出入口，才叫一个独立渠道分区。

## 4. 基组合原则（Base Combination Principles）

- `R1` 收消息段成立
  - `R1.1` 参与基：`B1 + B3`。
  - `R1.2` 组合方式：先把收消息段挂到唯一会话主轴上，形成单渠道的收消息入口线。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`INGRESS/CARRIER/ISOLATE`。
- `R2` 发消息段成立
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：把发消息段挂到同一会话主轴上，形成单渠道的发消息出口线。
  - `R2.3` 输出能力：`C2`。
  - `R2.4` 边界绑定：`EGRESS/CARRIER/BRIDGE/ISOLATE`。
- `R3` 单渠道线成立
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：把收消息段、发消息段和渠道会话挂点接成一条从收消息到发消息的完整渠道线。
  - `R3.3` 输出能力：`C1 + C2`。
  - `R3.4` 边界绑定：`INGRESS/EGRESS/CARRIER/BRIDGE/ISOLATE`。
- `R4` 多渠道并行成立
  - `R4.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R4.2` 组合方式：把每条完整渠道线分别装进独立渠道分区里，使多渠道能并行存在而不串线。
  - `R4.3` 输出能力：`C1 + C2 + C3`。
  - `R4.4` 边界绑定：`INGRESS/EGRESS/CARRIER/ISOLATE/BRIDGE`。
- `R5` 禁止组合
  - `R5.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R5.2` 组合方式：收入口重复、发出口歧义、消息挂错会话线或不同渠道共用同一活动线的组合无效。
  - `R5.3` 输出能力：`C4`。
  - `R5.4` 边界绑定：`INGRESS/EGRESS/CARRIER/ISOLATE/BRIDGE`。

## 5. 验证（Verification）

- `V1` 收消息有效性：收消息段必须满足 `INGRESS/CARRIER` 约束。
- `V2` 发消息有效性：发消息段必须满足 `EGRESS/CARRIER/BRIDGE` 约束。
- `V3` 挂线有效性：渠道会话挂点必须满足 `ISOLATE` 约束。
- `V4` 分区有效性：渠道分区必须能把不同渠道和不同会话隔开。
- `V5` 单线有效性：同一渠道消息必须能从收消息段顺序追到发消息段。
- `V6` 最小必要性：移除任一 `B*` 后，`V2~V5` 至少一项失败。
- `V7` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
