# 后台延续编排模块:BackgroundContinuationOrchestration

@framework

本模块把触发窗格、会话记载结构和工作腔位结构组合成后台延续回路。它只负责脱离当前即时回合后的唤醒、展开和回投，不负责前台渠道接入与回复外形。

## 1. 能力声明（Capability Statement）

- `C1` 唤醒触发能力：把后台延续所需的触发条件收敛为稳定唤醒结构。
- `C2` 脱离工作能力：把后台工作单元收敛为稳定脱离结构。
- `C3` 结果回投能力：把后台结果重新并回主回路和主会话。
- `C4` 非能力项：不负责前台消息承接、渠道隔离规则与具体产品提醒文案。

## 2. 边界定义（Boundary / 参数）

- `PLAN` 计划边界：每个后台延续单元只能对应一个声明过的唤醒条件集合。来源：`C1 + C2`。
- `DETACH` 脱离边界：每个后台延续单元都必须指回一个来源会话主轴或来源回合。来源：`C1 + C2`。
- `RUNNER` 工作边界：后台工作链必须有显式起点、终点和中间腔位，不得悬空展开。来源：`C2`。
- `REJOIN` 回投边界：每个后台结果都必须并回一个明确主回路和一个明确会话主轴。来源：`C3`。
- `RESUME` 恢复边界：暂停、失败、重试和恢复落点必须显式可见。来源：`C2 + C3`。
- `TRACE` 追踪边界：唤醒原因、工作历程与回投结果必须可回放。来源：`C1 + C2 + C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 后台唤醒点基：L0.M1[R1] + L0.M2[R3]。来源：`C1 + PLAN + DETACH + TRACE`。
  说明：后台延续从哪里起步、因为什么被叫醒，先落在这个点上。
- `B2` 后台工作链基：L0.M1[R3] + L0.M2[R2,R3]。来源：`C2 + DETACH + RUNNER + RESUME + TRACE`。
  说明：任务脱离当前回合后，在后台继续往下走，就沿着这条链展开。
- `B3` 后台回投链基：L0.M0[R3] + L0.M1[R3] + L0.M2[R3]。来源：`C3 + REJOIN + RESUME + TRACE`。
  说明：后台做出来的结果，最后沿这条链回到原来的消息回路和会话里。

## 4. 基组合原则（Base Combination Principles）

- `R1` 唤醒成立
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先把来源主轴和触发窗格收成唤醒窗，再允许后继后台工作链附着其上。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`PLAN/DETACH/TRACE`。
- `R2` 脱离闭合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：在已建立的唤醒窗后挂接后台工作链，使工作单元脱离即时回合而独立延续。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`PLAN/DETACH/RUNNER/RESUME/TRACE`。
- `R3` 回投闭合
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：把后台结果沿结果回投链重新挂回主消息回路和主会话主轴，形成完整后台延续闭环。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`PLAN/DETACH/RUNNER/REJOIN/RESUME/TRACE`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：后台延续单元脱离来源主轴、工作链无显式起止、恢复落点缺失或结果无法并回主回路的组合无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`PLAN/DETACH/RUNNER/REJOIN/RESUME/TRACE`。

## 5. 验证（Verification）

- `V1` 唤醒有效性：唤醒窗必须满足 `PLAN/DETACH/TRACE` 约束。
- `V2` 工作链有效性：后台工作链必须满足 `RUNNER/RESUME/TRACE` 约束。
- `V3` 回投有效性：结果回投链必须满足 `REJOIN/RESUME/TRACE` 约束。
- `V4` 回路闭合性：同一后台延续单元的唤醒、工作和回投必须可以顺序回放到同一主会话。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
