# 认证服务与会话契约原子模块:AuthServiceSessionContracts

@framework

本模块定义 auth 子域里偏后端承接面的最小结构语言：认证请求如何形成稳定契约、会话结果如何被稳定判定、身份主体如何被统一表达、认证失败如何被结构化返回。它不定义登录页长什么样，也不定义具体产品要求使用用户名密码、验证码、SSO 还是某个身份提供商；这些分别属于更上层的场景语义与产品真相。本模块只负责把“后端认证服务应稳定提供什么叶子结构”沉成可复用的原子，供 auth 领域更高层编排复用。

## 1. 能力声明（Capability Statement）

- `C1` 认证请求契约能力：把登录建立、会话探测和登出结束组织成稳定的请求契约。
- `C2` 会话状态表达能力：把已登录、未登录、已失效和已结束这些状态组织成稳定的会话语义。
- `C3` 身份主体承载能力：把主体标识、主体类别和主体作用域组织成稳定的身份上下文。
- `C4` 失败结果结构化能力：把认证失败、会话失效、权限不足和服务拒绝组织成稳定错误结构。
- `C5` 非能力项：不负责具体协议、Header/Cookie/Token 名称、SDK、加密算法、存储介质或第三方身份服务选型。

## 2. 边界定义（Boundary / 参数）

- `AUTHREQUEST` 认证请求边界：登录建立、会话探测、登出结束和刷新重验这些请求类型要稳定。来源：`C1 + C2`。
- `SESSIONSTATE` 会话状态边界：已登录、未登录、已失效、已结束和待重验这些状态值域要稳定。来源：`C1 + C2`。
- `IDENTITYSUBJECT` 身份主体边界：主体标识、主体类别、主体角色集合和作用域集合的表达方式要稳定。来源：`C2 + C3`。
- `AUTHERROR` 认证错误边界：失败类别、失败原因、是否可重试和是否需要重新认证的表达方式要稳定。来源：`C1 + C4`。
- `AUTHSERVICE` 服务承接边界：认证服务必须能被上层以统一契约调用，而不是依赖某个具体协议细节。来源：`C1 + C3 + C4`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 认证请求结构基：由请求类型、输入载荷和调用上下文构成的请求契约结构。来源：`C1 + AUTHREQUEST + AUTHSERVICE`。
- `B2` 会话结果结构基：由会话标识、当前状态、有效性判定和结束语义构成的会话结果结构。来源：`C1 + C2 + SESSIONSTATE + AUTHSERVICE`。
- `B3` 主体上下文结构基：由主体标识、主体类别、角色集合和作用域集合构成的主体上下文结构。来源：`C2 + C3 + IDENTITYSUBJECT`。
- `B4` 认证错误结果结构基：由失败类别、失败原因、可重试标记和重新认证标记构成的错误结果结构。来源：`C1 + C4 + AUTHERROR + AUTHSERVICE`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 认证请求必须回到稳定会话结果
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：将认证请求结构与会话结果结构串联后，每个认证请求都必须落到可判定的会话结果；因此登录建立、会话探测、刷新重验和登出结束都不得只返回模糊成功，而必须同时给出稳定的会话状态。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`AUTHREQUEST/SESSIONSTATE/AUTHSERVICE`。
- `R2` 已登录会话必须能承接身份主体
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：将会话结果结构与身份主体结构串联后，凡是可被判定为已登录或可继续放行的会话，都必须能导出稳定主体上下文；因此主体信息不得脱离会话结果孤立存在。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 边界绑定：`SESSIONSTATE/IDENTITYSUBJECT/AUTHSERVICE`。
- `R3` 认证失败必须可被结构化处理
  - `R3.1` 参与基：`B1 + B4`。
  - `R3.2` 组合方式：将认证请求结构与认证错误结果结构串联后，请求失败、会话失效和权限不足都必须导出可区分的失败结果；因此上层不得从自由文本里猜测失败类型。
  - `R3.3` 输出能力：`C1 + C4`。
  - `R3.4` 边界绑定：`AUTHREQUEST/AUTHERROR/AUTHSERVICE`。
- `R4` 完整认证服务闭环
  - `R4.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R4.2` 组合方式：将认证请求、会话结果、身份主体和失败结构统一到同一服务承接边界后，认证服务才能稳定支持“发起请求 -> 判定会话 -> 导出主体 -> 失败回传”闭环；缺失任一结构都会使上层编排退化为实现细节拼装。
  - `R4.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R4.4` 边界绑定：`AUTHREQUEST/SESSIONSTATE/IDENTITYSUBJECT/AUTHERROR/AUTHSERVICE`。
- `R5` 无效方案
  - `R5.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R5.2` 组合方式：以下方案无效：只有认证请求而没有会话结果结构，无法判断服务结果是否可继续放行；只有会话结果而没有身份主体结构，已登录结果无法承接稳定主体；只有失败文本而没有失败类别结构，上层无法稳定区分重新认证、重试或拒绝访问；只有主体结构而没有请求契约结构，主体上下文缺少可信来源。
  - `R5.3` 输出能力：`C5`。
  - `R5.4` 边界绑定：`AUTHREQUEST/SESSIONSTATE/IDENTITYSUBJECT/AUTHERROR/AUTHSERVICE`。

## 5. 验证（Verification）

- `V1` 认证请求契约有效性：通过条件：登录建立、会话探测、刷新重验和登出结束都能被 `AUTHREQUEST` 稳定表达。失败原因：请求类型缺失、请求载荷不可判定或调用上下文混乱。证据：请求契约映射、服务调用样例与状态结果说明。
- `V2` 会话状态有效性：通过条件：`SESSIONSTATE` 能稳定覆盖已登录、未登录、已失效、已结束和待重验。失败原因：关键状态缺失、状态切换不可解释或有效性判定缺失。证据：状态结构映射、状态图与运行样例。
- `V3` 身份主体有效性：通过条件：已登录会话能导出 `IDENTITYSUBJECT` 对应的主体标识、角色和作用域。失败原因：主体上下文缺失、主体来源不明或角色作用域表达不稳定。证据：主体结构映射、服务返回样例与权限判定样例。
- `V4` 失败结构有效性：通过条件：`AUTHERROR` 能稳定区分认证失败、会话失效、权限不足和服务拒绝。失败原因：失败类别混淆、失败结果只剩自由文本或无法判断是否需要重试/重新认证。证据：错误结构映射、失败样例与处理结论。
- `V5` 服务闭环必要性：通过条件：保留全部 `B*` 时，`V1~V4` 可同时成立。失败原因：移除任一 `B*` 后，认证请求、会话结果、身份主体或失败结构至少一项失效。证据：`B1~B4` 到 `R1~R5` 的组合映射与校验结论。
- `V6` 结论表达：通过条件：逐条输出 `R* -> C* / 边界` 的通过或失败结论。失败原因：规则结论无法追溯到参与基、能力或边界绑定。证据：规则级验证报告与映射结果。
