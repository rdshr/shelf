# 认证入口保护壳与回跳基础模块:AuthEntryProtectedShellReturnBase

@framework

本模块定义 auth 子域里最小的一层“怎么把认证入口、保护壳和回跳壳组织起来”。因为认证场景要用到的输入框、触发器、容器、提示和页面跳转这些基础结构，frontend 已经提供了，所以 auth 不再自己重复造一套原子，而是直接从这一层开始，说明“认证入口怎么组织、受保护目标怎么包裹、认证成功后怎么回原页面、失败后怎么给出前端可观察提示”。本模块不直接定义具体业务页面清单、认证服务契约或项目实例值，只负责把前端基础原子组织成认证流程的前端编排结构；若需要引入认证请求、会话结果和错误结果这些服务契约，则由更上层场景模块组合 `L0.M0` 与 `L0.M1`。

## 1. 能力声明（Capability Statement）

- `C1` 认证入口组织能力：把输入、授权触发、提交和提示组织成稳定的认证入口。
- `C2` 保护壳编排能力：把进入前拦截、运行中失效占位和退出后回退组织成稳定流程。
- `C3` 返回原页面能力：把“记录原目标、登录成功后回去、回不去时走默认页面”组织成稳定流程。
- `C4` 失败提示能力：把登录失败、会话失效和取消登录时的提示组织成稳定流程。
- `C5` 非能力项：不负责具体受保护页面清单、项目文案、接口路径与前后端技术选型。

## 2. 边界定义（Boundary / 参数）

- `AUTHENTRYSHELL` 认证入口壳边界：认证入口页面、授权触发、提交动作和辅助动作的顺序要稳定。来源：`C1 + C4`。
- `PROTECTEDTARGET` 受保护目标边界：哪些页面、路由、动作或资源进入前必须先被保护壳包裹，哪些目标允许匿名访问，以及判定失败后应进入哪条承接回路，都要保持稳定。来源：`C1 + C2 + C3`。
- `PROTECTEDSHELL` 保护壳边界：进入前拦截、失效占位和放行前壳体包裹的顺序要稳定。来源：`C2`。
- `RETURNSHELL` 回跳壳边界：记录原目标、回到原目标和退回默认页面的顺序要稳定。来源：`C2 + C3`。
- `FEEDBACKSHELL` 反馈壳边界：登录失败、会话失效和取消登录时的提示方式要稳定。来源：`C1 + C4`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 受保护目标先包裹
  - `R1.1` 参与基：`B2 + B3`。
  - `R1.2` 组合方式：将受保护目标保护壳结构与返回恢复结构串联后，页面、路由和动作在进入前必须先被保护壳包裹，并预留统一回退与恢复承接；因此所有受保护目标都不得绕过保护壳直接暴露到主流程。
  - `R1.3` 输出能力：`C2 + C3`。
  - `R1.4` 边界绑定：`PROTECTEDTARGET/PROTECTEDSHELL/RETURNSHELL`。
- `R2` 认证入口接入完整回路
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：将认证入口结构与返回恢复结构串联后，认证入口的提交、辅助动作和授权触发都必须接入稳定回退与恢复承接；因此认证入口不能只是静态表单或孤立触发器。
  - `R2.3` 输出能力：`C1 + C3 + C4`。
  - `R2.4` 边界绑定：`AUTHENTRYSHELL/RETURNSHELL/FEEDBACKSHELL`。
- `R3` 失效后重新进入入口
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：将受保护目标保护壳、认证入口结构和返回恢复结构串联后，当前目标一旦需要重新认证，流程必须沿既有保护壳回路切换到认证入口，并保持原目标恢复承接；因此失效提示不得脱离保护壳与回退结构单独漂浮。
  - `R3.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R3.4` 边界绑定：`AUTHENTRYSHELL/PROTECTEDTARGET/PROTECTEDSHELL/RETURNSHELL/FEEDBACKSHELL`。
- `R4` 无效方案
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：以下方案无效：只有认证入口结构但没有返回恢复结构，认证入口缺少稳定回退承接；只有保护壳结构但没有返回恢复结构，受保护目标失去统一回退出口；只有返回恢复结构但没有认证入口结构，重新进入流程缺少稳定入口；只有入口与返回结构但没有保护壳结构，受保护目标在进入前无法被统一包裹。
  - `R4.3` 输出能力：`C5`。
  - `R4.4` 边界绑定：`AUTHENTRYSHELL/PROTECTEDTARGET/PROTECTEDSHELL/RETURNSHELL/FEEDBACKSHELL`。

## 5. 验证（Verification）

- `V1` 认证入口有效性：通过条件：认证入口能稳定承接输入、授权触发、提交和失败提示。失败原因：缺少输入承接、触发出口、提交出口或失败提示任一结构。证据：`AUTHENTRYSHELL/FEEDBACKSHELL` 对应的结构映射、生成页面与交互样例。
- `V2` 保护壳有效性：通过条件：所有受保护页面、路由或动作进入前都能先落入稳定保护壳。失败原因：受保护目标未被承接、进入前缺少保护壳或保护壳与回退承接脱节。证据：`PROTECTEDTARGET/PROTECTEDSHELL` 对应的结构映射、页面进入样例与运行结果。
- `V3` 返回原页面有效性：通过条件：登录成功后能回到原目标，或退回默认页面。失败原因：原目标未记录、返回恢复链路缺失或默认回退不可解释。证据：`RETURNSHELL` 对应的结构映射、路由恢复样例与运行结果。
- `V4` 编排闭环有效性：通过条件：认证入口、保护壳和返回恢复结构能形成稳定的前端编排闭环。失败原因：入口、保护壳和回退三者之间至少有一处无法串联。证据：`AUTHENTRYSHELL/PROTECTEDTARGET/RETURNSHELL/FEEDBACKSHELL` 对应的结构映射与交互样例。
- `V5` 编排必要性：通过条件：保留全部 `B*` 时，`V1~V4` 可同时成立。失败原因：移除任一 `B*` 后，认证入口、保护壳、返回恢复或前端编排闭环至少一项失效。证据：`B1~B3` 到 `R1~R4` 的组合映射与校验结论。
- `V6` 结论表达：通过条件：逐条输出 `R* -> C* / 边界` 的通过或失败结论。失败原因：规则结论无法追溯到参与基、能力或边界绑定。证据：规则级验证报告与映射结果。
