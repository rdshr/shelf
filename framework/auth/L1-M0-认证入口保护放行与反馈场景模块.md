# 认证入口保护放行与反馈场景模块:AuthEntryGuardFeedbackScenarioModule

@framework

本模块是跨项目复用的认证与会话场景标准模块。它只定义“受保护页面怎么进入认证入口、认证成功后怎么回原页面、认证失败怎么提示、会话怎么形成完整流程”这一类通用结构语言。项目实例应把本模块编译成明确的 auth contract、session contract 和验证结论，而不是在具体项目代码或生成器里临时拼装这些规则。本模块在 `Framework -> Product Spec -> Implementation Config -> Code` 分层中只负责认证与会话的共同结构、边界、场景基和组合规则；具体采用用户名密码、邮箱密码还是验证码登录，属于 `Product Spec`；具体采用 header、cookie、token 还是第三方身份服务，属于 `Implementation Config`。认证入口与回跳壳编排沉在 `auth.L0.M0`，偏后端承接面的认证服务与会话契约沉在 `auth.L0.M1`，本模块则负责把这两类 L0 原子组合成可讨论、可约束的场景语义。

## 1. 能力声明（Capability Statement）

- `C1` 认证入口承接能力：支持在受保护页面进入前，稳定展示认证入口或认证请求触发点。
- `C2` 会话完整流程能力：支持会话建立、会话检查、会话结束这三类稳定动作。
- `C3` 返回原页面能力：支持登录成功后回到原目标页，或者退回默认页面。
- `C4` 失败提示能力：支持登录失败、会话失效或取消登录时给出稳定提示。
- `C5` 非能力项：不负责具体字段文案、UI 样式、接口路径、SDK 选型、存储介质和加密实现。

## 2. 边界定义（Boundary / 参数）

- `AUTHENTRYFLOW` 认证入口流边界：定义认证入口是否承载账号输入、授权触发、提交区和辅助动作区。来源：`C1 + C4`。合法入口变体包括：`username_password`、`email_password`、`phone_code`、`passwordless_code`、`sso_redirect`、`oauth_button`、`magic_link`。
- `SESSIONFLOW` 会话流边界：定义认证请求如何接入会话结果、会话结果如何驱动放行，以及失效后如何重新进入。来源：`C2 + C4`。
- `RETURNFLOW` 返回流边界：定义认证成功后回原页面、去默认页面和恢复原上下文的语义。来源：`C2 + C3`。
- `FEEDBACKFLOW` 反馈流边界：定义认证失败、会话失效、取消登录和重新认证时用户能看到的提示语义。来源：`C1 + C4`。
- `GUARDFLOW` 保护放行边界：定义哪些页面是受保护目标、什么时候必须先登录以及失败后如何切回入口。来源：`C1 + C2 + C3`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 受保护页面先登录
  - `R1.1` 参与基：`B2`。
  - `R1.2` 组合方式：将受保护目标场景、会话结果场景和返回语义场景绑定到同一承接结构后，受保护页面、路由和动作在进入前必须先被保护壳包裹，再由会话结果决定是否放行；当判定不通过时，当前流程不得继续放行，而必须切换到认证入口与回退语义所定义的承接回路。
  - `R1.3` 输出能力：`C2 + C3`。
  - `R1.4` 输出结构：`GUARDFLOW + RETURNFLOW + SESSIONFLOW`。
  - `R1.5` 边界绑定：`GUARDFLOW/SESSIONFLOW/RETURNFLOW`。
- `R2` 认证入口必须接上完整流程
  - `R2.1` 参与基：`B1`。
  - `R2.2` 组合方式：将认证入口场景绑定到已接入完整回路的入口承接结构后，认证入口的输入、触发和提交出口必须同时接上认证请求发起、失败结果承接和认证成功后的恢复链路；因此认证入口不能只是静态表单或孤立按钮。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 输出结构：`AUTHENTRYFLOW + SESSIONFLOW + RETURNFLOW`。
  - `R2.5` 边界绑定：`AUTHENTRYFLOW/SESSIONFLOW/RETURNFLOW`。
- `R3` 认证成功后回去
  - `R3.1` 参与基：`B2`。
  - `R3.2` 组合方式：将会话结果场景、受保护目标场景和返回语义场景绑定到同一承接结构后，认证成功后的出口必须优先恢复原目标；当原目标不可恢复时，流程必须切换到默认回退语义所定义的稳定页面。
  - `R3.3` 输出能力：`C3`。
  - `R3.4` 输出结构：`RETURNFLOW + GUARDFLOW + SESSIONFLOW`。
  - `R3.5` 边界绑定：`RETURNFLOW/GUARDFLOW/SESSIONFLOW`。
- `R4` 认证失败和会话失效都要提示
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：将认证入口接入结构、保护放行结构和反馈承接结构串联后，认证失败、会话失效和取消登录的状态切换必须沿既有入口、放行和回退回路导出用户可观察提示；因此提示内容必须和当前会话结果、主体状态与重新认证要求保持一致，而不能脱离既有结构单独定义。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 输出结构：`FEEDBACKFLOW + AUTHENTRYFLOW + SESSIONFLOW`。
  - `R4.5` 边界绑定：`FEEDBACKFLOW/AUTHENTRYFLOW/SESSIONFLOW`。
- `R5` 无效方案
  - `R5.1` 参与基：`B1 + B2 + B3`。
  - `R5.2` 组合方式：以下方案无效：缺少 `B1` 时，认证入口场景无法形成稳定的输入、触发和提交承接；缺少 `B2` 时，受保护目标判定、会话放行和返回恢复无法形成稳定回路；缺少 `B3` 时，认证失败、会话失效和重新认证要求无法稳定落到主体与反馈语义；只有入口与放行结构但缺少与之绑定的反馈承接结构时，认证失败、会话失效和取消登录会退化为不可观察状态切换。
  - `R5.3` 输出能力：`C5`。
  - `R5.4` 输出结构：`AUTHENTRYFLOW + SESSIONFLOW + RETURNFLOW + FEEDBACKFLOW + GUARDFLOW`。
  - `R5.5` 边界绑定：`AUTHENTRYFLOW/SESSIONFLOW/RETURNFLOW/FEEDBACKFLOW/GUARDFLOW`。

## 5. 验证（Verification）

- `V1` 保护放行语义验证：通过条件：所有需要登录后才能访问的页面或动作，都能落入 `GUARDFLOW` 的受保护目标解释，并在进入前被稳定包裹到保护壳与放行判定语义中。失败原因：受保护目标缺失、进入前包裹不清、放行条件不明或保护范围与返回语义冲突。证据：`GUARDFLOW` 对应的产品 section、规则映射与页面进入样例。
- `V2` 会话流语义验证：通过条件：`SESSIONFLOW` 能解释认证请求如何接入会话结果，以及已登录、未登录、已失效和结束后的状态切换。失败原因：关键状态缺失、状态切换不可判定或结束语义无法落点。证据：`SESSIONFLOW` 对应的产品 section、状态图和运行样例。
- `V3` 返回流语义验证：通过条件：`RETURNFLOW` 能解释“回原目标”和“退回默认页”两类结果，且切换条件清晰。失败原因：返回目标记录缺失、默认回退未定义或两类返回条件互相冲突。证据：`RETURNFLOW` 对应的产品 section、路由回路说明与运行样例。
- `V4` 反馈流语义验证：通过条件：`FEEDBACKFLOW` 覆盖认证失败、会话失效和取消登录三类用户可见提示，且提示语义与当前状态一致。失败原因：提示类别缺失、提示不可观察或提示内容与状态不一致。证据：`FEEDBACKFLOW` 对应的产品 section、提示样例与运行结果。
- `V5` 入口流语义验证：通过条件：`AUTHENTRYFLOW` 的入口变体属于 `username_password`、`email_password`、`phone_code`、`passwordless_code`、`sso_redirect`、`oauth_button`、`magic_link` 之一，且输入区、触发区、提交区和辅助动作区职责清晰。失败原因：入口变体超出合法值域、入口职责混乱或辅助动作不可解释。证据：`AUTHENTRYFLOW` 对应的产品 section、入口结构样例与字段映射。
- `V6` 场景最小必要性验证：通过条件：保留全部 `B*` 时，`AUTHENTRYFLOW/SESSIONFLOW/RETURNFLOW/FEEDBACKFLOW/GUARDFLOW` 可被完整解释。失败原因：移除任一 `B*` 后，至少一个场景边界失去承接结构，或反馈语义脱离既有入口、主体与放行回路。证据：`B1~B3` 到 `R1~R5` 的组合映射与校验结论。
