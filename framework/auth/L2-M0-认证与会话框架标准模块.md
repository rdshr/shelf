# 认证与会话框架标准模块:AuthenticationSessionFramework

@framework

本模块建立在“认证入口与保护壳基础 -> 认证服务与会话契约 -> 认证场景编排”这条压缩链上。登录入口、受保护目标保护壳、返回恢复壳与反馈壳位于 `auth/L0-M0`，认证请求、会话状态、已认证主体、认证错误与服务承接位于 `auth/L0-M1`，认证入口、会话闭环、返回恢复、反馈与保护检查场景位于 `auth/L1-M0`，本模块只定义 auth 子域的通用框架如何承接这些结构。项目实例应把本模块编译成显式 auth contract、session contract 和规则校验，而不是在实例配置、页面代码或接口实现里重写这些结构。实例配置采用“大类边界 -> 大类 section”约定：`ENTRYMODEL -> [auth.entry]`、`SESSIONMODEL -> [auth.session]`、`IDENTITYMODEL -> [auth.identity]`、`GUARDMODEL -> [auth.guard]`、`RETURNMODEL -> [auth.return]`、`FEEDBACKMODEL -> [auth.feedback]`、`SERVICEMODEL -> [auth.service]`。对应的 `product_spec.toml` 与 `implementation_config.toml` 应默认使用中文详细注释，以便认证入口、会话状态、当前主体、保护检查、返回恢复、反馈语义与服务承接在同一套仓库语言里被清楚讨论。在 `Framework -> Product Spec -> Implementation Config -> Code` 分层中，本模块只负责认证与会话的共同结构语言，不直接固化项目的登录文案、协议细节、身份提供商、Cookie/Header/Token 实现、SDK 选型或页面样式。

## 1. 能力声明（Capability Statement）

- `C1` 认证入口模型收敛能力：把认证入口变体、认证请求触发与入口承接收敛为稳定认证入口模型。
- `C2` 会话与主体模型收敛能力：把会话状态、已认证主体与重新认证要求收敛为稳定会话模型。
- `C3` 保护与返回编排能力：把受保护目标检查、登录成功返回恢复与运行中重新进入收敛为稳定认证编排模型。
- `C4` 反馈与服务映射能力：把认证失败、会话失效、重新认证反馈与服务承接收敛为稳定实例映射边界。
- `C5` 非能力项：不负责资源授权、角色权限、访问控制策略、具体协议参数、第三方身份服务实现、数据库结构、缓存方案、SDK 选型与项目文案。

## 2. 边界定义（Boundary / 参数）

- `ENTRYMODEL` 认证入口模型边界：认证入口变体、输入区、触发区、提交区与辅助动作区需稳定。来源：`C1 + C4`。
- `SESSIONMODEL` 会话模型边界：会话建立、检查、失效、结束、待重验与继续访问判定需稳定。来源：`C2 + C3 + C4`。
- `IDENTITYMODEL` 已认证主体模型边界：主体标识、主体类别、认证来源、认证强度与最小主体快照需稳定。来源：`C2 + C4`。
- `GUARDMODEL` 认证保护模型边界：页面进入、动作触发、运行中重新认证与切回入口承接方式需稳定。来源：`C3 + C4`。
- `RETURNMODEL` 返回恢复模型边界：恢复原目标、转到默认页面与恢复原上下文语义需稳定。来源：`C3 + C4`。
- `FEEDBACKMODEL` 认证反馈模型边界：入口内反馈、场景级反馈、会话失效反馈与重新认证反馈需稳定。来源：`C3 + C4`。
- `SERVICEMODEL` 认证服务模型边界：认证服务统一调用方式、认证请求承接、会话结果承接与错误结果承接需稳定。来源：`C1 + C2 + C4`。

### 2.1 下层边界归并表（Boundary Merge Table）

本表用于显式说明 auth 子域在 `L2` 如何把 `L0/L1` 的原子契约边界与场景流转边界收敛为统一实例开放边界；项目实例应优先面对 `ENTRYMODEL ~ SERVICEMODEL`，而不是直接暴露下层边界名。

| `L2` 大类边界 | 主要归并的下层边界 | 归并说明 |
| --- | --- | --- |
| `ENTRYMODEL` | `AUTHENTRYFLOW + AUTHREQUEST` | 统一承接“用户从哪里发起认证”与“认证请求怎样被稳定表达”，避免把入口变体拆散到入口流和请求契约两处分别定义。 |
| `SESSIONMODEL` | `SESSIONFLOW + SESSIONSTATE` | 统一承接“会话如何流转”与“会话状态如何表达”，避免一部分配置按流程语义定义会话，一部分配置按状态语义重复定义会话。 |
| `IDENTITYMODEL` | `IDENTITYSUBJECT` | 统一承接认证成功后的当前主体表达，使主体信息不再散落于会话结果字段、服务返回字段或上层场景特判。 |
| `GUARDMODEL` | `GUARDFLOW` | 统一承接受保护目标、检查时机与检查失败后的切回入口语义，使需要登录的访问入口只有一套稳定保护边界。 |
| `RETURNMODEL` | `RETURNFLOW` | 统一承接登录成功后恢复原目标、转到默认页面与恢复原上下文的语义，使返回恢复不再依赖零散路由规则或页面特判。 |
| `FEEDBACKMODEL` | `FEEDBACKFLOW + AUTHERROR` | 统一承接入口内反馈、场景级反馈与认证错误表达，避免同一失败真相同时散落在场景提示和服务错误字段中。 |
| `SERVICEMODEL` | `AUTHSERVICE` | 统一承接 auth 子域对上层暴露的服务调用方式与契约承接面，使服务实现细节不反向污染入口、会话或反馈边界。 |

### 2.2 最小实例 section 示例（Minimum Instance Section Example）

以下示例只用于说明 `Product Spec` 在实例层应如何围绕 `ENTRYMODEL ~ SERVICEMODEL` 组织 auth 子域真相；示例中的值不是框架默认值，也不限制具体项目必须采用相同登录方式、页面路径或文案键。

```toml
# [auth.entry]
# 讨论产品允许哪些认证入口变体，以及入口默认怎么向用户暴露。
[auth.entry]
variants = ["email_password", "sso_redirect"]
default_entry = "email_password"

# [auth.session]
# 讨论产品层承认哪些会话状态，以及哪些状态会要求重新认证。
[auth.session]
states = ["authenticated", "anonymous", "expired"]
expired_requires_reauth = true

# [auth.identity]
# 讨论认证成功后，产品最少需要稳定表达哪些当前主体字段。
[auth.identity]
fields = ["subject_id", "subject_type", "auth_source"]

# [auth.guard]
# 讨论哪些页面或动作必须先登录才能继续进入。
[auth.guard]
protected_routes = ["/workspace"]
protected_actions = ["project.create"]

# [auth.return]
# 讨论登录成功后优先恢复原目标，恢复失败时回到哪个默认页面。
[auth.return]
restore_original_target = true
default_route = "/workspace"

# [auth.feedback]
# 讨论入口内反馈与场景级反馈怎样区分，以及会话失效时用户看到什么反馈。
[auth.feedback]
entry_feedback_mode = "inline"
session_expired_mode = "scenario_feedback"
reauth_message_key = "auth.relogin_required"

# [auth.service]
# 讨论 auth 子域需要哪些服务承接能力；协议、Header/Cookie/Token 细节应继续下沉到 Implementation Config。
[auth.service]
supports_session_probe = true
supports_reauth = true
```

该示例的意图是让实例层只面对一套 auth 子域语言：认证入口在 `[auth.entry]` 讨论，会话与主体分别在 `[auth.session]`、`[auth.identity]` 讨论，受保护目标与返回恢复分别在 `[auth.guard]`、`[auth.return]` 讨论，反馈与服务承接分别在 `[auth.feedback]`、`[auth.service]` 讨论。项目实例若仍倾向于把同一真相拆回 `AUTHREQUEST / SESSIONFLOW / AUTHERROR` 等下层名词，说明 auth 子域的 `L2` 收敛尚未真正被实例层接纳。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 认证入口与服务接入基：L1.M0[R2,R4] + L0.M1[R1,R3]。来源：`C1 + C4 + ENTRYMODEL + FEEDBACKMODEL + SERVICEMODEL`。
- `B2` 会话与主体统一基：L0.M1[R1,R2,R4]。来源：`C2 + SESSIONMODEL + IDENTITYMODEL + SERVICEMODEL`。
- `B3` 保护检查与返回恢复基：L1.M0[R1,R3]。来源：`C3 + GUARDMODEL + RETURNMODEL + SESSIONMODEL`。
- `B4` 项目实例开放基：L1.M0[R1,R2,R3,R4] + L0.M1[R1,R2,R3,R4]。来源：`C1 + C2 + C3 + C4 + ENTRYMODEL + SESSIONMODEL + IDENTITYMODEL + GUARDMODEL + RETURNMODEL + FEEDBACKMODEL + SERVICEMODEL`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 认证入口必须先映射到统一入口模型
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先固定认证入口变体、入口触发与认证请求承接之间的主关系，再允许具体项目选择用户名密码、验证码、SSO 或其他合法入口变体；因此项目实例不得把入口变体散落到场景字段、服务字段或代码常量中分别定义。
  - `R1.3` 输出能力：`C1 + C4`。
  - `R1.4` 边界绑定：`ENTRYMODEL/FEEDBACKMODEL/SERVICEMODEL`。
- `R2` 会话结果必须收敛为统一会话与主体模型
  - `R2.1` 参与基：`B2`。
  - `R2.2` 组合方式：将会话状态、已认证主体和认证错误结果绑定到同一会话语义后，项目实例中的“已登录/未登录/已失效/已结束/待重验”与“当前主体是谁”都必须通过统一会话模型表达；因此项目不得一部分按场景流转字段定义会话，一部分按服务契约字段再重复定义主体或错误。
  - `R2.3` 输出能力：`C2 + C4`。
  - `R2.4` 边界绑定：`SESSIONMODEL/IDENTITYMODEL/FEEDBACKMODEL/SERVICEMODEL`。
- `R3` 受保护访问必须经由统一保护与返回模型
  - `R3.1` 参与基：`B2 + B3`。
  - `R3.2` 组合方式：将受保护目标检查、会话结果判定和返回恢复语义绑定到同一认证回路后，所有需要登录才能继续的页面、路由和动作都必须先进入统一保护模型，再由会话结果决定是继续访问、回切入口还是恢复原目标；因此项目代码不得绕过统一 guard 直接暴露受保护目标。
  - `R3.3` 输出能力：`C2 + C3`。
  - `R3.4` 边界绑定：`SESSIONMODEL/GUARDMODEL/RETURNMODEL`。
- `R4` 项目实例必须按 auth 大类边界映射
  - `R4.1` 参与基：`B4`。
  - `R4.2` 组合方式：项目实例必须把认证入口、会话、主体、保护、返回、反馈和服务承接分别映射到稳定 section，而不是让一半配置按流程语义组织、一半配置按契约语义组织，或让同一产品真相散落在多个 section 与代码常量中重复表达。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`ENTRYMODEL/SESSIONMODEL/IDENTITYMODEL/GUARDMODEL/RETURNMODEL/FEEDBACKMODEL/SERVICEMODEL`。
- `R5` 禁止组合
  - `R5.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R5.2` 组合方式：把入口变体同时写进入口流、请求契约与服务实现三处，把“会话失效需要重新认证”拆散到状态、反馈和代码特判，把保护目标与返回恢复分开定义到互不关联字段，或把“未登录”和“已登录但无权限”混成同一结果的方案均无效。
  - `R5.3` 输出能力：`C5`。
  - `R5.4` 边界绑定：`ENTRYMODEL/SESSIONMODEL/IDENTITYMODEL/GUARDMODEL/RETURNMODEL/FEEDBACKMODEL/SERVICEMODEL`。

## 5. 验证（Verification）

- `V1` 入口模型稳定性：`[auth.entry]` 中的字段必须只表达认证入口变体、入口承接与入口辅助动作，不得混入服务协议或授权语义。
- `V2` 会话与主体稳定性：`[auth.session]` 与 `[auth.identity]` 必须能稳定解释会话状态、重新认证要求和当前主体表达，不得相互缺位或重复定义。
- `V3` 保护与返回一致性：`[auth.guard]` 与 `[auth.return]` 必须能共同解释受保护目标、检查失败切回入口与登录成功后的恢复目标，不得各自独立漂移。
- `V4` 反馈与服务一致性：`[auth.feedback]` 与 `[auth.service]` 必须能共同解释入口内反馈、场景级反馈、会话失效与重新认证结果，不得让同一失败真相分别落在互相无关的字段中。
- `V5` 实例映射稳定性：`[auth.entry]/[auth.session]/[auth.identity]/[auth.guard]/[auth.return]/[auth.feedback]/[auth.service]` 中的字段必须只落到对应大类边界，不得跨大类游离。
- `V6` 场景与契约收敛性：认证入口、会话状态、主体表达、反馈与服务承接必须能被同一 auth 子域语言解释，不得同时暴露为两套互不归并的实例 section 体系。
- `V7` 最小必要性：移除任一 `B*` 后，`V2~V6` 至少一项失败。
- `V8` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
