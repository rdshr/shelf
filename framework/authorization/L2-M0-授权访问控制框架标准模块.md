# 授权访问控制框架标准模块:AuthorizationAccessControlFramework

@framework

本模块建立在“授权原子契约 -> 授权场景编排 -> 授权领域标准”这条压缩链上。访问主体、资源动作、策略输入、授权结果与授权错误位于 `authorization/L0-M0`，授权保护检查、拒绝反馈与认证回切位于 `authorization/L1-M0`，本模块只定义 authorization 子域的通用框架如何承接这些结构。项目实例应把本模块编译成显式 authorization contract 和规则校验，而不是在实例配置、页面代码或接口实现里重写这些结构。实例配置采用“大类边界 -> 大类 section”约定：`SUBJECTMODEL -> [authorization.subject]`、`RESOURCEMODEL -> [authorization.resource]`、`ACTIONMODEL -> [authorization.action]`、`POLICYMODEL -> [authorization.policy]`、`GUARDMODEL -> [authorization.guard]`、`DENIEDMODEL -> [authorization.denied]`、`AUTHRETURNMODEL -> [authorization.auth_return]`。对应的 `product_spec.toml` 与 `implementation_config.toml` 应默认使用中文详细注释，以便主体、资源、动作、策略、拒绝反馈与认证回切在同一套仓库语言里被清楚讨论。在 `Framework -> Product Spec -> Implementation Config -> Code` 分层中，本模块只负责授权访问控制的共同结构语言，不直接固化项目的角色实例值、权限清单、规则引擎、数据库实现或页面文案。

## 1. 能力声明（Capability Statement）

- `C1` 授权模型收敛能力：把访问主体、资源、动作与策略输入收敛为稳定授权判定模型。
- `C2` 授权保护编排能力：把页面守卫、动作守卫、条件允许和拒绝分流收敛为稳定访问控制流程。
- `C3` 认证衔接能力：把“需要重新认证/需要更高认证强度”的授权结果稳定衔接到 auth 回路。
- `C4` 项目实例映射能力：为 `Product Spec` 与 `Implementation Config` 输出稳定 section 边界和实例化入口。
- `C5` 非能力项：不负责登录协议、具体角色名单、策略引擎实现、数据库结构、缓存方案、SDK 选型与项目文案。

## 2. 边界定义（Boundary / 参数）

- `SUBJECTMODEL` 主体模型边界：访问主体标识、主体类别、作用域和最小授权上下文需稳定。来源：`C1 + C4`。
- `RESOURCEMODEL` 资源模型边界：受保护资源的标识、类别、归属与最小资源上下文需稳定。来源：`C1 + C4`。
- `ACTIONMODEL` 动作模型边界：可授权动作、动作强度和动作上下文需稳定。来源：`C1 + C2`。
- `POLICYMODEL` 策略模型边界：角色、权限项、作用域、归属关系、条件规则与替代策略需稳定。来源：`C1 + C4`。
- `GUARDMODEL` 保护模型边界：页面进入、动作触发、运行中复检和条件允许承接方式需稳定。来源：`C2 + C4`。
- `DENIEDMODEL` 拒绝模型边界：整页拒绝、局部拒绝、动作拒绝和资源不可见反馈需稳定。来源：`C2 + C4`。
- `AUTHRETURNMODEL` 认证回切模型边界：需要重新认证、需要更高认证强度和回切后目标恢复语义需稳定。来源：`C2 + C3 + C4`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 授权判定模型基：L0.M0[R1,R2]。来源：`C1 + SUBJECTMODEL + RESOURCEMODEL + ACTIONMODEL + POLICYMODEL`。
- `B2` 授权保护编排基：L1.M0[R1,R4]。来源：`C1 + C2 + GUARDMODEL + DENIEDMODEL`。
- `B3` 拒绝反馈与认证回切基：L1.M0[R2,R3]。来源：`C2 + C3 + DENIEDMODEL + AUTHRETURNMODEL`。
- `B4` 项目实例映射基：L1.M0[R1,R2,R3,R4]。来源：`C1 + C2 + C3 + C4 + SUBJECTMODEL + RESOURCEMODEL + ACTIONMODEL + POLICYMODEL + GUARDMODEL + DENIEDMODEL + AUTHRETURNMODEL`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 授权模型先行
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先固定访问主体、资源、动作与策略输入之间的主关系，再允许更上层场景编排和项目实例值附着其上；因此任何项目权限规则都不得脱离主体、资源、动作和策略输入四元组孤立存在。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`SUBJECTMODEL/RESOURCEMODEL/ACTIONMODEL/POLICYMODEL`。
- `R2` 授权保护闭合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：在统一授权模型上接入页面守卫、动作守卫、条件允许和拒绝分流，约束受保护访问必须先判权，再进入页面、动作或局部区域编排；因此项目代码不得绕过授权保护直接暴露受限目标。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`SUBJECTMODEL/RESOURCEMODEL/ACTIONMODEL/POLICYMODEL/GUARDMODEL/DENIEDMODEL`。
- `R3` 回切 auth 闭合
  - `R3.1` 参与基：`B2 + B3`。
  - `R3.2` 组合方式：将授权结果分流、拒绝反馈和认证回切语义绑定到同一访问控制回路后，凡是需要重新认证或提升认证强度的场景，都必须稳定切回 auth 并保留原目标承接；因此这类结果不得被降格为普通权限拒绝。
  - `R3.3` 输出能力：`C2 + C3`。
  - `R3.4` 边界绑定：`GUARDMODEL/DENIEDMODEL/AUTHRETURNMODEL`。
- `R4` 项目实例必须按大类边界映射
  - `R4.1` 参与基：`B4`。
  - `R4.2` 组合方式：项目实例必须把主体模型、资源模型、动作模型、策略模型、保护模型、拒绝模型和认证回切模型分别映射到稳定 section，而不是把角色、页面、按钮、反馈和回切条件散落到无结构字段或实现代码中。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`SUBJECTMODEL/RESOURCEMODEL/ACTIONMODEL/POLICYMODEL/GUARDMODEL/DENIEDMODEL/AUTHRETURNMODEL`。
- `R5` 禁止组合
  - `R5.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R5.2` 组合方式：将“未登录”和“已登录但无权限”混成同一种 guard 结果、把条件允许伪装成完全放行、把需要重新认证伪装成普通 deny、或让项目直接在代码里散写页面/按钮判权而不落到稳定 section 的方案均无效。
  - `R5.3` 输出能力：`C5`。
  - `R5.4` 边界绑定：`SUBJECTMODEL/RESOURCEMODEL/ACTIONMODEL/POLICYMODEL/GUARDMODEL/DENIEDMODEL/AUTHRETURNMODEL`。

## 5. 验证（Verification）

- `V1` 授权模型完整性：访问主体、资源、动作和策略输入必须同时存在且能形成稳定判权输入。
- `V2` 保护编排一致性：页面守卫、动作守卫、条件允许和拒绝分流必须共享统一授权结果语义。
- `V3` 回切有效性：需要重新认证和需要更高认证强度的结果必须能稳定回切 auth 并保留目标恢复承接。
- `V4` 实例映射稳定性：`[authorization.subject]/[authorization.resource]/[authorization.action]/[authorization.policy]/[authorization.guard]/[authorization.denied]/[authorization.auth_return]` 中的字段必须只落到对应大类边界，不得跨大类游离。
- `V5` 场景边界一致性：条件允许、整页拒绝、动作拒绝和认证回切必须能在同一访问控制回路里被解释，不得互相混写。
- `V6` 最小必要性：移除任一 `B*` 后，`V2~V5` 至少一项失败。
- `V7` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
