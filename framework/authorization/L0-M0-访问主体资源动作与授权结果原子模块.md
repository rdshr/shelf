# 访问主体资源动作与授权结果原子模块:AuthorizationSubjectResourceActionResultContracts

@framework

本模块定义 authorization 子域里偏后端承接面的最小结构语言：访问主体如何被稳定承接、资源与动作如何被统一表达、授权判定如何形成稳定结果、访问拒绝如何被结构化返回。它不定义登录页长什么样，也不定义具体产品最终采用 RBAC、ABAC、ACL、租户隔离还是 ownership 判定作为授权策略主体；这些分别属于更上层的场景语义、产品真相与实现细化。本模块只负责把“授权服务应稳定提供什么叶子结构”沉成可复用原子，供 authorization 领域更高层编排复用。需要特别说明的是：本模块以前置 auth 已完成“认证是否成立、会话是否有效、当前主体是谁”的闭环为前提；authorization 只消费已认证主体与会话状态，不在此重复定义登录建立、会话探测或重新认证请求契约。若判定过程中出现“需要重新认证”或“认证强度不足”，本模块只输出结构化授权结果，由更上层场景决定是否回切到 auth 流程。本模块也不直接固化页面清单、接口路径、角色命名、策略存储介质与规则编排引擎。


## 1. 能力声明（Capability Statement）

- `C1` 访问主体承接能力：把已认证主体、主体范围和主体上下文组织成稳定的访问主体表达。
- `C2` 资源动作契约能力：把资源标识、资源类别、操作动作和访问上下文组织成稳定的授权输入契约。
- `C3` 授权结果表达能力：把允许、拒绝、条件允许和需提升校验这些结果组织成稳定授权语义。
- `C4` 拒绝与缺失结构化能力：把权限不足、策略缺失、作用域不匹配和上下文不足组织成稳定错误结构。
- `C5` 非能力项：不负责登录建立、会话探测、认证入口编排、页面样式、角色实例值、具体策略引擎、存储介质与项目专属访问规则文案。

## 2. 边界定义（Boundary / 参数）

<!--   说明：这一段回答“授权判定时，系统到底把谁当作当前访问主体”。它承接的是 auth 已经输出的主体身份快照，再补上进入授权判定所需的最小作用域信息，例如租户、组织、项目或主体归属上下文。这里的重点不是重新认证，而是让授权层稳定知道‘谁在请求访问’。 -->
- `ACCESSSUBJECT` 访问主体边界：主体标识、主体类别、认证强度、主体作用域与主体上下文的表达方式要稳定。它以前置 auth 的 `IDENTITYSUBJECT` 为输入来源，但不在本模块内重新定义认证链路。来源：`C1 + C3`。
<!--   说明：这一段回答“当前请求想碰的到底是什么对象”。资源可以是页面、路由、接口、文档、项目、租户内对象或其他业务实体；这里需要稳定表达资源的类别、标识与归属范围，但不在此直接固化具体产品资源清单。 -->
- `RESOURCE` 资源边界：资源标识、资源类别、资源归属和资源上下文的表达方式要稳定。来源：`C2 + C4`。
<!--   说明：这一段回答“主体想对资源做什么动作”。动作至少应能表达读、写、删除、发布、审批、管理这类访问语义，或其项目化等价动作。这里定义的是动作表达边界，不直接定义项目中的动作全集。 -->
- `ACTION` 动作边界：动作标识、动作类别、动作强度和动作上下文的表达方式要稳定。来源：`C2 + C3`。
<!--   说明：这一段回答“授权判定依赖哪些规则来源”。它承接角色、权限项、租户范围、归属关系、环境限制与附加条件等规则输入，让上层能解释为什么允许或拒绝，而不是把判定退化成不可追溯的黑盒布尔值。 -->
- `POLICYINPUT` 策略输入边界：角色、权限项、作用域、归属关系、环境条件和附加限制的表达方式要稳定。来源：`C1 + C2 + C4`。
<!--   说明：这一段回答“授权判定之后系统到底给出什么结论”。它至少应区分允许、拒绝、条件允许、需要重新认证和需要更高认证强度这几类结果，使上层能区分纯权限拒绝与回切 auth 的场景。 -->
- `AUTHZRESULT` 授权结果边界：结论类别、判定理由、可继续放行标记和是否需要回切认证的表达方式要稳定。来源：`C2 + C3 + C4`。
<!--   说明：这一段回答“授权没法稳定判定时，要怎么把失败表达出来”。这里覆盖的是权限不足、策略缺失、规则冲突、上下文不完整等授权链路内问题，而不是 auth 模块里的登录失败或会话失效。 -->
- `AUTHZERROR` 授权错误边界：错误类别、错误原因、是否可重试和是否需要补充上下文的表达方式要稳定。来源：`C3 + C4`。
<!--   说明：这一段回答“授权服务如何被上层稳定调用”。它回答的是输入与输出的契约承接，不回答具体规则引擎、数据库或缓存如何实现。 -->
- `AUTHZSERVICE` 授权服务边界：授权服务必须能被上层以统一契约调用，而不是依赖某个项目专属角色命名或策略引擎细节。来源：`C1 + C2 + C3 + C4`。

## 3. 最小可行基（Minimum Viable Bases）

<!--   说明：`B1` 回答“当前访问请求中的主体是谁，且其授权作用域是什么”。它承接 auth 已认证主体快照，再补充授权判定必须使用的最小作用域字段，例如 tenant_id、org_id、project_scope 或 ownership_context。没有这一基，上层就只能把主体字段散落在 token、session 或业务对象里临时拼装。 -->
- `B1` 访问主体结构基：由主体标识、主体类别、认证强度、主体作用域和主体上下文构成的访问主体结构。来源：`C1 + ACCESSSUBJECT + AUTHZSERVICE`。
<!--   说明：`B2` 回答“当前请求要对哪个资源做什么动作”。它要求系统把资源与动作一起组织成统一授权输入，而不是只传一个含糊页面名、接口名或按钮名。这样上层才能稳定讨论授权判定是在保护页面进入、接口调用还是业务动作执行。 -->
- `B2` 资源动作结构基：由资源标识、资源类别、动作标识、动作类别和访问上下文构成的资源动作结构。来源：`C2 + RESOURCE + ACTION + AUTHZSERVICE`。
<!--   说明：`B3` 回答“授权判断依赖了哪些最小规则输入”。它承接角色、权限项、作用域和归属关系，让结论至少可解释为‘因为主体具备某权限’、‘因为资源属于当前租户’或‘因为该主体拥有该对象’。这里定义的是最小输入结构，不提前绑定某一种具体模型。 -->
- `B3` 策略输入结构基：由角色、权限项、作用域、归属关系和附加条件构成的策略输入结构。来源：`C1 + C2 + POLICYINPUT + AUTHZSERVICE`。
<!--   说明：`B4` 回答“授权判定完成后，系统到底输出什么结论”。它要求结果不能只剩一个 `true/false`，而要说明是允许、拒绝、条件允许、需要重新认证还是需要更高认证强度，并携带最小理由与回切标记。 -->
- `B4` 授权结果结构基：由结论类别、判定理由、放行标记和回切认证标记构成的授权结果结构。来源：`C2 + C3 + AUTHZRESULT + AUTHZSERVICE`。
<!--   说明：`B5` 回答“如果授权链路本身缺资料或规则不成立，要怎么稳定失败”。它要求失败不能退化成自由文本，而要区分权限不足、策略缺失、作用域不匹配、规则冲突和上下文不足等失败类型。 -->
- `B5` 授权错误结构基：由错误类别、错误原因、可重试标记和补充上下文标记构成的授权错误结构。来源：`C3 + C4 + AUTHZERROR + AUTHZSERVICE`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 授权请求必须同时绑定主体与资源动作
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：将访问主体结构与资源动作结构串联后，每次授权判定都必须同时回答“谁”对“什么资源”执行“什么动作”；因此授权服务不得只基于页面名、角色名或接口名单独做模糊判断。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 边界绑定：`ACCESSSUBJECT/RESOURCE/ACTION/AUTHZSERVICE`。
- `R2` 授权判定必须可追溯到策略输入
  - `R2.1` 参与基：`B1 + B2 + B3`。
  - `R2.2` 组合方式：将访问主体结构、资源动作结构与策略输入结构串联后，允许与拒绝都必须能回溯到至少一类稳定规则输入；因此上层不得把授权判定退化成不可解释的硬编码布尔值。
  - `R2.3` 输出能力：`C1 + C2 + C3`。
  - `R2.4` 边界绑定：`ACCESSSUBJECT/RESOURCE/ACTION/POLICYINPUT/AUTHZSERVICE`。
- `R3` 授权结果必须区分放行、拒绝与回切认证
  - `R3.1` 参与基：`B2 + B4`。
  - `R3.2` 组合方式：将资源动作结构与授权结果结构串联后，授权服务必须能区分纯权限拒绝、条件允许、需要重新认证和需要更高认证强度这些结果；因此上层不得把“未登录”“已登录但无权限”“需提升认证强度”混写成同一种失败。
  - `R3.3` 输出能力：`C2 + C3`。
  - `R3.4` 边界绑定：`RESOURCE/ACTION/AUTHZRESULT/AUTHZSERVICE`。
- `R4` 授权失败必须可被结构化处理
  - `R4.1` 参与基：`B2 + B5`。
  - `R4.2` 组合方式：将资源动作结构与授权错误结构串联后，权限不足、策略缺失、作用域不匹配、规则冲突和上下文不足都必须导出可区分失败结果；因此上层不得从自由文本猜测当前到底是缺权限还是缺上下文。
  - `R4.3` 输出能力：`C2 + C4`。
  - `R4.4` 边界绑定：`RESOURCE/ACTION/AUTHZERROR/AUTHZSERVICE`。
- `R5` 完整授权服务闭环
  - `R5.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R5.2` 组合方式：将访问主体、资源动作、策略输入、授权结果和授权错误统一到同一服务承接边界后，授权服务才能稳定支持“承接主体 -> 对齐资源动作 -> 引入策略输入 -> 产出放行或拒绝 -> 失败回传”闭环；缺失任一结构都会使上层编排退化为项目代码里的零散特判。
  - `R5.3` 输出能力：`C1 + C2 + C3 + C4`。
  - `R5.4` 边界绑定：`ACCESSSUBJECT/RESOURCE/ACTION/POLICYINPUT/AUTHZRESULT/AUTHZERROR/AUTHZSERVICE`。
- `R6` 无效方案
  - `R6.1` 参与基：`B1 + B2 + B3 + B4 + B5`。
  - `R6.2` 组合方式：以下方案无效：只有访问主体而没有资源动作结构，无法判断主体到底想访问什么；只有资源动作而没有策略输入结构，允许与拒绝缺少可追溯依据；只有布尔结果而没有结论类别结构，上层无法稳定区分拒绝、条件允许与回切认证；只有错误文本而没有错误类别结构，上层无法稳定区分权限不足、策略缺失和上下文不足；把 auth 的会话失效、登录失败直接混入本模块错误结构，也属于职责越界。
  - `R6.3` 输出能力：`C5`。
  - `R6.4` 边界绑定：`ACCESSSUBJECT/RESOURCE/ACTION/POLICYINPUT/AUTHZRESULT/AUTHZERROR/AUTHZSERVICE`。

## 5. 验证（Verification）

- `V1` 访问主体有效性：通过条件：`ACCESSSUBJECT` 能稳定承接主体标识、主体类别、认证强度与最小授权作用域。失败原因：主体来源不明、作用域缺失或上下文表达不稳定。证据：主体结构映射、上游 auth 主体样例与授权输入样例。
- `V2` 资源动作契约有效性：通过条件：`RESOURCE` 与 `ACTION` 能稳定覆盖页面、接口、对象或业务动作的最小授权输入。失败原因：资源类别混淆、动作表达不清或访问上下文缺失。证据：资源动作映射、服务调用样例与判定输入说明。
- `V3` 策略输入有效性：通过条件：`POLICYINPUT` 能稳定表达角色、权限项、作用域、归属关系和附加条件。失败原因：规则来源不明、作用域无法解释或归属关系缺失。证据：策略输入映射、判定依据样例与规则解释说明。
- `V4` 授权结果有效性：通过条件：`AUTHZRESULT` 能稳定区分允许、拒绝、条件允许、需要重新认证和需要更高认证强度。失败原因：结论类别混淆、结果只剩布尔值或无法判断是否需要回切 auth。证据：结果结构映射、判定样例与处理结论。
- `V5` 授权错误有效性：通过条件：`AUTHZERROR` 能稳定区分权限不足、策略缺失、规则冲突、作用域不匹配和上下文不足。失败原因：错误类别混淆、失败结果只剩自由文本或无法判断是否可补全上下文。证据：错误结构映射、失败样例与处理结论。
- `V6` 服务闭环必要性：通过条件：保留全部 `B*` 时，`V1~V5` 可同时成立。失败原因：移除任一 `B*` 后，访问主体、资源动作、策略输入、授权结果或授权错误至少一项失效。证据：`B1~B5` 到 `R1~R6` 的组合映射与校验结论。
- `V7` 结论表达：通过条件：逐条输出 `R* -> C* / 边界` 的通过或失败结论。失败原因：规则结论无法追溯到参与基、能力或边界绑定。证据：规则级验证报告与映射结果。
