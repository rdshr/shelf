# 授权保护检查与拒绝反馈场景模块:AuthorizationGuardCheckDeniedFeedbackScenarioModule

@framework

本模块是跨项目复用的授权与访问控制场景标准模块。它只定义一类通用结构语言：需要权限才能进入的页面怎么先拦住，已登录主体在动作触发前如何先过授权检查，授权拒绝后前端怎么反馈，遇到“需要重新认证”或“需要更高认证强度”时如何回切到 auth 流程，整条访问控制流程怎么闭环。项目实例应把本模块编译成明确的 authorization contract、guard contract 和验证结论，而不是在具体项目代码或生成器里临时拼装这些规则。本模块在 `Framework -> Product Spec -> Implementation Config -> Code` 分层中只负责授权场景的共同结构、边界、场景基和组合规则；具体哪些角色拥有哪些权限、哪些资源受哪些限制，属于 `Product Spec`；具体使用 RBAC、ABAC、ACL、ownership 还是混合策略，以及规则引擎如何实现，属于 `Implementation Config`。访问主体、资源动作、策略输入、授权结果与授权错误沉在 `authorization.L0.M0`，本模块则负责把这些 L0 原子与现有 auth 场景的回切关系组合成可讨论、可约束的访问控制场景语义。本模块聚焦“已进入授权判定后如何决定能否访问”，不负责重新定义登录建立、会话探测、登录入口组织或认证失败提示；这些仍由 auth 子域承接。


## 1. 能力声明（Capability Statement）

- `C1` 授权保护承接能力：支持在页面进入、动作触发和运行中切换前稳定发起授权检查。
- `C2` 访问结果分流能力：支持把放行、拒绝、条件允许和回切认证分流到稳定后续流程。
- `C3` 拒绝反馈能力：支持权限不足、资源不可见、作用域不匹配和规则缺失时给出稳定反馈。
- `C4` 认证回切衔接能力：支持在需要重新认证或提升认证强度时稳定切回 auth 流程，并保留原访问目标。
- `C5` 非能力项：不负责登录入口字段、具体角色清单、策略引擎实现、接口路径、UI 样式和项目专属文案。

## 2. 边界定义（Boundary / 参数）

<!--   说明：这一段回答“哪些访问目标需要先过授权检查”。这里至少要覆盖三类：页面或路由进入前的授权保护、动作触发前的授权保护，以及运行中因上下文变化而需要重新判权的场景。这样授权不会只覆盖首屏进入，而漏掉按钮、菜单和运行中切换。 -->
- `AUTHZGUARDFLOW` 授权保护边界：定义哪些页面、路由、动作或资源进入前必须先经过授权检查，以及检查发生在何时。它至少应能覆盖 `page_route`、`action_trigger` 和 `runtime_recheck` 三类授权保护场景。来源：`C1 + C2`。
<!--   说明：这一段回答“授权通过后系统怎么继续走”。除了完全放行，也要覆盖条件允许，例如允许看列表但禁用敏感动作，或者允许进入页面但隐藏部分区域。 -->
- `ALLOWFLOW` 放行边界：定义授权通过、条件允许和限制放行时，页面、动作和局部区域如何继续进入后续流程。来源：`C2`。
<!--   说明：这一段回答“授权不通过时用户会看到什么”。这里最好把拒绝分成两类：一类是目标级拒绝，例如整页 403、无权限页、跳到默认页；另一类是动作级拒绝，例如按钮禁用、弹窗提示或局部占位。 -->
- `DENIEDFLOW` 拒绝反馈边界：定义授权拒绝、作用域不匹配、资源不可见和策略缺失时用户能看到什么反馈。反馈至少应区分 `target_denied` 与 `action_denied` 两类。来源：`C3`。
<!--   说明：这一段回答“如果授权层发现需要重新认证或提升认证强度，怎么回到 auth”。这里的关键不是重新发明 auth，而是稳定承接‘回切到 auth，再尽量恢复原目标’这条路。 -->
- `AUTHRETURNFLOW` 认证回切边界：定义需要重新认证、需要更高认证强度或需要重新建立授权上下文时，如何切回 auth 并保留原访问目标。来源：`C2 + C4`。
<!--   说明：这一段回答“授权检查时至少要拿到哪些输入”。它将访问主体、资源动作和策略输入三者绑定到同一场景里，避免上层只凭页面名或角色名做模糊判断。 -->
- `CHECKFLOW` 判权输入边界：定义访问主体、资源动作和策略输入如何形成统一授权检查输入。来源：`C1 + C2`。

## 3. 最小可行基（Minimum Viable Bases）

<!--   说明：`B1` 复用 `authorization.L0.M0[R1,R2]`，承接“授权检查前必须先把主体、资源动作和策略输入对齐”这一类场景结构；它关注的是判权输入的稳定组合，而不是放行或拒绝后的前端反馈。 -->
- `B1` 判权输入接入结构基：authorization.L0.M0[R1,R2]。来源：`C1 + C2 + CHECKFLOW`。
<!--   说明：`B2` 复用 `authorization.L0.M0[R3,R5] + frontend.L3.M0[R1,R2]`，承接“授权结果如何把页面或动作分流到放行、条件允许或拒绝承接”这一类场景结构；它借助前端页面与运行编排能力，让授权结果真正接入路由、动作与页面状态。 -->
- `B2` 结果分流结构基：authorization.L0.M0[R3,R5] + frontend.L3.M0[R1,R2]。来源：`C1 + C2 + ALLOWFLOW + AUTHZGUARDFLOW`。
<!--   说明：`B3` 复用 `authorization.L0.M0[R4,R5] + frontend.L1.M4[R1,R2] + frontend.L3.M0[R2]`，承接“授权拒绝后如何形成用户可观察反馈”这一类场景结构。它主要负责 target_denied 和 action_denied 两类反馈，而不是 auth 里的登录失败反馈。 -->
- `B3` 拒绝反馈承接结构基：authorization.L0.M0[R4,R5] + frontend.L1.M4[R1,R2] + frontend.L3.M0[R2]。来源：`C2 + C3 + DENIEDFLOW`。
<!--   说明：`B4` 复用 `authorization.L0.M0[R3] + auth.L1.M0[R1,R3]`，承接“授权层需要重新认证或提升认证强度时，如何稳定回切 auth 并尽量恢复原目标”这一类场景结构。它关注的是 authorization 到 auth 的衔接，而不是重写 auth 本身。 -->
- `B4` 认证回切衔接结构基：authorization.L0.M0[R3] + auth.L1.M0[R1,R3]。来源：`C2 + C4 + AUTHRETURNFLOW`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 所有受保护访问先做授权检查
  - `R1.1` 参与基：`B1 + B2`。
  - `R1.2` 组合方式：将判权输入接入结构与结果分流结构串联后，所有需要权限保护的页面、路由和动作在进入前必须先被授权检查承接，再由授权结果决定继续放行、限制放行或拒绝；因此上层不得绕过授权保护直接暴露受限目标。
  - `R1.3` 输出能力：`C1 + C2`。
  - `R1.4` 输出结构：`AUTHZGUARDFLOW + CHECKFLOW + ALLOWFLOW`。
  - `R1.5` 边界绑定：`AUTHZGUARDFLOW/CHECKFLOW/ALLOWFLOW`。
- `R2` 授权拒绝必须导出稳定反馈
  - `R2.1` 参与基：`B2 + B3`。
  - `R2.2` 组合方式：将结果分流结构与拒绝反馈承接结构串联后，整页拒绝、局部拒绝、动作拒绝和资源不可见都必须沿既有授权结果回路导出稳定用户反馈；因此拒绝提示不得脱离授权结果与当前目标单独定义。
  - `R2.3` 输出能力：`C2 + C3`。
  - `R2.4` 输出结构：`DENIEDFLOW + AUTHZGUARDFLOW + ALLOWFLOW`。
  - `R2.5` 边界绑定：`DENIEDFLOW/AUTHZGUARDFLOW/ALLOWFLOW`。
- `R3` 需提升认证时必须稳定回切 auth
  - `R3.1` 参与基：`B2 + B4`。
  - `R3.2` 组合方式：将结果分流结构与认证回切衔接结构串联后，凡是授权结果判定为需要重新认证或需要更高认证强度的场景，都必须切回 auth 既有流程，并保留原访问目标或动作恢复承接；因此上层不得把这类结果误当作普通权限拒绝直接吞掉。
  - `R3.3` 输出能力：`C2 + C4`。
  - `R3.4` 输出结构：`AUTHRETURNFLOW + AUTHZGUARDFLOW + ALLOWFLOW`。
  - `R3.5` 边界绑定：`AUTHRETURNFLOW/AUTHZGUARDFLOW/ALLOWFLOW`。
- `R4` 条件允许必须和完全放行分开表达
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：将判权输入接入结构、结果分流结构和拒绝反馈承接结构串联后，条件允许场景必须显式承接为“可进入但受限”或“可见但不可操作”的结构结果，而不是被伪装成完全放行或完全拒绝。更直白地说：能进页面但按钮禁用，不应被写成“有权限”；按钮点了才拒绝，也不应被伪装成“页面无权限”。
  - `R4.3` 输出能力：`C1 + C2 + C3`。
  - `R4.4` 输出结构：`CHECKFLOW + ALLOWFLOW + DENIEDFLOW`。
  - `R4.5` 边界绑定：`CHECKFLOW/ALLOWFLOW/DENIEDFLOW`。
- `R5` 无效方案
  - `R5.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R5.2` 组合方式：以下方案无效：缺少 `B1` 时，授权检查输入无法稳定形成主体、资源动作和策略输入闭环；缺少 `B2` 时，授权结果无法真正进入页面、动作或路由编排；缺少 `B3` 时，拒绝结果只能退化成不可观察状态切换；缺少 `B4` 时，需要重新认证或提升认证强度的结果无法稳定回切 auth；把“未登录”和“已登录但无权限”合并成同一种 guard 结果，也属于无效场景。
  - `R5.3` 输出能力：`C5`。
  - `R5.4` 输出结构：`AUTHZGUARDFLOW/CHECKFLOW/ALLOWFLOW/DENIEDFLOW/AUTHRETURNFLOW`。
  - `R5.5` 边界绑定：`AUTHZGUARDFLOW/CHECKFLOW/ALLOWFLOW/DENIEDFLOW/AUTHRETURNFLOW`。

## 5. 验证（Verification）

- `V1` 授权保护语义验证：通过条件：所有需要权限保护的页面、路由和动作，都能落入 `AUTHZGUARDFLOW` 的受保护目标解释，并在进入前被稳定包裹到授权检查语义中。失败原因：受保护目标缺失、进入前未承接授权检查或保护范围与放行语义冲突。证据：`AUTHZGUARDFLOW` 对应的产品 section、规则映射与进入样例。
- `V2` 判权输入语义验证：通过条件：`CHECKFLOW` 能解释访问主体、资源动作和策略输入如何形成统一授权检查输入。失败原因：主体来源不清、资源动作表达不稳定或策略输入无法解释。证据：`CHECKFLOW` 对应的产品 section、输入样例与规则说明。
- `V3` 放行分流语义验证：通过条件：`ALLOWFLOW` 能同时解释完全放行和条件允许两类结果，且切换条件清晰。失败原因：条件允许缺失、放行与受限放行混写或分流结果无法落到页面/动作编排。证据：`ALLOWFLOW` 对应的产品 section、分流样例与运行结果。
- `V4` 拒绝反馈语义验证：通过条件：`DENIEDFLOW` 能同时覆盖 `target_denied` 与 `action_denied` 两类反馈，且反馈语义与授权结果一致。失败原因：拒绝类别缺失、反馈不可观察或反馈内容与授权结果不一致。证据：`DENIEDFLOW` 对应的产品 section、提示样例与运行结果。
- `V5` 认证回切语义验证：通过条件：`AUTHRETURNFLOW` 能解释需要重新认证和需要更高认证强度两类结果，且能稳定切回 auth 并保留原目标承接。失败原因：回切条件不清、原目标丢失或与 auth 返回恢复语义脱节。证据：`AUTHRETURNFLOW` 对应的产品 section、回切样例与运行结果。
- `V6` 场景最小必要性验证：通过条件：保留全部 `B*` 时，`AUTHZGUARDFLOW/CHECKFLOW/ALLOWFLOW/DENIEDFLOW/AUTHRETURNFLOW` 可被完整解释。失败原因：移除任一 `B*` 后，至少一个场景边界失去承接结构，或授权结果、拒绝反馈与 auth 回切之间出现脱节。证据：`B1~B4` 到 `R1~R5` 的组合映射与校验结论。
- `V7` 结论表达：通过条件：逐条输出 `R* -> C* / 边界` 的通过或失败结论。失败原因：规则结论无法追溯到参与基、能力或边界绑定。证据：规则级验证报告与映射结果。
