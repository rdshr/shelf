# 前端输入与触发原子模块:FrontendInputTriggerAtoms

@framework

## 1. 能力声明（Capability Statement）

- `C1` 单值输入承载能力：沉淀输入框等单值输入原子的稳定结构。
- `C2` 触发提交承载能力：沉淀按钮等触发与提交原子的稳定结构。
- `C3` 多行与文件入口承载能力：沉淀文本域、文件选择器等扩展输入原子的稳定结构。
- `C4` 非能力项：不负责领域对象语义、页面场景流与样式实例值。

## 2. 边界定义（Boundary / 参数）

- `FIELD` 字段边界：输入值、占位内容与清空规则需稳定。来源：`C1 + C3`。
- `ACTION` 触发边界：点击、提交、禁用与确认出口需稳定。来源：`C2 + C3`。
- `TEXTAREA` 多行边界：段落输入、滚动方式与提交时机需稳定。来源：`C1 + C3`。
- `FILE` 文件边界：文件选择、替换与取消入口需稳定。来源：`C2 + C3`。
- `EVENT` 事件边界：输入、提交、重置与上传事件出口需一致。来源：`C1 + C2`。
- `STATE` 状态边界：默认、聚焦、禁用与处理中状态需统一。来源：`C1 + C2`。
- `A11Y` 可访问边界：焦点顺序、标签关系与键盘触发需达标。来源：`C1 + C2`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 按钮结构基：由标签槽位、触发出口与状态位构成的点击结构。来源：`C2 + ACTION + EVENT + STATE`。
- `B2` 输入框结构基：由值槽位、占位槽位与聚焦边界构成的单值输入结构。来源：`C1 + FIELD + EVENT + A11Y`。
- `B3` 文本域结构基：由多行值槽位、滚动区与提交出口构成的多行输入结构。来源：`C1 + C3 + TEXTAREA + EVENT`。
- `B4` 文件选择结构基：由选择入口、文件槽位与取消出口构成的文件摄取结构。来源：`C2 + C3 + FILE + ACTION`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 单值输入成立
  - `R1.1` 参与基：`B2`。
  - `R1.2` 组合方式：先固定值槽位、占位槽位与聚焦边界，再允许更上层结构接入字段语义。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`FIELD/STATE/A11Y`。
- `R2` 触发提交统一
  - `R2.1` 参与基：`B1 + B2 + B3`。
  - `R2.2` 组合方式：将按钮、输入框与文本域的提交出口统一到同一事件面上，形成稳定输入提交结构。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`ACTION/EVENT/STATE/TEXTAREA`。
- `R3` 文件与扩展输入并轨
  - `R3.1` 参与基：`B1 + B3 + B4`。
  - `R3.2` 组合方式：将文件选择入口、多行输入和触发出口并入同一输入面，形成扩展输入结构。
  - `R3.3` 输出能力：`C2 + C3`。
  - `R3.4` 边界绑定：`FILE/TEXTAREA/ACTION/EVENT/A11Y`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3 + B4`。
  - `R4.2` 组合方式：缺失触发出口、字段边界不可解释或文件入口无法取消的方案均无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`FIELD/ACTION/TEXTAREA/FILE/EVENT/STATE/A11Y`。

## 5. 验证（Verification）

- `V1` 输入完整性：输入框、文本域与按钮的基本输入提交出口必须完整。
- `V2` 文件入口有效性：文件选择结构必须满足 `FILE/ACTION` 约束。
- `V3` 事件一致性：输入、提交与重置事件必须满足 `EVENT/STATE` 规则。
- `V4` 可访问符合性：有效组合必须满足 `A11Y` 约束。
- `V5` 最小必要性：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。
