# 前端技术栈与运行时模块:FrontendTechStackRuntime

@framework

本模块位于前端标准之上，负责把前端实现路径中的技术栈选择、运行时组织与证据出口收敛为稳定实现结构。它回答“同一份 Product Spec 如何落到 React / Vue 等不同前端实现族”，但不反向改写产品页面、交互语义、视觉语义或领域对象。

本模块默认与 `projects/<project_id>/implementation_config.toml` 对应，而不是与 `product_spec.toml` 对应。`implementation_config.toml` 中的前端 section 应承载这里允许变化的实现值；若某项变化会改写产品可观察真相，则该项不属于本模块合法实例值。

## 1. 能力声明（Capability Statement）

- `C1` 技术族收敛能力：把 React、Vue 等前端实现族收敛为稳定技术栈选择结构。
- `C2` 运行时编排能力：把渲染模式、路由承载、状态组织、数据访问与宿主启动收敛为稳定运行时结构。
- `C3` 实现证据暴露能力：把构建产物、调试入口、规格出口与运行时描述收敛为稳定证据结构。
- `C4` 非能力项：不负责定义产品页面、业务语义、视觉 token 真值与领域对象字段。

## 2. 边界定义（Boundary / 参数）

- `STACK` 技术栈边界：前端实现族、语言族与构建工具链需可判定。来源：`C1 + C2`。
- `RENDER` 渲染边界：CSR、SSR、SSG、流式渲染与 hydration 策略需可判定。来源：`C1 + C2`。
- `ROUTER` 路由器边界：文件路由、声明式路由、嵌套路由与深链承接方式需可判定。来源：`C1 + C2`。
- `STATE` 状态运行时边界：本地状态、远程状态、缓存与提交回路的组织方式需可判定。来源：`C2`。
- `DATA` 数据运行时边界：接口调用、查询缓存、流式数据与错误恢复方式需可判定。来源：`C2 + C3`。
- `HOST` 宿主运行时边界：浏览器入口、挂载点、环境变量、资源注入与启动顺序需可判定。来源：`C2 + C3`。
- `EVIDENCE` 证据边界：实现产物、调试描述、运行时摘要与规格出口需可追溯。来源：`C3`。

## 3. 最小可行基（Minimum Viable Bases）

- `B1` 技术栈选择结构基：frontend.L2.M0[R1,R2]。来源：`C1 + STACK + RENDER + ROUTER`。
- `B2` 运行时组织结构基：frontend.L2.M0[R1,R2] + runtime_env.L0.M0[R1,R2]。来源：`C1 + C2 + STATE + DATA + HOST`。
- `B3` 实现证据出口结构基：frontend.L2.M0[R2] + runtime_env.L0.M0[R2,R3]。来源：`C2 + C3 + EVIDENCE + HOST + DATA`。

## 4. 基组合原则（Base Combination Principles）

- `R1` 技术族先定
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先固定前端实现族、渲染模式与路由承接方式，再允许状态运行时、数据运行时与证据出口附着其上。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`STACK/RENDER/ROUTER`。
- `R2` 运行时闭合
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：将状态组织、数据访问、宿主启动与环境策略对齐到同一技术栈选择上，形成稳定前端运行时闭环。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`STATE/DATA/HOST/STACK/RENDER/ROUTER`。
- `R3` 证据附着
  - `R3.1` 参与基：`B2 + B3`。
  - `R3.2` 组合方式：将实现产物命名、调试入口、规格出口与运行时摘要绑定到既定运行时结构上，使实现路径可审查、可复现、可替换。
  - `R3.3` 输出能力：`C2 + C3`。
  - `R3.4` 边界绑定：`EVIDENCE/HOST/DATA/STACK`。
- `R4` 禁止组合
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：让技术栈选择反向改写产品真相、让运行时方案绕过前端标准边界、或让证据出口脱离实际运行时结构的方案均无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`STACK/RENDER/ROUTER/STATE/DATA/HOST/EVIDENCE`。

## 5. 验证（Verification）

- `V1` 技术栈一致性：实现配置中的 renderer family、render mode 与 router family 必须落在同一合法技术族内。
- `V2` 运行时完整性：状态组织、数据访问、宿主启动与环境策略必须满足 `STATE/DATA/HOST` 约束。
- `V3` 真相不泄漏性：实现配置不得改写 `product_spec.toml` 已确定的页面、交互、视觉与外部契约。
- `V4` 证据可追溯性：实现产物、规格出口与调试入口必须满足 `EVIDENCE` 约束，并可回溯到当前实现配置。
- `V5` 最小必要性：移除任一 `B*` 后，`V1~V4` 至少一项失败。
- `V6` 结论表达：逐条输出 `R* -> C* / 边界` 的通过或失败结论。

## 6. 实例化约定（Implementation Config Mapping）

建议由 `projects/<project_id>/implementation_config.toml` 的 `[frontend]` 承载本模块实例值，并按“边界大类 -> 同名子组或稳定字段族”组织。推荐但不强制的字段分组如下：

- `stack_family`：`react` / `vue` / `svelte`
- `meta_framework`：`none` / `nextjs` / `nuxt`
- `language`：`typescript` / `javascript`
- `builder`：`vite` / `rsbuild` / `webpack`
- `render_mode`：`csr` / `ssr` / `hybrid`
- `router_family`：`react_router` / `next_app_router` / `vue_router`
- `state_runtime`：`zustand` / `redux_toolkit` / `pinia`
- `data_runtime`：`tanstack_query` / `swr` / `apollo`
- `style_runtime`：`css_modules` / `tailwind` / `scss`
- `host_runtime`：`browser_only` / `node_browser` / `edge_browser`
- `artifact_profile`：实现产物命名与调试出口所用 profile

若仓库后续要把这些字段编译进验证器，应优先把校验写在 `implementation_config -> generated/*` 的链路里，而不是回写 `product_spec.toml` 或前端 `L2` 标准模块。
