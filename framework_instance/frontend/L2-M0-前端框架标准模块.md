# 工程实例层：前端组合框架模块:FrontendCompositionFramework

@framework_instance

- source_standard: framework/frontend/L2-M0-前端框架标准模块.md
- sync_rule: 当标准分层文件变更时，必须同步更新本实例分层文件。

## 1. 实例化能力声明（Capability Statement）

- `C1` 基础组件沉淀能力（对应标准 `C1`）：沉淀 `Button/Input/FormField/Dialog/Toast` 等原子组件到 `packages/ui/src/atoms`，统一 props 与事件契约。
- `C2` 页面组装能力（对应标准 `C2`）：将自研原子组件、白名单开源组件与基础页面模板组合为可直接运行页面。
- `C3` 场景闭环能力（对应标准 `C3`）：页面继续编排为登录等业务流程，并保持跨页状态一致与可观测。
- `C4` 非能力项（对应标准 `C4`）：本层不定义后端业务规则、数据库模型和基础设施部署策略。

## 2. 实例化边界定义（Boundary / 参数）

- `BA`（对应标准 `BA`）：核心交互（输入、点击、选择、提交）必须有自研原子组件兜底，禁止完全依赖外部组件。
- `DS`（对应标准 `DS`）：组件样式与交互契约绑定设计系统版本（如 `design-system@1.x`），升级需走兼容评审。
- `OS`（对应标准 `OS`）：外部组件必须先通过白名单准入，再完成二次封装（主题 token 适配、API 收敛、定制能力扩展、无障碍补齐）后沉淀到 `packages/ui`，业务层不得直接依赖第三方组件。
- `PT`（对应标准 `PT`）：基础模板至少提供 `login/register/forgot-password` 三类模板并暴露统一扩展点。
- `RT`（对应标准 `RT`）：路由嵌套深度默认不超过 3 级；受限路由必须声明 guard 与 fallback。
- `SLA`（对应标准 `SLA`）：关键流程（如登录）端到端交互需满足既定耗时阈值（以监控面板配置为准）。
- `UX`（对应标准 `UX`）：同类控件的加载、成功、失败、禁用和错误提示表现保持一致。

## 3. 实例化最小可行基（Minimum Viable Bases）

- `B1` 原子组件基（对应标准 `B1`）：`packages/ui/src/atoms` 原子组件库 + 类型定义 + 可访问性属性规范 + 交互单测。
- `B2` 资源装配基（对应标准 `B2`）：第三方组件适配封装层（落库 `packages/ui/src/adapters`）、页面模板层、主题 token 映射层与白名单治理清单。
- `B3` 页面编排基（对应标准 `B3`）：路由编排、页面守卫、流程状态衔接、关键链路性能与观测配置。

## 4. 实例化基组合原则（Base Combination Principles）

- `R1` 原子组件标准化（对应标准 `R1`）
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：统一 props、事件、无障碍属性和 token 映射，形成稳定原子组件目录。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`BA/DS/UX`。
- `R2` 基础资源并轨（对应标准 `R2`）
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：对外部组件执行“评审准入 -> 二次封装 -> 主题适配 -> 定制扩展 -> 统一导出”流水线，最终由 `packages/ui` 对外提供稳定接口。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`OS/PT/DS/UX`。
- `R3` 页面到流程组装（对应标准 `R3`）
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：以模板为骨架装配原子组件，落地登录类页面并接入跨页流程编排。
  - `R3.3` 输出能力：`C2 + C3`。
  - `R3.4` 边界绑定：`BA/OS/PT/RT/SLA/UX`。
- `R4` 禁止组合（对应标准 `R4`）
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：业务代码直接引用第三方组件、跳过二次封装流程、未做主题适配或未沉淀到 `packages/ui` 的方案均判定无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`BA/DS/OS/PT/RT/SLA/UX`。

## 5. 实例化验证（Verification）

- `V1` 基础完整性（对应标准 `V1`）：`B1` 必须覆盖核心输入、点击、提交、反馈四类交互组件。
- `V2` 外部一致性（对应标准 `V2`）：`B2` 中外部资源必须全部来自 `OS` 白名单，且完成二次封装、主题适配与 `packages/ui` 统一导出，并满足 `DS` 版本约束。
- `V2.1` 静态扫描门禁：业务层（`apps/**/src/app|pages|features|processes|entities`）禁止直接 `import` 第三方 UI 包，CI 必须通过该扫描规则。
- `V3` 组装覆盖性（对应标准 `V3`）：`R2~R3` 必须可产出登录类页面并可挂接流程状态与观测。
- `V4` 约束符合性（对应标准 `V4`）：所有有效组合必须满足 `RT/SLA/UX` 约束。
- `V5` 最小必要性（对应标准 `V5`）：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达（对应标准 `V6`）：输出 `R1~R4 -> C* / 边界` 的通过或失败清单并附验证记录。
