## 1. 实例化能力声明（Capability Statement）

- `C1` 结构抽象能力（对应标准 `C1`）：采用 `apps/web + packages/*` 分层，统一通过 `@/` 与包导出边界访问，不允许跨层直连。
- `C2` 运行承载能力（对应标准 `C2`）：采用 `pnpm + React + TypeScript（严格模式） + Vite + Tailwind CSS + shadcn/ui + lucide-react`，支持本地开发、CI 构建、静态部署三种运行形态。
- `C3` 约束外化能力（对应标准 `C3`）：把安全、兼容、观测写入 CI 校验与运行时中间件，形成可自动验收门禁。
- `C4` 非能力项（对应标准 `C4`）：本层不定义具体业务页面编排与视觉主题实例取值。

## 2. 实例化边界定义（Boundary / 参数）

- `ARCH`（对应标准 `ARCH`）：目录采用双层定义：物理结构为 `apps/web/src/{app,pages,features,processes,entities,shared}`，逻辑依赖主链为 `app -> pages -> features -> entities -> shared`。
- `RUNTIME`（对应标准 `RUNTIME`）：运行环境为现代浏览器（Chrome/Edge/Safari 最新两个大版本）与 Node.js 20 LTS 构建环境。
- `BUILD`（对应标准 `BUILD`）：构建工具为 `Vite`；产物包含 `index.html + assets/*`；采用按路由动态分包与 vendor 分包策略。
- `ASSET`（对应标准 `ASSET`）：静态资源采用内容哈希命名；缓存策略为 `immutable + long max-age`；入口 HTML 短缓存并可回源。
- `SEC`（对应标准 `SEC`）：启用依赖漏洞扫描、输入白名单校验、XSS 转义、CSP 基线策略（script-src/style-src 限制）。
- `OBS`（对应标准 `OBS`）：接入前端埋点 SDK，覆盖 PV、路由切换耗时、接口失败率、JS Error 与资源加载失败。
- `COMPAT`（对应标准 `COMPAT`）：最低兼容 ES2020；旧能力通过 polyfill 按需注入并提供降级提示页。

### `ARCH` 分层细化定义

- 物理目录结构（目录层）：
- `apps/web/src/app`：应用壳、路由注册、全局 Provider、启动入口。
- `apps/web/src/pages`：页面入口与页面级编排（按路由聚合 feature）。
- `apps/web/src/features`：可复用业务能力单元。
- `apps/web/src/processes`：跨页面流程编排（可选，用于 L6 场景闭环）。
- `apps/web/src/entities`：领域模型、领域状态、领域 API 适配。
- `apps/web/src/shared`：纯通用能力（UI 基础、工具、常量、基础 hooks）。
- 逻辑依赖主链（架构层）：
- `app`：仅承载应用壳与装配，不承载具体业务规则。
- `pages`：按路由组织页面，编排 `features/entities/shared`，不反向依赖 `app` 细节。
- `features`：承载可复用业务能力单元（如登录、搜索）；可编排多个 `entities/shared`，不反向依赖其他 feature 内部实现。
- `entities`：承载领域模型、领域状态与领域 API 适配；不依赖 `features/app`。
- `shared`：承载纯通用能力（UI 基础组件、工具函数、常量、基础 hooks）；禁止引用 `entities/features/app`。
- 允许依赖方向：`app -> pages -> features -> entities -> shared`，以及 `pages -> processes -> features/entities/shared`（流程编排场景）。
- 禁止依赖示例：`shared -> entities`、`entities -> features`、`features -> pages`、`pages -> app`。
- 跨层导入必须走公开导出（`index.ts`）而非深层文件路径。

## 3. 实例化最小可行基（Minimum Viable Bases）

- `B1` 分层基（对应标准 `B1`）：`apps/web/src/{app,pages,features,processes,entities,shared}` + `packages/{ui,core,config}` + lint 依赖方向规则。
- `B2` 构建运行基（对应标准 `B2`）：`vite.config.ts`、环境变量分层（`.env.development/.env.production`）、静态资源与路由分包配置。
- `B3` 治理观测基（对应标准 `B3`）：`eslint + tsc` 质量门禁、安全扫描任务、观测埋点与错误上报初始化代码。

## 4. 实例化基组合原则（Base Combination Principles）

- `R1` 结构先行组合（对应标准 `R1`）
  - `R1.1` 参与基：`B1`。
  - `R1.2` 组合方式：先创建目录与依赖边界，再接入运行配置与治理能力。
  - `R1.3` 输出能力：`C1`。
  - `R1.4` 边界绑定：`ARCH/COMPAT`。
- `R2` 运行承载组合（对应标准 `R2`）
  - `R2.1` 参与基：`B1 + B2`。
  - `R2.2` 组合方式：在固定分层上叠加构建配置、环境变量、资源缓存与部署拓扑。
  - `R2.3` 输出能力：`C1 + C2`。
  - `R2.4` 边界绑定：`RUNTIME/BUILD/ASSET/ARCH`。
- `R3` 约束闭环组合（对应标准 `R3`）
  - `R3.1` 参与基：`B1 + B2 + B3`。
  - `R3.2` 组合方式：将安全扫描与观测埋点绑定到构建和发布链路，失败即阻断。
  - `R3.3` 输出能力：`C1 + C2 + C3`。
  - `R3.4` 边界绑定：`SEC/OBS/COMPAT/RUNTIME/BUILD`。
- `R4` 禁止组合（对应标准 `R4`）
  - `R4.1` 参与基：`B1 + B2 + B3`。
  - `R4.2` 组合方式：未定义分层直接堆代码、绕过 CI 安全/观测校验、无兼容策略发布均判定无效。
  - `R4.3` 输出能力：`C4`。
  - `R4.4` 边界绑定：`ARCH/RUNTIME/BUILD/ASSET/SEC/OBS/COMPAT`。

## 5. 实例化验证（Verification）

- `V1` 结构一致性（对应标准 `V1`）：执行静态依赖规则检查，禁止 `shared -> features/app` 逆向依赖。
- `V2` 承载完整性（对应标准 `V2`）：`dev/build/preview` 三条链路必须可运行，产物可部署且路由回退可用。
- `V3` 治理有效性（对应标准 `V3`）：安全扫描与错误上报初始化必须启用，关键埋点字段完整。
- `V4` 边界符合性（对应标准 `V4`）：兼容性检查通过，最低浏览器目标与 polyfill 策略一致。
- `V5` 最小必要性（对应标准 `V5`）：移除任一 `B*` 后，`V2~V4` 至少一项失败。
- `V6` 结论表达（对应标准 `V6`）：输出 `R1~R4 -> C* / 边界` 的通过或失败清单并附 CI 结果链接。
