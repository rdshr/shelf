# 逻辑自洽框架设计标准（通用）

## 0. 治理地位（强制）
本文件是本仓库的最高设计规范。

适用要求：
- 新功能设计、重构方案、规则变更，必须先对齐本标准再落地实现。
- 评审时以本标准作为一致性与可验证性的判定依据。
- 当具体领域标准与本标准冲突时，以本标准为准；如需例外，必须记录原因与回滚策略。

## 1. 文档定位
本标准用于定义“如何设计一个框架”，而不是定义某个具体业务。

目标是让任何框架都满足：
- 可解释：每个设计决策有来源和作用
- 可验证：每个关键结论可被验证或推翻
- 可演化：规则可随证据迭代，而不是固定教条

## 2. 适用范围
适用于以下场景：
- 新框架设计
- 旧框架重构
- 跨团队的框架评审与治理

不限制业务领域，不限制架构风格。

## 3. 抽象元模型（统一语言）
任何框架都应被抽象为以下元素：
- `G` Goal：目标，定义“要优化什么”
- `B` Boundary：边界，定义“约束和适用范围”
- `M` Module：模块，定义“可组合能力单元”
- `R` Rule：规则，定义“模块间关系与约束”
- `H` Hypothesis：假设，定义“可被检验的主张”
- `V` Verification：验证，定义“如何判定主张成立”
- `C` Conclusion：结论，定义“在当前证据下是否采纳”

推荐依赖链：
`G -> B -> M -> R -> H -> V -> C`

## 4. 设计原则（强制）

### P1. 来源一致性
每个 `B/M/R` 必须能追溯到 `G`，禁止“经验硬塞”。

### P2. 作用单一性
每个模块应有主职责；一个模块承担多目标时必须显式拆分。

### P3. 规则可推导
`R` 必须由 `B + M` 推导，不允许先定规则再反推边界。

### P4. 假设可证伪
`H` 必须可以被数据或实验推翻，不能是不可检验表述。

### P5. 约束最小化
只保留对目标达成有净贡献的约束；无贡献约束降级或移除。

### P6. 变更可逆
规则或模块变更必须带影响分析和回滚路径。

## 5. 设计接口标准

### 5.1 Boundary 接口
- 输入：约束维度列表（数量、性能、成本、时延、容量等）
- 输出：`is_valid` 结果与错误原因
- 要求：每个维度都可测量

### 5.2 Module 接口
- 输入：模块定义与能力声明
- 输出：可组合的模块集合
- 要求：模块职责可枚举、可替换

### 5.3 Rule 接口
- 输入：模块组合与边界条件
- 输出：布尔判定 + 规则 ID
- 要求：每条规则可独立执行

### 5.4 Verification 接口
- 输入：边界、组合、基线、目标、上下文
- 输出：`passed` 与 `reasons[]`
- 要求：失败必须可解释，不能只返回 true/false

### 5.5 LogicRecord 接口
- 输入：步骤列表（`step_id`, `depends_on`, `evidence`）
- 输出：自洽性检查结果
- 要求：支持审计导出（如 JSON）

## 6. 自洽性判定标准

### 6.1 完整性
- 每个 `G` 至少关联一个 `B`
- 每个 `C` 必须有 `V` 证据

### 6.2 无冲突性
- 同层规则不可互相矛盾
- 出现冲突时必须定义优先级或权重

### 6.3 可追溯性
- `step_id` 唯一
- `depends_on` 只能依赖前序步骤
- 不允许孤儿结论

### 6.4 可删减性
- 删除某规则后若目标不受损，该规则不应是强制规则

## 7. 验证设计标准
每个验证用例至少包含：
- 基线（baseline）
- 目标（target）
- 判定函数（metric or predicate）
- 通过条件（pass criteria）
- 失败解释（failure reasons）

建议覆盖：
- 正常路径
- 边界路径
- 失败路径
- 回滚路径

## 8. 演化与治理标准
规则或模块变更必须记录四项：
1. 变更动机（为什么改）
2. 影响范围（影响哪些 `B/M/R/H/V`）
3. 验证证据（怎么证明改动有效）
4. 回滚策略（失败时如何撤回）

## 9. 工程落地约束（仓库规范）
- Python 版本与依赖管理必须使用 `uv`。
- 开发与运行命令必须通过 `uv run ...` 执行。
- 新增依赖必须通过 `uv add <package>`，并提交 `pyproject.toml` 与 `uv.lock`。
- 不允许把 `pip install` 写入仓库标准流程。

推荐运行：
```bash
uv sync
uv run python main.py
```

## 10. 最小交付产物
一个符合标准的框架至少应产出：
- 一份 `G/B/M/R/H/V/C` 定义文档
- 一份可执行验证入口
- 一份逻辑记录导出（含自洽性检查）
- 一份变更记录（含影响分析与回滚说明）
